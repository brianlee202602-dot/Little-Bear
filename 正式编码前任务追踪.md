# 正式编码前任务追踪

> 目标：在进入正式编码前，把当前设计文档收敛为可执行、可验证、可追踪的工程契约，降低后续返工、权限绕过、接口不一致和数据模型反复调整的风险。

## 1. 编码准入标准

只有当以下 P0 项全部完成后，才建议进入正式业务编码：

- [x] MVP 闭环范围已冻结。
- [x] 初始化配置与 `active_config v1` schema 已明确。
- [x] RESTful API 已落成 OpenAPI 契约。
- [x] 核心数据库 schema 已明确字段、约束、索引和枚举。
- [x] 接口权限矩阵已明确到 endpoint 级别。
- [ ] 核心状态机已明确允许迁移和失败处理。
- [ ] P0 审计事件字典和脱敏规则已明确。
- [ ] P0 测试计划已覆盖初始化、认证、权限、导入、索引和查询闭环。
- [ ] 本地开发环境和最小依赖启动方式已明确。

### 2.1 冻结 MVP 闭环范围

状态：已完成

产物：

- [x] `docs/MVP范围说明.md`
- [x] P0 功能列表

建议 P0 闭环：

1. 初始化系统。
2. 创建首个系统管理员。
3. 登录并获取 access/refresh JWT。
4. 创建组织、用户和角色绑定。
5. 创建知识库。
6. 上传文档并创建导入任务。
7. Worker 解析、切块、向量化、写 draft 索引。
8. 发布 active index。
9. 用户按权限查询。
10. 返回带 citation 的答案。
11. 写查询、导入、权限和管理员操作审计。

完成标准：

- P0 范围可以支持“文档导入到可权限过滤查询”的完整闭环。
- P0 不包含复杂审批流、多租户运营后台、外部组织同步、高级评测平台、复杂多 Agent 编排等非首阶段能力。
- 每个 P0 功能都能映射到 API、数据表、状态机和测试用例。

## 4. 建议编码顺序

1. 工程骨架：FastAPI、错误结构、request_id、trace_id、日志中间件。
2. 数据库基础：SQLAlchemy、Alembic、事务管理、repository 基类。
3. 配置契约：初始化配置 schema、`active_config v1` schema、Secret 引用、模型 mock 配置。
4. Setup Service：初始化状态、setup JWT、初始化配置校验、active config v1。
5. Config Service：active config 读取、缓存、版本和发布；P0 不实现完整审批/回滚。
6. Auth Service：用户、密码、登录、refresh token rotation、JWT 状态。
7. Org Service：企业、部门、成员关系、组织版本。
8. Permission Service：角色、绑定、权限上下文、资源权限策略。
9. KnowledgeBase / Document 元数据。
10. Import Service 与 Worker：任务创建、领取、解析、切块、embedding。
11. Indexing Service：draft 写入、校验、active 发布、兼容性校验和删除阻断。
12. Query Service：查询入口、权限过滤、召回、rerank、上下文、答案和 citation。
13. Audit / Observability：P0 审计事件、查询日志、模型调用日志、基础指标。
14. P0 管理后台 API 按 OpenAPI 补齐。

## 5. 风险清单

| 风险 | 影响 | 编码前动作 |
| --- | --- | --- |
| API 只有路径，没有 schema | 前后端联调反复返工 | 先完成 OpenAPI |
| 数据模型缺少约束和索引 | 查询慢、状态错乱、权限绕过 | 先完成 schema 和 migration 草案 |
| 权限矩阵不明确 | 管理后台越权风险 | endpoint 级权限矩阵先行 |
| 状态机没有固化 | Worker 重试、取消、补偿行为不一致 | 状态机设计先行 |
| 真实模型依赖过早接入 | 开发和测试不稳定 | 先做模型 mock 和 adapter |
| 配置和 secret 处理不清晰 | 部署后难以恢复和审计 | 初始化配置 schema、active_config v1 schema 和 Secret Store 策略先行 |
| 缓存 key 设计不完整 | 跨用户、跨权限串权 | 查询缓存测试和 key 规范先行 |
