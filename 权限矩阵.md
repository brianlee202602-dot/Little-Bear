# P0 接口权限矩阵

## 1. 文档目标

本文定义 Little Bear RAG 后端 P0 阶段所有对外和内部接口的权限边界，作为后续 Auth Middleware、Permission Service、审计事件、API contract 测试和前端菜单权限映射的输入。

依据文档：

- `openapi.yaml`
- `database-schema.md`
- `docs/modules/00-公共实现约束.md`
- `docs/modules/02-认证服务设计实现文档.md`
- `docs/modules/04-权限服务设计实现文档.md`
- `docs/modules/11-审计与可观测性设计实现文档.md`
- `docs/modules/15-前端与管理后台API设计实现文档.md`

## 2. 判定原则

- 前端菜单隐藏不是安全边界。所有业务接口必须由后端校验 token、scope 和资源范围。
- `enterprise_id` 只能从 JWT、setup token、service token 或权限上下文解析，禁止信任前端传入。
- JWT 中不存完整权限上下文，Permission Service 必须按用户、组织、角色和权限版本构建。
- 查询与文档读取必须执行资源可见性校验，不能只检查 `document:read` scope。
- 写接口默认要求审计；高风险写接口必须有二次确认或等价的命令行/后台确认机制。
- 内部模型网关接口只允许 service token 访问，不允许普通用户或管理后台直接调用。
- 初始化生命周期门禁独立于 token 和 scope 校验；未初始化时必须 fail closed，初始化完成后必须关闭初始化写接口。
- P0 权限策略只支持 `department` 和 `enterprise` 可见性，以及资源归属部门维护；不得引入指定用户共享、项目组共享或自定义组织 ACL。

## 3. 用户类型

| user_type | 说明 |
| --- | --- |
| anonymous | 未登录访问，仅允许健康检查、有限 setup 状态和登录 |
| setup | setup JWT，仅允许初始化校验和初始化提交 |
| user | 普通登录用户，可查询和访问自己有权限的知识库/文档 |
| admin | 登录用户且具备管理 scope，可访问管理后台接口 |
| service | 服务间调用 token，用于模型网关和后续 Worker 内部接口 |

## 4. 鉴权类型与 Scope 命名

`required_auth` 表示接口接受的 token 类型，`required_scope` 只表示 token 通过身份校验后仍需具备的业务 scope。二者必须分开校验，不能把 refresh token、setup token 或 service token 当作 scope 处理。

| required_auth | 说明 |
| --- | --- |
| none | 不需要 Authorization header |
| access_token | 普通用户 access JWT |
| refresh_token | refresh JWT，仅用于 token rotation |
| setup_token | setup JWT，仅用于初始化配置校验和初始化提交 |
| service_token | 服务间调用 JWT |

## 5. 初始化生命周期门禁

| lifecycle_condition | 允许和拒绝规则 |
| --- | --- |
| `before_initialized` | 仅允许 `/health/live`、`/health/ready`、`GET /internal/v1/setup-state`、`POST /internal/v1/setup-config-validations`、`PUT /internal/v1/setup-initialization`；其他业务、管理和服务间接口必须返回 `SYSTEM_NOT_INITIALIZED`。 |
| `after_initialized` | 允许登录和普通业务接口；`POST /internal/v1/setup-config-validations` 和 `PUT /internal/v1/setup-initialization` 必须返回 `SETUP_CLOSED`，不得再次写入初始化配置。 |
| `always` | 健康检查和有限 setup 状态查询可在初始化前后访问，但响应不得包含 secret、token 明文或可复用初始化凭证。 |

鉴权中间件执行顺序必须为：初始化生命周期门禁 -> token 类型校验 -> scope 校验 -> resource_check -> 审计与业务处理。

## 6. 资源校验类型

| resource_check | 说明 |
| --- | --- |
| none | 不需要资源范围校验 |
| setup_state | 只返回有限初始化状态，不返回 secret 或 token 明文 |
| self_user | 只能访问当前登录用户 |
| query_visibility | 根据用户部门、文档 visibility、access block、permission_version 和 active index 过滤 |
| kb_read | 校验知识库对当前用户可见 |
| document_list_visibility | 在知识库可见基础上，逐文档过滤 `visibility`、`owner_department_id`、文档状态、access block、permission_version 和 active index |
| document_read | 校验文档、版本、chunk、预览对当前用户可见 |
| document_import | 校验 `document:import` 和目标知识库写入权限，不要求具备知识库管理权限 |
| kb_manage | 校验操作者对知识库具备管理范围 |
| folder_manage | 校验操作者对文件夹所在知识库具备管理范围 |
| document_manage | 校验操作者对文档所在知识库具备管理范围 |
| import_job_owner | 只能操作本人创建的导入任务 |
| import_job_admin | 校验管理后台导入任务查看权限和资源范围 |
| user_admin | 校验目标用户在操作者可管理范围内 |
| org_admin | 校验部门在操作者可管理范围内 |
| role_admin | 校验角色和绑定 scope 在操作者可管理范围内 |
| permission_admin | 校验 P0 资源权限策略在操作者可管理范围内，只接受 `visibility` 和 `owner_department_id` |
| target_user_admin | 校验权限评估目标用户在操作者可管理范围内 |
| config_admin | 校验配置管理权限 |
| audit_admin | 校验审计查看权限和脱敏策略 |
| service_only | 只允许受信服务 token |

P0 权限策略请求中若出现 `user_ids`、`project_ids`、`group_ids`、`custom_org_ids`、`acl_entries` 或递归部门继承配置，必须拒绝并返回 `UNSUPPORTED_PERMISSION_POLICY`。

## 7. 补偿策略代码

| compensation_strategy | 说明 |
| --- | --- |
| `N/A` | 无需补偿 |
| `DENY_ONLY` | 鉴权失败直接拒绝，不产生业务状态变更 |
| `AUDIT_FAILURE` | 失败也必须写审计，不能静默失败 |
| `ROLLBACK_TX` | 数据库事务失败必须整体回滚 |
| `REVOKE_SETUP_TOKEN` | 初始化成功或失败后按状态吊销/标记 setup token |
| `ROTATE_REFRESH_TOKEN` | refresh token 使用后必须一次性失效并签发新 token |
| `ACCESS_BLOCK_THEN_ASYNC_DELETE` | 先写 access block，再异步删除索引/对象 |
| `INVALIDATE_PERMISSION_CACHE` | 权限变更后递增版本并失效权限缓存 |
| `INVALIDATE_QUERY_CACHE` | 文档、索引、权限或配置变更后失效相关查询缓存 |
| `ENQUEUE_INDEX_JOB` | 创建导入、重建、删除或权限刷新任务 |
| `WORKER_RETRY` | 异步任务失败进入重试或人工处置 |
| `CONFIG_VALIDATE_BEFORE_PUBLISH` | 配置发布前必须校验依赖和 schema |
| `MASK_SENSITIVE_OUTPUT` | 返回和日志必须脱敏 |
| `RATE_LIMIT` | 登录、查询或模型调用失败时限流 |

## 17. 高风险操作覆盖清单

| 高风险操作 | 覆盖接口 | 要求 |
| --- | --- | --- |
| 初始化提交 | `PUT /internal/v1/setup-initialization` | setup JWT、二次确认、事务提交、setup token 置 used、初始化审计 |
| setup JWT 签发和轮换 | CLI `rag-admin setup-token issue/rotate` | 每次生成新 token，旧 active token 立即 revoked，写审计 |
| 系统管理员创建、禁用、删除 | `POST/PATCH/DELETE /internal/v1/admin/users...`、角色绑定接口 | `user:manage` + `role:manage`，高风险审计，涉及最后管理员时拒绝或强确认 |
| 高风险角色绑定 | `POST/PUT /internal/v1/admin/users/{user_id}/role-bindings` | 高风险角色必须确认，递增 permission_version，失效权限缓存 |
| 文档可见性从 department 改为 enterprise | `PATCH /internal/v1/admin/documents/{doc_id}`、`PUT /internal/v1/documents/{doc_id}/permissions` | 扩大可见范围必须确认和审计，刷新权限快照与索引 payload |
| 权限收紧 | `PUT /internal/v1/knowledge-bases/{kb_id}/permissions`、`PUT /internal/v1/documents/{doc_id}/permissions` | 先写 access block 或阻断旧索引，再刷新索引和缓存 |
| 越界权限策略 | `PUT /internal/v1/knowledge-bases/{kb_id}/permissions`、`PUT /internal/v1/documents/{doc_id}/permissions` | 仅允许 `department` / `enterprise` 可见性和归属部门变更；指定用户、项目组、自定义组织 ACL 必须拒绝 |
| 配置发布 | `PATCH /internal/v1/admin/config-versions/{version}` | 发布前 schema 和依赖校验，critical 审计，失败回滚 active 指针 |
| 文档删除 | `DELETE /internal/v1/admin/documents/{doc_id}` | 先 access block，再标记 deleted，再异步删除索引和对象 |
| 知识库删除 | `DELETE /internal/v1/admin/knowledge-bases/{kb_id}` | 先阻断知识库下所有查询可见性，再异步清理 |
| 索引重建和发布 | `POST /internal/v1/admin/documents/{doc_id}/index-jobs` | 创建任务，旧 active index 继续服务，新索引 ready 后发布，失败不影响旧版本 |
