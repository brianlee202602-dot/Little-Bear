# P0 审计事件字典与脱敏规则

> 目标：为 Little Bear RAG 后端 P0 阶段提供可执行的审计契约，明确哪些行为必须写 `audit_logs`，哪些信息进入 `query_logs` / `model_call_logs`，以及哪些字段必须脱敏或禁止记录。

## 1. 文档范围

本文覆盖：

- 初始化 validate / initialize。
- setup JWT 签发、轮换、使用和失效。
- 登录成功、登录失败、refresh token rotation。
- 用户创建、禁用、软删除、重置密码、解锁。
- 角色绑定、替换和撤销。
- 文档导入、导入失败、导入重试、导入取消。
- 文档可见性扩大和权限收紧。
- 文档删除、知识库删除、access block 创建。
- 查询、候选阻断、citation 校验失败、降级响应。
- 模型调用、模型 mock 失败和降级。

本文不负责：

- 指定 P1/P2 审批流、配置回滚平台、审计导出和复杂审计展示。
- 代替 `query_logs`、`model_call_logs` 保存高频明细。
- 保存密码、token、secret value、完整 prompt、未脱敏敏感原文。

## 2. 设计原则

- `audit_logs` 记录“关键错误、拒绝、高风险写操作和异常结果”，用于追责、排障和高风险操作追踪。
- `query_logs` 记录“每次查询链路摘要”，用于检索、降级、权限和引用链排障。
- `model_call_logs` 记录“每次模型/模型 mock 调用摘要”，用于成本、性能和降级排障。
- 一个请求可以同时写三类日志，但职责不能混淆。
- 高风险写操作必须和状态变更在同一事务内写审计；外部物理清理失败不能回滚安全阻断状态。
- 事件名固定，小写点分风格；禁止后续实现自由命名。

## 3. 日志边界

### 3.1 `audit_logs`

适用场景：

- 初始化、认证安全事件、管理员操作、权限变更、配置发布、导入任务关键状态、高风险删除、查询拒绝/降级/失败/候选阻断。

最小字段边界：

| 字段 | 说明 |
| --- | --- |
| `event_name` | 固定事件名 |
| `actor_type` / `actor_id` | `anonymous` / `setup` / `user` / `service` / `system` |
| `resource_type` / `resource_id` | 触发行为的主资源 |
| `action` | create / validate / initialize / rotate / delete / publish / query 等 |
| `result` | `success` / `failure` / `denied` |
| `risk_level` | `low` / `medium` / `high` / `critical` |
| `summary_json` | 脱敏摘要 |
| `error_code` | 失败或拒绝时必填 |
| `config_version` | setup 早期可空 |
| `permission_version` | setup/global 事件可空 |
| `index_version_hash` | 查询、导入、发布、权限刷新时按需填写 |

### 3.2 `query_logs`

适用场景：

- 每次 `POST /internal/v1/queries` 非流式或流式查询。

记录重点：

- `query_hash`
- `kb_ids`
- `status`
- `degraded` / `degrade_reason`
- `permission_filter_hash`
- `index_version_hash`
- `model_route_hash`
- `candidate_count`
- `citation_count`
- `error_code`
- `cache_key_hash`
- `cache_hit`
- `cache_policy`
- `cache_invalidated_by_permission_version`

说明：

- 查询文本默认只存 hash，不进 `audit_logs.summary_json`。
- 每次查询都必须写 `query_logs`。
- 查询级拒绝、失败、降级、候选阻断、citation 失败应同时写 `query_logs` 和对应 `audit_logs` 事件。

### 3.3 `model_call_logs`

适用场景：

- embedding / rerank / chat 调用，包括 mock、单一 adapter 和真实外部模型 adapter。

记录重点：

- `caller`
- `model_type`
- `model_name`
- `model_version`
- `model_route_hash`
- `status`
- `degraded`
- `latency_ms`
- `token_usage_json`
- `prompt_hash`
- `input_hash`
- `output_hash`
- `error_code`

说明：

- `model_call_logs` 不替代审计。只有高风险、失败或降级场景需要额外写 `audit_logs`。

### 4.4 `risk_level`

- `low`
- `medium`
- `high`

风险判断基线：

- `low`：只读或低风险系统事件。
- `medium`：普通写操作、普通失败、普通降级。
- `high`：用户/权限/导入/删除/访问阻断相关操作。
- `critical`：初始化提交、setup token 轮换、配置发布、系统管理员相关高风险操作。

## 5. 脱敏与禁止记录规则

### 5.1 全局禁止记录字段

以下内容不得进入 `audit_logs.summary_json`、普通日志、默认 API 响应：

- 密码明文、密码摘要材料、重置密码明文。
- access / refresh / setup / service token 明文。
- secret value、API key、对象存储 access key。
- 完整 prompt。
- 未脱敏用户输入原文。
- 未脱敏文档原文、大段 chunk 正文。
- 完整个人敏感信息。

### 5.2 允许的替代表达

用以下方式替代敏感值：

- token -> `jti`、`token_type`、前缀摘要或 hash。
- secret -> `secret_ref`、provider 类型、路径摘要。
- query -> `query_hash`、长度、语言、标签。
- prompt -> `prompt_template_id`、`prompt_template_version`、`prompt_hash`、变量摘要。
- 文档内容 -> `content_hash`、snippet hash、限长 `text_preview`。
- 个人信息 -> 掩码值，例如手机号后四位、邮箱局部遮罩。

### 5.3 `summary_json` 通用要求

- 只存排障需要的最小摘要。
- 所有 hash 字段必须稳定、可复现。
- 列表字段必须限长；超限时记录数量和首 N 项摘要。
- 错误信息必须使用脱敏后的 `error_code` + `error_message` 摘要，不得原样拼接第三方错误响应。

## 6. `summary_json` 推荐公共字段

并非所有事件都必须全部包含，但命名应统一：

| 字段 | 说明 |
| --- | --- |
| `request_path` | API 路径或 CLI 命令名 |
| `request_method` | HTTP 方法或 `cli` |
| `reason` | 删除、权限收紧、超时、降级等原因 |
| `before` | 变更前摘要 |
| `after` | 变更后摘要 |
| `job_type` | import job 类型 |
| `job_stage` | import job 当前阶段 |
| `attempt_count` | 当前尝试次数 |
| `permission_filter_hash` | 权限过滤 hash |
| `config_hash` | 配置 hash，不能保存 secret value |
| `content_hash` | 文档或 chunk 内容 hash |
| `degraded` | 是否降级 |
| `degrade_reason` | 降级原因 |
| `candidate_summary` | 候选数量、阻断数量、阻断原因摘要 |
| `citation_summary` | citation 数量、chunk/document_version/page 回溯摘要、校验结果摘要 |
| `model_summary` | 模型类型、路由 hash、状态、耗时摘要 |
| `changed_fields` | 本次修改字段集合 |
| `actor_roles` | 操作者角色/权限摘要，管理员也使用此字段，不扩展 `actor_type` 枚举 |
| `cache_summary` | 查询缓存命中、key hash、失效原因摘要 |

- 不保存可直接复原敏感内容的用户输入明文，除非后续明确设计受控留存策略，且不属于 P0 默认行为。

## 11. 实现要求

- 审计写入失败不能静默吞掉；至少写错误日志并返回 `AUDIT_WRITE_FAILED` 或触发告警。
- 对“高风险状态变更”和“审计记录”必须在同一事务提交。
- Worker / Query / Model Gateway 必须带 `request_id` 或 `job_id` 贯穿事件。
- `summary_json` 字段名必须复用本文命名，避免模块各自发明字段。
- 事件名、`resource_type`、`risk_level` 应抽成常量，放入对应模块 `events.py`。
