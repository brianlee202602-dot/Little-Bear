# P0 核心状态机设计

## 1. 文档目标

本文定义 Little Bear RAG 后端 P0 阶段的核心状态机，作为 API Service、Worker、Repository、审计事件字典、状态迁移测试和正式编码的共同输入。

覆盖范围：

- setup 初始化状态机。
- setup JWT 状态机。
- access / refresh token 状态机。
- 企业、组织、账号与角色绑定状态机。
- import_job 状态机。
- knowledge_base 与 folder 状态机。
- document lifecycle、可见性与 index 状态机。
- document_version 状态机。
- chunk 生命周期状态机。
- index_version 状态机。
- chunk_index_ref 与 keyword_index_entry 可见性状态机。
- config_version 状态机。
- access_block 生命周期。

本文不引入完整配置审批、通用配置回滚、多租户运营后台、外部组织同步或复杂 Agent 编排。

## 2. 通用状态迁移约束

- PostgreSQL 是状态事实源；Qdrant、关键词索引、Redis 缓存和对象存储都是派生状态。
- 所有状态变更必须通过领域服务或 Repository 的显式 transition 方法完成，禁止业务代码直接 `UPDATE status`。
- 状态迁移必须使用 compare-and-set 语义，例如 `WHERE id = ? AND status IN (...)`；影响行数为 0 时返回 `INVALID_STATE_TRANSITION` 或按幂等规则处理。
- 高风险状态变更必须和审计记录在同一事务中提交；异步外部清理失败不能回滚安全阻断状态。
- 查询链路必须 fail closed，只接受明确 active、indexed、未 blocked、权限版本有效的数据。
- 幂等请求必须使用 `idempotency_key`、资源当前状态和请求摘要共同判断；同一请求重复到达时返回当前结果，不重复创建任务或重复发布。
- 终态默认不可逆；需要重新处理时优先创建新版本或新任务，而不是直接把 `failed` / `cancelled` 记录改回处理中。
- P0 审计事件名在本文中给出建议名，最终字段细节以后续 `docs/contracts/审计事件字典.md` 为准。

建议所有状态迁移方法接受以下上下文：

| 字段 | 说明 |
| --- | --- |
| `actor_type` / `actor_id` | anonymous、setup、user、service、system |
| `request_id` / `trace_id` | 请求和链路追踪 |
| `idempotency_key` | 可选，写接口和导入任务强烈建议提供 |
| `expected_state` | 调用方认为的当前状态，用于 CAS |
| `target_state` | 目标状态 |
| `reason` | 状态变更原因，例如 `deleted`、`permission_tightened` |
| `metadata_json` | 脱敏摘要，不得包含 password、token、secret value、完整 prompt 或未脱敏原文 |

## 4. Setup 初始化状态机

事实源：

- `system_state.setup_status`
- `system_state.initialized`
- `system_state.active_config_version`
- `system_state.setup_attempt_count`
- `system_state.setup_locked_until`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `not_initialized` | 空库或迁移后尚未进入初始化 |
| `setup_required` | 可执行 setup 校验和初始化提交 |
| `validating_config` | 正在校验初始化配置 schema、Secret ref 和安全策略 |
| `testing_dependencies` | 正在测试 Redis、对象存储、向量库、关键词检索和外部模型 provider |
| `creating_admin` | 正在创建首个系统管理员、默认企业、默认部门和默认角色 |
| `publishing_config` | 正在写入 `config_versions`、`system_configs` 并发布 `active_config v1` |
| `initialized` | 初始化完成，普通业务 API 可用，setup 写接口关闭 |
| `validation_failed` | 初始化配置校验失败 |
| `dependency_test_failed` | 依赖连通性或外部模型 provider 健康检查失败 |
| `initialization_failed` | 创建业务初始数据或发布配置失败 |
| `recovery_required` | 已初始化系统检测到 active config 缺失或损坏 |
| `recovery_validating_config` | 受控恢复初始化校验中，P0 不提供普通后台入口 |
| `recovery_publishing_config` | 受控恢复发布中，P0 不提供完整回滚平台 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| `not_initialized` | `setup_required` | API 启动或 migration seed | `initialized=false` |
| `setup_required` | `validating_config` | `POST /internal/v1/setup-config-validations` 或 `PUT /internal/v1/setup-initialization` | setup JWT active，未超过锁定次数 |
| `validation_failed` | `validating_config` | 重试校验或初始化 | 请求体 hash 允许变化 |
| `dependency_test_failed` | `validating_config` | 重试校验或初始化 | 依赖可能恢复 |
| `initialization_failed` | `validating_config` | 重试初始化 | 失败事务未留下半初始化 active 状态 |
| `validating_config` | `setup_required` | 配置校验成功但未提交初始化 | 仅校验接口使用 |
| `validating_config` | `validation_failed` | 校验失败 | 写入脱敏错误摘要 |
| `validating_config` | `testing_dependencies` | 初始化提交继续 | schema、Secret ref、安全策略校验通过 |
| `testing_dependencies` | `dependency_test_failed` | 依赖测试失败 | 可重试错误写入摘要 |
| `testing_dependencies` | `creating_admin` | 依赖测试通过 | ServiceBootstrap 预检 ready |
| `creating_admin` | `publishing_config` | 业务初始数据创建成功 | 企业、部门、管理员、角色和绑定在同一事务内可见 |
| `creating_admin` | `initialization_failed` | 创建失败 | 事务回滚或补偿到无 active 初始化 |
| `publishing_config` | `initialized` | active config 发布成功 | `active_config_version=1`，`initialized=true` |
| `publishing_config` | `initialization_failed` | 配置发布失败 | 不得留下多个 active config |
| `initialized` | `recovery_required` | ServiceBootstrap 或巡检 | active config 缺失、损坏或 bootstrap 失败 |
| `recovery_required` | `recovery_validating_config` | 受控 CLI | `recovery_setup_allowed=true` |
| `recovery_validating_config` | `recovery_publishing_config` | 受控 CLI | 校验通过 |
| `recovery_publishing_config` | `initialized` | 受控 CLI | 重新发布 active config 成功 |

禁止迁移：

- `initialized -> setup_required`、`initialized -> not_initialized`。
- 任意状态直接跳到 `initialized`，除非 `config_versions` 存在唯一 active 版本且 ServiceBootstrap ready。
- `recovery_required` 通过普通管理后台或普通 setup API 自动恢复。
- 初始化完成后再次调用 `POST /internal/v1/setup-config-validations` 或 `PUT /internal/v1/setup-initialization` 写入配置。

幂等行为：

- 重复 `GET /internal/v1/setup-state` 只读，不迁移状态。
- 同一 `PUT /setup-initialization` 在 `initialized=true` 且请求配置 hash 与 active config hash 一致时返回已初始化摘要；hash 不一致必须返回 `SETUP_CLOSED`。
- `POST /internal/v1/setup-config-validations` 不得创建管理员、角色或 active config。

失败重试：

- `validation_failed`、`dependency_test_failed`、`initialization_failed` 可重试。
- 达到 `setup_attempt_count` 上限后设置 `setup_locked_until`，锁定期内拒绝写接口。
- retryable 错误只记录脱敏摘要，Secret value 不进入状态或审计。

取消行为：

- P0 不提供初始化取消接口。
- 如果初始化进程崩溃，下一次启动按当前状态恢复：中间态超过超时时间后转入对应失败态。

补偿或阻断策略：

- 初始化期间普通业务、管理后台和服务间接口必须返回 `SETUP_REQUIRED`。
- 业务初始数据、配置发布和 `initialized=true` 必须在事务边界内保持一致。
- 发布失败不得留下可被 ServiceBootstrap 使用的半成品 active config。

审计事件：

- `setup.config_validated`
- `setup.config_validation_failed`
- `setup.initialization_started`
- `setup.initialization_failed`
- `setup.initialized`
- `setup.recovery_required`

## 5. Setup JWT 状态机

事实源：

- `setup_tokens.status`
- `jwt_tokens.status`
- `jwt_tokens.token_type='setup'`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `active` | 当前可用 setup JWT，系统内最多一条 |
| `used` | 已用于初始化提交的一次性 token |
| `revoked` | 被新 token 轮换、管理员吊销或初始化完成后失效 |
| `expired` | 到期后由校验或清理任务标记 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `active` | 受控 CLI | 签发新 setup JWT |
| `active` | `revoked` | 受控 CLI | 签发新 token 前同事务失效旧 active token |
| `active` | `used` | `PUT /setup-initialization` | 初始化提交事务成功 |
| `active` | `expired` | token 校验或清理任务 | `expires_at <= now()` |

禁止迁移：

- `used`、`revoked`、`expired` 回到 `active`。
- setup JWT 获取普通业务、管理员或服务间 scope。
- 多个 `active` setup token 同时存在。

幂等行为：

- 对同一 active token 重复调用配置校验不改变状态。
- 初始化提交成功后的重复请求使用同一 token 时，若 token 已 `used` 且系统已 `initialized`，返回初始化已完成摘要；不得再次写入配置。

失败重试：

- 校验失败不消耗 setup token。
- 初始化事务失败时可以保留 token active，除非失败原因表明 token 泄露、过期或安全策略不满足。

取消行为：

- setup JWT 无取消状态；通过 `revoked` 表示失效。

补偿或阻断策略：

- 每次签发新 setup JWT 必须先失效旧 active token。
- 初始化成功后必须将当前 setup token 和对应 `jwt_tokens` 标记为 `used` 或 `revoked`。

审计事件：

- `setup_token.issued`
- `setup_token.rotated`
- `setup_token.revoked`
- `setup_token.used`
- `setup_token.expired`

## 6. Access / Refresh Token 状态机

事实源：

- `jwt_tokens.status`
- `jwt_tokens.token_type`
- `jwt_tokens.replaced_by_jti`

状态枚举：

| 状态 | access token | refresh token |
| --- | --- | --- |
| `active` | 可访问业务 API | 可用于 refresh rotation |
| `used` | 不适用 | 已完成一次 rotation |
| `revoked` | 已登出、管理员禁用或安全撤销 | 已登出、管理员禁用或安全撤销 |
| `expired` | 已过期 | 已过期 |

允许迁移：

| token_type | from | to | 触发方 | 条件 |
| --- | --- | --- | --- | --- |
| access | none | `active` | 登录或 refresh rotation | 用户 active，凭证校验通过 |
| access | `active` | `revoked` | 登出、用户禁用、软删除、安全撤销 | 当前 jti 或用户会话范围 |
| access | `active` | `expired` | token 校验或清理任务 | `expires_at <= now()` |
| refresh | none | `active` | 登录或 refresh rotation | 用户 active，凭证校验通过 |
| refresh | `active` | `used` | `POST /token-refreshes` | rotation 成功，写入 `replaced_by_jti` |
| refresh | `active` | `revoked` | 登出、用户禁用、软删除、安全撤销 | 当前 jti 或用户会话范围 |
| refresh | `active` | `expired` | token 校验或清理任务 | `expires_at <= now()` |

禁止迁移：

- `used`、`revoked`、`expired` 回到 `active`。
- refresh token 访问普通业务 API。
- access token 调用 refresh endpoint。
- JWT 中保存完整权限上下文。

幂等行为：

- 重复登出同一 session：如果 token 已 `revoked`，返回成功或 204。
- refresh token rotation 不幂等；同一 refresh token 第二次使用必须拒绝。

失败重试：

- 登录失败只增加失败计数和限流，不创建 token。
- refresh rotation 写旧 token `used` 和新 token `active` 必须同事务；事务失败时旧 token 仍可按原状态处理。

取消行为：

- 无取消状态；通过 `revoked` 表示主动失效。

补偿或阻断策略：

- 发现 `used` refresh token 被重放时，拒绝请求并可撤销同用户当前 active refresh token。
- 用户 `disabled`、`locked` 或 `deleted` 后，其 active access/refresh token 必须不可继续使用。

审计事件：

- `auth.login_succeeded`
- `auth.login_failed`
- `auth.session_revoked`
- `auth.refresh_rotated`
- `auth.refresh_replay_denied`
- `auth.token_expired`

## 7. 企业、组织、账号与角色绑定状态机

事实源：

- `enterprises.status`
- `users.status`
- `departments.status`
- `user_department_memberships.status`
- `role_bindings.status`
- `enterprises.permission_version`
- `enterprises.org_version`

状态枚举：

| 对象 | 状态 | 说明 |
| --- | --- | --- |
| enterprise | `active` | P0 默认唯一企业，承载 `org_version` 和 `permission_version` |
| enterprise | `disabled` | 受控运维态，阻断该企业所有业务流量 |
| enterprise | `deleted` | 软删除或离线清理终态，P0 不暴露普通入口 |
| user | `active` | 可登录并访问授权资源 |
| user | `disabled` | 管理员禁用，不能登录或继续使用 token |
| user | `locked` | 登录失败或安全策略锁定，可解锁 |
| user | `deleted` | 软删除终态，不可登录 |
| department | `active` | 可作为用户所属部门和资源归属部门 |
| department | `disabled` | 不再用于新增授权或资源归属 |
| department | `deleted` | 软删除终态 |
| membership | `active` | 用户属于该部门 |
| membership | `deleted` | 用户部门关系已删除 |
| role_binding | `active` | 角色绑定生效 |
| role_binding | `revoked` | 角色绑定已撤销 |

P0 边界：

- 初始化只创建一个默认 enterprise。
- 普通管理后台不提供第二个 enterprise 的创建、切换或跨 enterprise 迁移。
- `enterprises.status in ('disabled', 'deleted')` 仅用于受控运维或离线恢复，不作为日常业务流转入口。

允许迁移：

| 对象 | from | to | 触发方 | 条件 |
| --- | --- | --- | --- | --- |
| enterprise | none | `active` | setup 初始化 | 创建默认企业，`org_version=1`，`permission_version=1` |
| enterprise | `active` | `disabled` | 受控 CLI | P0 不暴露普通后台入口；禁用后阻断该企业业务 API 并撤销 active token |
| enterprise | `disabled` | `active` | 受控 CLI | 恢复后重新校验权限缓存、查询缓存和 active config |
| enterprise | `active/disabled` | `deleted` | 离线清理脚本 | 仅空系统或灾备恢复流程，且不存在在线流量 |
| user | none | `active` | 管理员 | 创建用户并写入凭证 |
| user | `active` | `disabled` | 管理员 | 禁用用户，撤销 active token |
| user | `disabled` | `active` | 管理员 | 重新启用，权限上下文可重建 |
| user | `active` | `locked` | 登录策略或管理员 | 登录失败超过阈值或手动锁定 |
| user | `locked` | `active` | 管理员 | 解锁用户并清理锁定计数 |
| user | `active/disabled/locked` | `deleted` | 管理员 | 软删除用户，撤销 token，保留审计 |
| department | none | `active` | 管理员 | 创建部门，递增 `org_version` |
| department | `active` | `disabled` | 管理员 | 不再允许新增成员或资源归属 |
| department | `disabled` | `active` | 管理员 | 恢复部门，递增 `org_version` |
| department | `active/disabled` | `deleted` | 管理员 | 无 active 主部门成员，或已完成成员迁移 |
| membership | none | `active` | 管理员 | 用户和部门均 active |
| membership | `active` | `deleted` | 管理员 | 移除成员或替换部门 |
| role_binding | none | `active` | 管理员 | 角色、用户和 scope 均有效 |
| role_binding | `active` | `revoked` | 管理员 | 撤销角色或替换绑定 |

禁止迁移：

- `deleted -> active`，用户和部门恢复必须新建或走后续 P1 恢复流程。
- 普通管理后台创建第二个 enterprise，或在 P0 运行期切换当前 active enterprise。
- `revoked -> active`，角色绑定恢复必须创建新绑定。
- 删除最后一个有效 `system_admin` 绑定。
- 删除默认部门前未指定新的默认部门或未迁移 active 成员。
- enterprise `disabled` 或 `deleted` 后继续允许该企业的 token、查询、导入或管理写操作。
- 用户 `disabled`、`locked` 或 `deleted` 后继续使用旧 access / refresh token。
- 组织、用户、角色绑定变更后不递增权限版本或不失效权限缓存。

幂等行为：

- enterprise 禁用、恢复和删除必须返回当前状态，不重复撤销 token 或重复失效缓存。
- 重复禁用、锁定、删除用户返回当前状态，不重复撤销 token。
- 重复替换用户部门或角色绑定时，以请求中声明的目标集合为准，缺失项转 deleted/revoked，新增项创建 active。
- 重复撤销同一 role_binding 返回 revoked 状态。

失败重试：

- enterprise 状态变更必须和企业级 token 撤销、缓存失效和审计同事务或同一受控步骤完成。
- 组织和权限变更必须在事务中更新状态、版本号和审计；事务失败后不得留下部分 active 绑定。
- token 撤销失败时，用户禁用或删除必须失败，避免旧 token 继续可用。

取消行为：

- P0 不提供用户、部门、角色绑定变更的异步取消。
- 替换部门或角色绑定一旦提交，只能通过新的替换请求纠正。

补偿或阻断策略：

- enterprise `disabled` 或 `deleted` 后必须撤销该企业 active access / refresh token，并失效查询缓存和权限缓存。
- 用户禁用、锁定、删除后必须撤销 active access / refresh token，并失效权限缓存。
- 部门删除或禁用后，相关资源归属和用户 membership 必须先迁移或拒绝操作。

## 8. Import Job 状态机

事实源：

- `import_jobs.status`
- `import_jobs.stage`
- `import_jobs.attempt_count`
- `import_jobs.locked_by`
- `import_jobs.locked_until`
- `import_jobs.next_retry_at`
- `import_jobs.cancel_requested_at`
- `import_jobs.cancel_requested_by`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `queued` | 等待 Worker 领取 |
| `running` | Worker 已领取，阶段处理中 |
| `retrying` | 失败后等待自动重试 |
| `partial_success` | 批量任务部分成功，终态 |
| `success` | 任务成功，终态 |
| `failed` | 超过重试次数或不可重试失败，终态 |
| `cancelled` | 用户或管理员取消，终态 |

阶段枚举：

`validate -> parse -> clean -> chunk -> embed -> index -> publish -> cleanup -> finished`

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `queued` | API 或系统任务 | 创建 upload、url、metadata_batch、index_rebuild、permission_refresh、index_delete 任务 |
| `queued` | `running` | Worker claim | `next_retry_at is null or <= now()`，锁为空或过期 |
| `retrying` | `running` | Worker claim | 到达 `next_retry_at` |
| `running` | `running` | Worker | 阶段向前推进或续租 lock |
| `running` | `running` | API 或管理员 | 仅写入 `cancel_requested_at/cancel_requested_by`，状态保持 `running` |
| `running` | `retrying` | Worker | retryable 错误且 `attempt_count < max_attempts` |
| `running` | `success` | Worker | 阶段到 `finished`，所有目标资源处理成功 |
| `running` | `partial_success` | Worker | 批量任务至少一个目标成功且至少一个目标失败 |
| `running` | `failed` | Worker | 不可重试错误或达到最大重试次数 |
| `queued` | `cancelled` | API | 尚未开始处理 |
| `retrying` | `cancelled` | API | 尚未重新处理 |
| `running` | `cancelled` | Worker | `cancel_requested_at is not null`，且当前阶段停在可安全中断的边界 |

禁止迁移：

- `success`、`partial_success`、`cancelled` 自动回到处理中。
- `failed -> running` 绕过 `queued`。
- `failed` 或 `cancelled` 记录被直接复用为“重试中”任务；手动重试必须创建新 job。
- API 直接把 `running` 任务写成 `cancelled`，绕过 Worker 阶段边界确认。
- 阶段倒退，例如 `embed -> chunk`，除非显式清理派生数据并记录 retry 摘要。
- 未持有有效 lock 的 Worker 更新状态。

幂等行为：

- 带相同 `idempotency_key` 的导入请求返回已有 job。
- Worker 阶段写入必须按内容 hash、version_no、index_version_id 去重。
- 重复取消终态任务返回当前状态，不创建额外清理任务。
- 对 `running` 任务重复发起取消请求时，返回当前 job 和已有 `cancel_requested_*` 标记，不重复写终态。

失败重试：

- retryable 阶段：依赖超时、外部模型 provider 临时不可用、对象存储临时失败、Worker crash。
- non-retryable 阶段：schema 不合法、权限失败、文件类型不支持、embedding 维度不匹配。
- Worker crash 后，其他 Worker 只能在 `locked_until < now()` 后接管。
- 手动重试必须创建新的 `queued` job；原 `failed` / `cancelled` job 保持只读，可通过请求摘要或审计关联到新 job。

取消行为：

- `queued` 和 `retrying` 可立即取消。
- `running` 只允许写入取消意图；Worker 必须在阶段边界确认后再转 `cancelled`，已写入的安全阻断、document_version 或 index_version 不得被粗暴删除。
- 如果取消请求到达时当前阶段已经跨过 publish、block 或 delete 的不可逆边界，Worker 必须先完成当前一致性步骤，再以 `success` 或 `failed` 结束，而不是强行转 `cancelled`。
- `index_delete` 和权限收紧任务不允许普通用户取消，只允许系统或管理员处理失败。

补偿或阻断策略：

- 文档删除、权限收紧和 index_delete 任务失败时，access block 必须保持 active。
- draft index 发布失败时，旧 active index 继续服务。
- 导入任务失败不得让 draft chunk 或 draft index 进入查询。
- 运行中任务收到取消请求后，不得回滚已提交的安全阻断或已完成的 active index 切换。

审计事件：

- `import_job.created`
- `import_job.claimed`
- `import_job.stage_advanced`
- `import_job.retry_scheduled`
- `import_job.cancel_requested`
- `import_job.failed`
- `import_job.cancelled`
- `import_job.succeeded`
- `import_job.partial_succeeded`

## 10. Document Lifecycle 与 Index 状态机

事实源：

- `documents.lifecycle_status`
- `documents.index_status`
- `documents.current_version_id`
- `documents.permission_snapshot_id`
- `access_blocks`

生命周期状态：

| 状态 | 说明 |
| --- | --- |
| `draft` | 元数据已创建，尚无可查询 active index |
| `active` | 文档可被权限过滤查询命中 |
| `archived` | 文档保留但不可被普通查询命中 |
| `deleted` | 软删除终态，不可查询 |

索引状态：

| 状态 | 说明 |
| --- | --- |
| `none` | 尚未开始索引 |
| `indexing` | 首次索引处理中 |
| `indexed` | 存在 active index，可查询还需满足权限和 access block |
| `index_failed` | 首次索引失败，可重试 |
| `blocked` | 查询已阻断，通常来自删除或安全阻断 |

可见性枚举（P0 仅两种）：

| visibility | 说明 |
| --- | --- |
| `department` | 仅归属部门成员可见 |
| `enterprise` | 企业内授权用户可见 |

允许迁移：

| lifecycle from | index from | lifecycle to | index to | 触发方 | 条件 |
| --- | --- | --- | --- | --- | --- |
| none | none | `draft` | `none` | 导入 API | document 元数据创建成功 |
| `draft` | `none` | `draft` | `indexing` | Worker | import_job 开始处理 |
| `draft` | `indexing` | `draft` | `index_failed` | Worker | 首次索引失败 |
| `draft` | `index_failed` | `draft` | `indexing` | Worker 或手动重试 | 重试任务创建 |
| `draft` | `indexing` | `active` | `indexed` | Worker | document_version active，index_version active，chunks active |
| `draft` | `none/indexing/index_failed` | `deleted` | `blocked` | 管理员或系统 | 先写 active access block |
| `active` | `indexed` | `archived` | `indexed` | 管理员 | 归档后普通查询不可见 |
| `archived` | `indexed` | `active` | `indexed` | 管理员 | 重新启用，必须重新校验权限快照、access block、active index 和查询缓存 |
| `active` | `indexed` | `deleted` | `blocked` | 管理员 | 先写 active access block，再异步清理索引 |
| `archived` | `indexed` | `deleted` | `blocked` | 管理员 | 先写 active access block |

可见性与权限快照迁移：

| from visibility | to visibility | 触发方 | 最小顺序 |
| --- | --- | --- | --- |
| `department` | `enterprise` | 管理员 | 创建新 `permission_snapshot` 和 `permission_refresh` job -> 构建新 draft refs / payload -> 新 refs active 且 `indexed_permission_version` 达到目标版本 -> 更新 `documents.visibility` 和 `permission_snapshot_id` -> 失效查询缓存 -> 写审计 |
| `enterprise` | `department` | 管理员 | 先写 `access_block(reason='permission_tightened')` -> 创建新 `permission_snapshot` 和 `permission_refresh` job -> 阻断旧 active refs -> 发布新受限 refs -> 更新 `documents.visibility` 和 `permission_snapshot_id` -> 失效查询缓存 -> 仅在旧 refs 不可见且新 refs active 后释放临时 access block |
| `department` | `department` | API / 管理员 | 目标一致，按幂等返回 |
| `enterprise` | `enterprise` | API / 管理员 | 目标一致，按幂等返回 |

说明：

- `visibility` 变更不改变 `documents.lifecycle_status`；它通过 `permission_snapshot`、`permission_version` 和索引引用轮换实现。
- 对 `draft` 或 `index_status!='indexed'` 的文档，可同步更新 `visibility` 和 `permission_snapshot_id`，但仍必须记录审计。

禁止迁移：

- `deleted` 回到任意非 deleted 状态。
- `active + indexed` 之前没有 active document_version 和 active index_version。
- `deleted + indexed` 或 `active + blocked` 被查询链路放行。
- 删除文档时先物理删索引再写 access block。
- 直接 `UPDATE documents.visibility` 而不刷新 `permission_snapshot`、`permission_version`、索引 payload 或 access block。
- `enterprise -> department` 收紧时，在旧 refs 仍可见或新 refs 未激活前释放 `permission_tightened` access block。

幂等行为：

- 重复删除文档：如果已有 active access block 且文档为 `deleted/blocked`，返回删除已受理。
- 重复归档或恢复 active：目标状态一致时返回当前文档。
- 重复把文档可见性改到相同目标时，返回当前文档或当前活跃的 `permission_refresh` job，不重复创建刷新任务。

失败重试：

- `index_failed` 可通过导入重试或索引重建任务回到 `indexing`。
- active 文档重建索引时，`documents.index_status` 保持 `indexed`，新 `index_versions` 使用 draft/ready 状态；失败不影响旧 active index。
- `department -> enterprise` 扩大可见性失败时，保持旧 `visibility`、旧 `permission_snapshot` 和旧 refs 继续服务。
- `enterprise -> department` 收紧失败时，`permission_tightened` access block 必须保持 active，直到新受限 refs 发布成功或管理员显式发起新的恢复操作。

取消行为：

- 首次导入任务取消时，draft 文档可以转 `deleted/blocked` 或保留 `draft/index_failed`，由取消策略决定。
- active 文档的 index_rebuild 取消不改变当前 `active/indexed` 文档状态。
- `permission_refresh` 任务沿用 import_job 取消规则；权限收紧产生的刷新任务不允许普通用户取消。

补偿或阻断策略：

- 删除最小事务：写 active access block -> 文档 `deleted/blocked` -> chunks `deleted` -> chunk_index_refs `blocked` -> 创建 `index_delete` job -> 写审计。
- 查询只允许 `lifecycle_status='active' and index_status='indexed'`，并且不存在 active access block。
- 权限收紧最小事务：写 `permission_tightened` access block -> 递增 `permission_version` 并创建新 `permission_snapshot` -> 阻断旧 refs -> 发布新 refs -> 更新文档 `visibility` / `permission_snapshot_id` -> 失效缓存 -> 释放临时 access block。
- 可见性扩大最小事务：保留旧 refs 服务 -> 构建并发布新 refs -> 更新文档 `visibility` / `permission_snapshot_id` -> 失效缓存。

审计事件：

- `document.created`
- `document.activated`
- `document.index_failed`
- `document.archived`
- `document.restored`
- `document.deleted`
- `document.visibility_expanded`
- `document.permission_tightened`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `draft` | 版本记录已创建，原始对象可用或待上传 |
| `parsed` | 解析产物已写入对象存储 |
| `chunked` | chunk 已生成，状态为 draft |
| `indexed` | draft index 已写入并通过基础校验 |

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |

| `draft` | `parsed` | Worker parse | 解析产物 hash 和 object key 已记录 |
| `parsed` | `chunked` | Worker chunk | chunks 创建完成，数量可对账 |
| `chunked` | `indexed` | Worker index | draft vector 和 keyword index 写入完成 |
| `indexed` | `active` | Worker publish | index_version active，permission snapshot 匹配 |
| `active` | `archived` | Worker publish 新版本 | 新版本 active 同事务替换 |
| `draft/parsed/chunked/indexed` | `failed` | Worker | 当前阶段失败 |

禁止迁移：

- `archived -> active` 作为配置式回滚；P0 通过重新导入或新版本处理。
- `failed -> active`。
- 多个 active document_version 同时存在。
- document_version active 但对应 document 未 `active/indexed`。

幂等行为：

- 同一 `content_hash` 和 `version_no` 重复导入返回已有版本或创建新 job 指向已有版本。
- 阶段重复执行必须按 object key、chunk ordinal、payload hash 去重。
- 对已 `failed` 的版本重复发起重试时，必须创建新的 `document_version` 和新 job；原失败版本保持只读。

失败重试：

- 解析、清洗、切块、embedding、索引均可在清理派生数据后重试。
- 如果失败发生在 publish 前，不影响旧 active version。
- 手动重试不得把 `failed` 版本改回 `draft`；应创建新版本号或新的受控版本记录。

取消行为：

- 非 active 版本可以随导入任务取消转 `failed` 或保持 `draft`，但不得被查询命中。
- active 版本不允许取消，只能被新版本替换为 archived 或随文档删除被阻断。

补偿或阻断策略：

- publish 前必须校验 chunk 数量、向量维度、payload hash、权限快照和 draft 不可见。
- active 切换必须同事务更新 `documents.current_version_id`。

审计事件：

- `document_version.created`
- `document_version.parsed`
- `document_version.chunked`
- `document_version.indexed`
- `document_version.activated`
- `document_version.failed`

## 12. Chunk 生命周期状态机

事实源：

- `chunks.status`
- `chunks.permission_snapshot_id`
- `document_versions.status`
- `documents.current_version_id`
- `documents.lifecycle_status`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `draft` | chunk 已生成，但尚未跟随 active 版本发布 |
| `active` | 可被 active document 和 active index 引用 |
| `archived` | 所属文档被归档，或所属版本被新版本替代后保留 |
| `deleted` | 逻辑删除终态，不可查询 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `draft` | Worker chunk | `document_version.status in ('parsed', 'chunked')`，按 `document_version_id + ordinal` 创建 |
| `draft` | `active` | Worker publish | 所属 `document_version` 成为 active，`documents.current_version_id` 指向该版本，且 `permission_snapshot_id` 已写入 |
| `active` | `archived` | Worker publish 新版本或管理员归档文档 | 所属 `document_version` 被替代，或 document `active -> archived` |
| `archived` | `active` | 管理员恢复文档 | 仅当 chunk 属于 `documents.current_version_id` 且 document `archived -> active` |
| `draft/active/archived` | `deleted` | 管理员、系统或清理任务 | 文档删除、知识库删除、draft 清理或取消清理；高风险删除前必须已有 active access block |

禁止迁移：

- `deleted -> draft/active/archived`。
- 被新版本替代的旧 version chunk `archived -> active`；恢复必须通过新版本重新发布，而不是复活旧 chunk。
- `active` chunk 对应 `document_version` 非 active，或 `document.lifecycle_status != 'active'`。
- 权限收紧时直接把 chunk 改成 `deleted` 或 `archived` 代替 permission snapshot / index ref 轮换。

幂等行为：

- 按 `document_version_id + ordinal` 重复生成 chunk 时返回已有记录，不重复插入。
- 重复删除已 `deleted` chunk 返回当前状态。
- 重复归档或恢复文档时，chunk 目标状态一致则按幂等返回。

失败重试：

- chunk 生成失败后，允许清理 draft chunk 并通过新 job / 新 version 重新生成。
- 删除后的对象存储派生数据清理失败，不得把 chunk 从 `deleted` 恢复。

取消行为：

- draft chunk 可随导入任务取消而清理为 `deleted`。
- active chunk 不支持单独取消，只能随文档归档转 `archived` 或随文档删除转 `deleted`。

补偿或阻断策略：

- 查询只允许 `chunks.status='active'`，并且其 document、document_version、index_version、chunk_index_ref 都处于允许状态。
- 文档恢复时，只能恢复 `documents.current_version_id` 下的 archived chunk；旧 superseded 版本保持 archived。

审计事件：

- `chunk.created`
- `chunk.activated`
- `chunk.archived`
- `chunk.deleted`

## 13. Index Version 状态机

事实源：

- `index_versions.status`
- `chunk_index_refs.visibility_state`
- `keyword_index_entries.visibility_state`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `draft` | draft 向量和关键词索引写入中，不可查询 |
| `ready` | 发布前校验通过，等待激活 |
| `active` | 当前可服务查询的索引版本 |
| `archived` | 被新 active index 替代，不再用于查询 |
| `pending_delete` | 等待异步物理清理 |
| `failed` | 构建或校验失败 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `draft` | Worker | 创建索引版本 |
| `draft` | `ready` | Worker | draft 索引写入完成且校验通过 |
| `draft` | `failed` | Worker | 写入或校验失败 |
| `ready` | `active` | Worker publish | 同文档不存在其他 active，或同事务归档旧 active |
| `ready` | `failed` | Worker | 发布前复检失败 |
| `active` | `archived` | Worker publish 新版本 | 新 active index 已发布 |
| `archived` | `pending_delete` | 清理任务 | 已不再被查询引用 |
| `failed` | `pending_delete` | 清理任务 | 清理失败草稿外部资源 |

禁止迁移：

- `failed -> active`。
- `archived -> active` 作为回滚。
- `pending_delete -> active`。
- draft 或 ready index 被查询链路使用。
- 同一 document 存在多个 active index_version。

幂等行为：

- 重复 publish 同一 ready index：如果已 active，返回当前 active 摘要。
- 重复 cleanup pending_delete：外部资源不存在时视为成功。

失败重试：

- index 构建失败时创建新的 index_version 或清理失败 draft 后重建。
- active 文档的 index_rebuild 失败不影响旧 active index。

取消行为：

- draft 或 ready index 可被取消并转 `failed` 或 `pending_delete`。
- active index 不允许取消，只能被替代后归档。

补偿或阻断策略：

- 发布顺序：ready 校验 -> 旧 active archived -> 新 ready active -> chunk refs active -> document indexed。
- 删除或权限收紧必须先 block refs，再异步 pending_delete。

审计事件：

- `index_version.created`
- `index_version.ready`
- `index_version.activated`
- `index_version.archived`
- `index_version.failed`
- `index_version.cleanup_requested`

## 14. Chunk Index Visibility 状态机

事实源：

- `chunk_index_refs.visibility_state`
- `keyword_index_entries.visibility_state`
- `chunks.status`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `draft` | 索引引用已创建，但 draft 不可见 |
| `active` | 可参与向量和关键词召回 |
| `blocked` | 被删除、权限收紧或安全事件阻断 |
| `deleted` | 外部索引物理清理完成或逻辑删除完成 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `draft` | Worker index | 写入 draft index ref |
| `draft` | `active` | Worker publish | index_version active，permission snapshot 校验通过，且 `chunks.status='active'` |
| `draft` | `deleted` | 清理任务 | draft 构建失败或取消 |
| `active` | `blocked` | 权限收紧、文档删除、知识库删除、安全阻断 | 先写 active access block 或同事务写阻断 |
| `blocked` | `deleted` | index_delete job | Qdrant point、关键词索引和对象存储派生数据清理完成 |

禁止迁移：

- `blocked -> active`；权限恢复必须创建新的 permission snapshot 和新的 active ref。
- `deleted -> active`。
- `draft` 或 `blocked` 进入查询上下文。

幂等行为：

- 重复 block active ref：目标已 `blocked` 时返回成功。
- 重复删除外部索引：外部资源不存在时视为成功。

失败重试：

- block 成功但物理删除失败时保持 `blocked`，index_delete job retry。
- 关键词索引和向量索引任何一侧失败都不得释放 blocked 状态。

取消行为：

- 安全阻断和删除清理任务不可由普通用户取消。
- draft index 取消只能进入 `deleted` 或随 index_version failed 清理。

补偿或阻断策略：

- 向量召回必须检查 `chunk_index_refs.visibility_state='active'`；关键词召回额外要求 `keyword_index_entries.visibility_state='active'`。
- `indexed_permission_version` 落后时必须回源确认或拒绝。

审计事件：

- `chunk_index_ref.created`
- `chunk_index_ref.activated`
- `chunk_index_ref.blocked`
- `chunk_index_ref.deleted`
- `keyword_index_entry.blocked`

## 15. Config Version 状态机

事实源：

- `config_versions.status`
- `system_configs.status`
- `system_state.active_config_version`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `draft` | 可编辑配置草稿 |
| `validating` | 正在执行 schema、Secret ref、依赖和安全策略校验 |
| `active` | ServiceBootstrap 当前使用的 active config |
| `archived` | 被新 active config 替代或废弃 |
| `failed` | 校验或发布失败 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `draft` | 初始化或管理后台保存配置 | 配置 schema 初步可解析 |
| `draft` | `validating` | 配置校验或发布 | 锁定该版本，防止并发编辑 |
| `validating` | `draft` | 仅校验成功 | 写入 validation_result，不发布 |
| `validating` | `active` | 发布成功 | schema、依赖、安全策略校验通过 |
| `validating` | `failed` | 校验或依赖测试失败 | 写入脱敏失败摘要 |
| `active` | `archived` | 新版本发布 | 与新 active 版本同事务 |
| `draft` | `archived` | 放弃草稿 | 无 active 指针引用 |

禁止迁移：

- `archived -> active` 作为通用配置回滚。
- `failed -> active`；必须创建新 draft 或重新保存草稿后再校验。
- `draft -> active` 绕过 `validating`。
- 多个 active config_version 同时存在。
- active config 中保存 secret value。

幂等行为：

- 重复保存同一配置 hash 返回已有 draft 或创建新版本策略由 Config Service 固定；P0 推荐相同 hash 返回已有 draft。
- 重复发布已 active 的同一版本返回当前 active 摘要。

失败重试：

- 校验失败后可创建新 draft 重试。
- dependency 临时失败可重新从 draft 进入 validating；failed 版本只读保留。

取消行为：

- P0 不提供 validating 取消接口；超时由后台任务转 failed。
- draft 可归档，active 不可直接取消。

补偿或阻断策略：

- 发布事务必须同时更新 `config_versions.status`、`system_configs.status` 和 `system_state.active_config_version`。
- 发布失败不得改变当前 active config。
- P0 不实现完整审批、双人复核或通用回滚平台。

审计事件：

- `config.draft_saved`
- `config.validation_started`
- `config.validation_failed`
- `config.validated`
- `config.reload_failed`
- `config.published`
- `config.archived`

## 16. Access Block 生命周期

事实源：

- `access_blocks.status`
- `access_blocks.reason`
- `access_blocks.block_level`
- `access_blocks.expires_at`

状态枚举：

| 状态 | 说明 |
| --- | --- |
| `active` | 查询、citation 或全部访问被阻断 |
| `released` | 阻断释放，历史记录保留 |

允许迁移：

| from | to | 触发方 | 条件 |
| --- | --- | --- | --- |
| none | `active` | 删除、权限收紧、安全事件、legal hold | 写入 resource_type、resource_id、reason、block_level |
| `active` | `released` | 系统或管理员 | 仅限非删除原因；安全条件满足，且不存在需要继续阻断的旧索引引用 |

禁止迁移：

- `released -> active` 复用旧记录；需要重新阻断时创建新 access_block。
- 删除原因的 access block 在 P0 不得自动释放。
- 物理删除失败时不得释放 access block，必须保持 `active` 并等待清理任务重试或人工处置。

幂等行为：

- 对同一资源、reason、block_level 重复创建 active block 时返回已有 active block 或保持单条有效记录。
- 释放已 released block 返回当前状态。

失败重试：

- access block 写入失败时，删除、权限收紧或安全阻断操作必须整体失败。
- 下游索引清理失败不影响 active block。

取消行为：

- active block 不支持普通用户取消。
- 管理员释放必须先校验新权限快照、active index 和索引引用都已安全；删除原因的阻断在 P0 不提供释放入口。

补偿或阻断策略：

- 查询和 citation 校验必须检查 knowledge_base、document、chunk 三层 active block。
- 删除原因的 active block 是安全边界，不能因为 Qdrant、关键词索引或对象存储物理清理完成而自动释放。
- `expires_at` 仅用于临时阻断；过期释放需要后台任务显式转 `released` 或查询条件排除过期 block。

审计事件：

- `access_block.created`
- `access_block.released`
- `access_block.release_denied`

## 17. 跨状态机不变量

| 不变量 | 要求 |
| --- | --- |
| 初始化门禁 | `initialized=false` 时只允许 setup 和 healthcheck；`initialized=true` 后 setup 写接口关闭 |
| 单企业边界 | P0 只允许一个默认 enterprise 进入运行态；普通管理后台不暴露第二个 enterprise 的创建、切换或迁移 |
| 查询可见性 | 只允许 active knowledge_base、active document、active chunk、active index、active index ref、无 active access block |
| draft 不可见 | draft document_version、draft index_version、draft chunk_index_ref 不得进入召回或 citation |
| 删除 fail closed | 文档、chunk、知识库删除必须先写 access block，再异步清理索引和对象 |
| 权限收紧 fail closed | 先写 access block 或阻断旧 index ref，再刷新权限快照和索引 payload |
| 运行中取消 | `running` import_job 在 Worker 阶段边界确认取消前保持 `running`；取消意图必须持久化 |
| 账号和组织变更 | 用户、部门、membership、role_binding 变化必须失效 token 或权限缓存并递增对应版本 |
| chunk 一致性 | active chunk 必须属于当前 document 的 active version；superseded version 的 archived chunk 不得重新激活 |
| active 唯一 | setup token、config version、document version、index version 的 active 唯一约束必须由 DB partial unique index 和事务共同保证 |
| 旧索引安全 | index rebuild 失败不影响旧 active index；新 index 只有 ready 校验通过后才能发布 |
| token 安全 | refresh/setup token 一次性使用，旧 token 不得复活 |
| 审计脱敏 | 状态迁移审计不得记录 password、token、secret value、完整 prompt 或未脱敏原文 |

## 18. API 与 Worker 实现约束

- `PATCH status` 类接口必须调用状态机 transition 方法，并校验 from 状态。
- Worker 领取任务必须基于 `locked_until` 和 CAS 更新，不能靠内存锁。
- Worker 每个阶段完成后必须持久化 stage、status、attempt_count、error_code 和脱敏 error_message。
- 运行中导入任务取消必须先持久化 `cancel_requested_*`，由 Worker 在阶段边界完成 `running -> cancelled`；API 不得直接写终态。
- 管理后台删除、权限收紧和配置发布属于高风险迁移，必须有审计和确认条件。
- 文档 `visibility` 变更必须调用 permission refresh 迁移方法，不能直接 `UPDATE documents.visibility`。
- 查询服务不得依赖 UI、缓存或 LLM 判断权限，必须执行数据库状态和权限过滤。
- Repository 层应提供 `transition_<entity>_<action>` 方法，并集中维护允许迁移表。
- P0 enterprise 生命周期只允许 setup 创建默认 enterprise；其他 enterprise 变更走受控 CLI 或离线流程。

## 19. P0 状态机测试要求

必须覆盖：

- 初始化前普通业务 API 被拒绝，初始化后 setup 写接口被拒绝。
- setup JWT 签发新 token 后旧 active token 立即 revoked。
- refresh token rotation 后旧 refresh token 再次使用被拒绝。
- P0 运行期不存在第二个 active enterprise；普通后台不能创建或切换 enterprise。
- 用户禁用、锁定或软删除后旧 access / refresh token 不可继续使用。
- 部门、membership、role_binding 变化后权限缓存失效，旧权限上下文不可复用。
- import_job queued/running/retrying/success/failed/cancelled 的合法迁移和非法迁移。
- running import_job 收到取消请求后先写 `cancel_requested_*`，只有 Worker 阶段边界才能转 `cancelled`。
- Worker crash 后 `locked_until` 过期可接管任务。
- disabled / archived / deleted knowledge_base 不能被普通查询命中。
- folder 删除非空目录时不得静默级联删除文档；涉及文档删除必须写 document 级 access block。
- draft document、draft chunk、draft index、blocked index ref 不能被查询命中。
- active 文档 index rebuild 失败时旧 active index 继续服务。
- 文档删除先写 active access block，再进入 deleted/blocked。
- `department -> enterprise` 扩大可见性后，新权限范围用户可见且旧缓存失效。
- 权限收紧后旧用户不能通过旧索引、旧缓存或旧 permission_version 访问。
- `enterprise -> department` 收紧期间 access block 保持 active，直到新受限 refs 发布完成。
- config_version 不允许 archived -> active，不实现通用配置回滚。
- access block active 时 query 和 citation 均被阻断。
- 删除原因的 access block 在物理索引清理失败或成功后都不得自动释放。
