# Little Bear RAG 项目执行流程图

本文基于当前设计文档整理项目总执行流程和各功能模块流程，用于后续实现、评审、联调和测试对齐。

当前项目设计主线是企业内部高并发 RAG 后端，采用“模块化单体 + 轻量 Worker”。核心约束包括：

- 权限过滤必须下推到向量检索和关键词检索。
- 导入链路异步化，HTTP 请求只创建任务。
- API、Worker 和 Model Gateway 管理进程只从启动配置读取数据库连接，业务配置来自数据库 active config。
- 查询链路支持 rewrite、embedding、rerank、LLM 等阶段降级。
- 查询答案必须带引用，引用必须可回溯到文档、页码和 chunk。
- 查询、导入、权限、配置、模型调用必须具备日志、指标、Trace 和审计。

## 1. 项目总执行流程

```mermaid
flowchart TD
    A[客户端/门户/IM Bot/API Client] --> B[API Gateway/WAF/限流]
    B --> C[FastAPI Middleware]
    C --> D{系统是否已初始化}
    D -- 否 --> E[Setup Service: setup-state/config-validations/initialization]
    E --> F[写入 system_state + active_config v1]
    F --> G[ServiceBootstrap 初始化依赖]
    D -- 是 --> H[Auth Service 鉴权]
    G --> H
    H --> I[Config Service 读取 active config]
    I --> J[Permission Service 构建权限上下文]

    J --> K{请求类型}
    K -- 查询 --> L[Query Service]
    L --> M[Retrieval: rewrite/embedding/召回/融合/rerank]
    M --> N[候选片段准入校验: 权限/索引/access_block]
    N --> O[Context Service 组装/压缩上下文]
    O --> P[Answer Service: Prompt + LLM + 引用校验]
    P --> Q[Audit/Logs/Metrics/Trace]
    Q --> R[返回答案/引用/trace_id]

    K -- 导入 --> S[Import Service 创建 import_job/document]
    S --> T[ObjectStorage 保存原始文件]
    T --> U[Import Worker 领取任务]
    U --> V[parse/clean/chunk/embed]
    V --> W[Indexing Service 写 draft 索引]
    W --> X[校验 draft 不可见 + 权限 payload]
    X --> Y[发布 active index version]
    Y --> Q
```

## 2. Setup 初始化模块

```mermaid
flowchart TD
    A[进程启动] --> B[读取数据库连接配置]
    B --> C{PostgreSQL 可连接?}
    C -- 否 --> D[readiness=false/启动失败]
    C -- 是 --> E[读取 system_state]
    E --> F{initialized 且 active_config 存在?}
    F -- 否 --> G[生成新的 setup JWT 并失效旧 token]
    G --> H[只开放 setup-state/config-validations/initialization]
    H --> I[validate 配置和依赖]
    I --> J[initialize 事务]
    J --> K[创建首个管理员/默认角色/active_config v1]
    K --> L[ServiceBootstrap 初始化 Redis/MinIO/Qdrant/模型网关]
    L --> M[开放业务 API]
    F -- 是 --> L
```

关键点：

- 数据库连接失败时不进入 setup mode。
- 未初始化时只开放 setup-state、setup-config-validations、setup-initialization。
- 初始化成功且 ServiceBootstrap 全部通过后才开放普通业务 API。
- setup JWT 只能访问初始化接口，不能访问普通业务 API 或管理员 API。

## 3. Auth 认证模块

```mermaid
flowchart TD
    A[登录请求 username/password] --> B[检查系统 initialized]
    B --> C[读取 active auth config]
    C --> D[查询用户]
    D --> E{用户 active?}
    E -- 否 --> F[返回 AUTH_USER_DISABLED]
    E -- 是 --> G[校验密码 hash]
    G --> H{密码正确且未锁定?}
    H -- 否 --> I[记录失败次数/审计]
    H -- 是 --> J[签发 access/refresh JWT]
    J --> K[写登录审计]
    K --> L[返回 token]
```

关键点：

- Auth Service 只负责账号、密码、会话和 JWT。
- JWT 不保存完整权限上下文。
- 资源访问权限由 Permission Service 构建和校验。

## 4. Org 组织模块

```mermaid
flowchart TD
    A[管理员/同步任务提交组织数据] --> B[校验企业/部门/成员关系]
    B --> C[写 staging 或业务表]
    C --> D[生成 diff]
    D --> E{是否发布}
    E -- 否 --> F[等待确认]
    E -- 是 --> G[更新 departments/memberships]
    G --> H[递增 org_version]
    H --> I[失效用户 subject/权限缓存]
    I --> J[供 Permission Service 构建上下文]
```

关键点：

- 最小生产阶段组织结构收敛为企业、部门、成员三层。
- 部门不建模上下级递归。
- 查询权限只使用用户直接所属部门和企业 ID。

## 5. Permission 权限模块

```mermaid
flowchart TD
    A[Auth 已确认用户身份] --> B[Org Service 获取用户部门]
    B --> C[读取角色/资源授权]
    C --> D[计算 scopes]
    D --> E[读取 permission_version]
    E --> F[生成 PermissionContext]
    F --> G[生成 PermissionFilter]
    G --> H[下推到 Qdrant/关键词检索/元数据查询]
    H --> I[查询后再次执行 access_block/权限快照 gate]
```

关键点：

- 权限由 RBAC 和文档可见性共同构成。
- 文档可见性只支持 `department` 和 `enterprise`。
- 权限过滤条件必须同时用于向量召回、关键词召回和元数据召回。
- 索引权限版本落后时必须拒绝候选或回源确认，不能直接放行。

## 6. Config 配置模块

```mermaid
flowchart TD
    A[管理员修改配置] --> B[保存 draft]
    B --> C[schema 校验]
    C --> D[依赖连通性测试]
    D --> E[生成 diff 和 risk_level]
    E --> F{高风险?}
    F -- 是 --> G[审批]
    F -- 否 --> H[发布 active version]
    G --> H
    H --> I[写 config_change_event]
    I --> J[Redis 通知或 DB 轮询]
    J --> K[各模块热更新配置]
    K --> L[写审计]
```

关键点：

- 业务模块不得直接从环境变量读取 Redis、MinIO、Qdrant、模型服务或业务策略。
- 高风险配置必须审批。
- 冷启动时 active config 不可用必须拒绝服务。
- 运行期热更新失败时，可以使用最近一次已加载成功的 active config。

## 7. Import 导入模块

```mermaid
flowchart TD
    A[上传/导入请求] --> B[Gateway 鉴权/限流/大小检查]
    B --> C[Permission 校验导入权限和 visibility]
    C --> D[Import Service 校验类型/大小/幂等键]
    D --> E[创建 import_job/document/permission_snapshot]
    E --> F[原始文件写入 MinIO]
    F --> G[返回 job_id]
    G --> H[Worker 领取 queued/retrying 任务]
    H --> I[validate]
    I --> J[parse]
    J --> K[clean]
    K --> L[chunk]
    L --> M[Model Gateway batch embedding]
    M --> N[Indexing 写 draft 索引]
    N --> O[publish active index]
    O --> P[任务 success/审计/指标]
```

关键点：

- HTTP 请求只创建导入任务并返回 `job_id`。
- Worker 基于 PostgreSQL 任务表领取任务。
- 每个阶段独立推进状态，支持重试、续约和崩溃接管。
- 发布前 `document.index_status` 不能变为 `indexed`，`chunk_index_refs.visibility_state` 必须保持 `draft`。

## 8. Indexing 索引模块

```mermaid
flowchart TD
    A[接收 chunks/embeddings/permission_snapshot] --> B[创建 draft index_version]
    B --> C[生成确定性 vector_id/keyword_id]
    C --> D[写 Qdrant draft payload]
    C --> E[写 PostgreSQL Full Text draft]
    D --> F[写 chunk_index_refs draft]
    E --> F
    F --> G[校验数量/维度/权限 payload/hash]
    G --> H[验证 draft 索引不可见]
    H --> I[标记 ready]
    I --> J[事务发布 active]
    J --> K[document/chunk/index_ref 改为 active]
    K --> L[旧版本 archived/pending_delete]
    L --> M[失效查询缓存/写审计]
```

关键点：

- Qdrant 和关键词索引必须写入同一套权限字段。
- draft 和 ready 索引不得被查询命中。
- `chunk_index_refs` 是索引事实账本。
- 权限收紧或删除时必须先阻断查询，再异步物理删除索引。

## 9. Query 查询入口模块

```mermaid
flowchart TD
    A[POST /queries 或 /query-streams] --> B[SetupGuard]
    B --> C[Auth 鉴权]
    C --> D[RateLimit]
    D --> E[校验请求参数]
    E --> F[读取 active config]
    F --> G[获取 active index versions]
    G --> H[Permission 构建上下文和过滤条件]
    H --> I[Retrieval 编排]
    I --> J[候选片段准入校验]
    J --> K[Context Service]
    K --> L[Answer Service]
    L --> M[答案引用有效性校验]
    M --> N[Safety Filter]
    N --> O[写 query_log/审计/指标]
    O --> P[返回答案或 SSE]
```

关键点：

- 查询请求必须带 request context，包括 `request_id`、`trace_id`、`config_version`、`permission_version` 和 active index 版本信息。
- 查询缓存命中后仍要执行 access block 和引用有效性轻量校验。
- 流式响应开始前必须完成权限过滤、active index、access block 和候选引用来源准入校验。
- 严格引用模式下，不能输出未经引用校验的最终 token。

## 10. Retrieval / Context / Answer 核心 RAG 模块

```mermaid
flowchart TD
    A[QueryContext + PermissionFilter] --> B[rewrite]
    B --> C[expansion]
    C --> D[query embedding]
    D --> E[向量召回]
    C --> F[关键词召回]
    C --> G[元数据/FAQ 召回]
    E --> H[权限和 active index 过滤]
    F --> H
    G --> H
    H --> I[去重/归一化/RRF 融合]
    I --> J[rerank]
    J --> K[候选片段准入校验]
    K --> L[Context Builder]
    L --> M[Context Compressor]
    M --> N[Prompt Builder]
    N --> O[LLM 生成]
    O --> P[引用校验]
    P --> Q[脱敏/安全后处理]
    Q --> R[最终响应]
```

关键点：

- rewrite 失败时回退原始查询，不能覆盖用户显式过滤条件。
- expansion 不能扩大权限范围。
- 向量召回、关键词召回、元数据召回都必须下推权限过滤和 active index 可见性过滤。
- rerank 超时后使用融合分数降级。
- LLM 超时时返回检索结果和引用。

## 11. Model Gateway 模型网关

```mermaid
flowchart TD
    A[业务模块调用模型 API] --> B[服务间认证]
    B --> C[读取模型路由配置]
    C --> D{调用类型}
    D -- query embedding --> E[在线 embedding 池]
    D -- batch embedding --> F[离线 embedding 池]
    D -- rerank --> G[Rerank 服务]
    D -- LLM --> H[vLLM/LLM 服务]
    E --> I[记录 latency/token/model_route_hash]
    F --> I
    G --> I
    H --> I
    I --> J{超时/过载?}
    J -- 是 --> K[熔断/降级原因]
    J -- 否 --> L[返回模型结果]
```

关键点：

- 业务模块只调用 Model Gateway，不直接调用 vLLM、TEI 或外部模型 SDK。
- 在线 embedding 和离线 batch embedding 分池。
- 每次路由必须输出 `model_route_hash`，用于查询日志和缓存 key。
- embedding 模型版本、维度和 normalize 配置必须和索引兼容。

## 12. Audit / Observability 审计可观测模块

```mermaid
flowchart TD
    A[业务事件/请求阶段] --> B[注入 request_id/trace_id]
    B --> C[结构化日志]
    B --> D[Trace Span]
    B --> E[Metrics]
    B --> F[Audit Event]
    F --> G[脱敏 Redactor]
    G --> H[写 audit_logs]
    C --> I[日志采集]
    D --> J[链路追踪]
    E --> K[指标/告警]
    H --> L[审计查询/质量评测]
```

关键点：

- 普通日志禁止记录密码、token、secret value、完整 prompt、文档原文和未脱敏个人信息。
- Prompt 相关日志默认只记录模板 ID、模板版本、prompt hash、变量摘要和模型版本。
- 查询、导入、权限变更、配置发布、初始化、登录和高风险模型调用都必须审计。

## 13. API Gateway 与高并发模块

```mermaid
flowchart TD
    A[外部请求] --> B[TLS/WAF/请求大小限制]
    B --> C[全局/IP 限流]
    C --> D[转发 FastAPI]
    D --> E[RequestContextMiddleware]
    E --> F[SetupGuard]
    F --> G[Auth dependency]
    G --> H[应用层限流]
    H --> I[超时预算上下文]
    I --> J[Route Handler]
    J --> K{资源压力/异常?}
    K -- 是 --> L[降级/429/排队提示]
    K -- 否 --> M[正常响应]
    L --> N[统一错误结构 + 日志]
    M --> N
```

关键点：

- Gateway 层处理 TLS、WAF、请求大小限制、全局限流和 IP 限流。
- FastAPI Middleware 注入 request context、setup guard、鉴权、应用限流和超时上下文。
- 导入积压时应降低导入速度，不能影响查询优先级。
- LLM、embedding batch、Qdrant 写入、PostgreSQL 连接池都需要反压策略。

## 14. Core Data Model 数据生命周期模块

```mermaid
flowchart TD
    A[Enterprise] --> B[KnowledgeBase]
    B --> C[Folder]
    B --> D[Document]
    D --> E[DocumentVersion]
    E --> F[Chunk]
    F --> G[PermissionSnapshot]
    F --> H[ChunkIndexRef]
    H --> I[IndexVersion]
    D --> J[AccessBlock/Tombstone]
    K[ResourcePolicy] --> G
    L[Query Cache] --> M[绑定 enterprise/user/permission/config/index/model/prompt 版本]
    H --> N{查询可见性}
    J --> N
    G --> N
    N --> O[active 文档 + active chunk + active index + 无 block + 权限匹配]
```

关键点：

- PostgreSQL 是业务事实源，Qdrant、关键词索引和缓存都是派生数据。
- 查询可见性必须同时满足业务状态、索引状态、权限快照和删除阻断。
- 权限收紧和文档删除必须 fail closed。
- 查询响应中的每个 citation 必须回溯到 active chunk、document version 和对象存储原文。

## 15. 部署与运维流程

```mermaid
flowchart TD
    A[部署 PostgreSQL/Redis/MinIO/Qdrant] --> B[部署模型服务和 Model Gateway]
    B --> C[部署 API/Worker/Scheduled Worker]
    C --> D[进程只读取 DATABASE_*]
    D --> E[获取 setup token]
    E --> F[创建 setup-config-validation]
    F --> G[提交 setup-initialization]
    G --> H[发布 active_config v1]
    H --> I[ServiceBootstrap 初始化依赖]
    I --> J[管理员登录]
    J --> K[创建用户/部门/知识库]
    K --> L[导入测试文档]
    L --> M[执行权限查询]
    M --> N[检查审计/日志/指标/备份]
```

关键点：

- API、Worker 和 Model Gateway 管理进程的 `.env` 只保存数据库连接内容。
- 初始化完成后 setup-initialization 写接口必须关闭，setup token 必须失效。
- 上线前必须验证权限过滤下推、draft 索引不可见、删除阻断、缓存隔离和 LLM 降级。

## 16. 设计一致性说明

当前设计文档中，根目录总设计曾出现“岗位级、项目级、文档级权限”等更复杂表达；拆分后的模块设计已经收敛为：

- 组织模型：企业、部门、成员。
- 文档可见性：`department` 和 `enterprise`。
- 管理授权：通过 RBAC 角色和作用域控制。
- 查询权限：通过 `visibility = enterprise OR owner_department_id IN user.departments` 下推到检索层。

后续实现建议以拆分后的模块文档为准。这个收敛版本更适合最小生产落地，也更容易保证权限过滤下推、索引 payload 一致、查询缓存隔离和故障时 fail closed。
