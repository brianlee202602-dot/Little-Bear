# 企业内部高并发生产级 RAG 后端架构设计

## 1. 架构目标

本系统面向企业内部知识检索与问答，核心目标是让用户只能检索自己有权限访问的资料，并在高并发、批量导入、模型服务波动的情况下保持可降级、可审计、可恢复。

架构必须优先保证：

- 权限前置：权限过滤进入向量索引和关键词索引查询条件。
- 最小启动配置：API、Worker 和 Model Gateway 管理进程的启动配置只包含数据库连接内容，业务配置通过初始化接口写入数据库并发布为 active config。
- 导入异步化：HTTP 只创建导入任务，解析、切块、向量化、索引写入由 Worker 执行。
- 查询链路可降级：rewrite、embedding、rerank、LLM 均可按策略降级。
- 结果可解释：答案必须带引用，引用必须可回溯到文档、页码、chunk。
- 运维可观测：查询、导入、权限、配置、模型调用全链路有日志、指标、Trace 和审计。

## 3. 架构风格

最小生产阶段采用模块化单体 + 轻量 Worker：

- FastAPI 承载 API 路由和核心业务模块。
- 导入 Worker 是独立进程，基于 PostgreSQL 任务表领取任务。
- 模型能力通过 Model Gateway 访问，业务代码不直接调用 vLLM、TEI 或第三方 SDK。
- 检索索引通过抽象端口访问，业务代码不直接依赖 Qdrant SDK 或 PostgreSQL Full Text SQL 细节。
- 后续可按热点将 Query、Import、Permission、Config、Model Gateway 拆成独立服务。

## 5. 启动与初始化架构

API、Worker 和 Model Gateway 管理进程的启动配置只允许读取数据库连接配置：

- `DATABASE_URL`
- `DATABASE_CONNECT_TIMEOUT_SECONDS`
- `DATABASE_POOL_SIZE`
- `DATABASE_POOL_MAX_OVERFLOW`
- `DATABASE_SSL_MODE`

`APP_ENV`、`SERVICE_NAME`、`SERVICE_PORT`、`LOG_LEVEL` 等进程运行参数可以由容器启动命令、默认值或部署编排系统提供，但不得作为业务配置来源。Redis、Secret Provider、MinIO、Qdrant、模型网关、检索策略、权限策略、缓存策略和审计策略都必须来自数据库中的 active config。

启动后执行：

```text
1. 校验数据库连接配置是否存在
2. 数据库连接配置缺失或 PostgreSQL 连接失败:
   - 进程启动失败或 readiness=false
   - 不进入 setup mode
   - 不开放任何业务 API 或 setup 初始化写接口
3. 连接 PostgreSQL 成功后读取 system_state 和 active_config_version
4. system_state 不存在、initialized = false，或数据库内无任何配置数据:
   - 进入 setup_required
   - 生成新的临时 system_token，并使旧 setup token 失效
   - 仅开放 setup-state / setup-config-validations / setup-initialization
   - 不开放普通业务 API；只有初始化成功并完成 ServiceBootstrap 后才开放业务 API
5. initialized = true 且 active config 存在:
   - 从数据库加载 active config
   - 执行 ServiceBootstrap 初始化 Redis、Secret Provider、对象存储、向量库、关键词检索、模型网关、审计和限流等模块
   - 全部初始化成功后开放业务 API
6. initialized = true 但 active_config_version 缺失、active config 记录损坏或模块初始化失败:
   - 进入拒绝模式并告警
   - 不回退到环境变量业务配置
   - 返回管理员提示：当前系统配置不可用，是否进入恢复初始化流程；确认后生成或轮换 system_token，并走 setup-config-validations / setup-initialization 流程
```

业务配置不从环境变量读取，包括 MinIO、Qdrant、vLLM、TEI、检索策略、中文分词与企业词库、导入策略、权限策略、查询缓存策略、限流策略和审计策略。首次初始化接口一次性提交完整业务配置，校验通过后发布 `active_config v1`。

ServiceBootstrap 负责在 active config 加载后初始化所有业务依赖和模块。它的输入只能是数据库 active config，输出是模块 ready 状态；任一关键模块未 ready 时，系统保持 live 但 not ready，普通业务 API 必须拒绝服务。

用户 access token、refresh token、service token 和 setup system token 统一采用 JWT，并统一使用 `Authorization: Bearer <jwt>`。setup JWT 只允许访问 setup-config-validations / setup-initialization，不能访问普通业务 API 或管理员 API。

## 6. 查询链路

```text
1. 接收查询请求
2. 鉴权和限流
3. 构建权限上下文
4. 加载 active config、知识库策略和 active index versions
5. query rewrite
6. query expansion
7. query embedding
8. 向量召回、关键词召回、元数据召回
9. 每一路召回下推权限过滤和 active index 可见性过滤
10. 候选去重、融合、归一化
11. rerank
12. 候选元数据 gate：校验文档状态、access block、索引版本和权限快照
13. 上下文组装和压缩
14. Prompt 构造
15. LLM 生成
16. 引用校验和权限二次校验
17. 写日志、审计、指标
18. 返回答案、引用、置信度、trace_id
```

关键约束：

- embedding 失败时降级关键词检索。
- rerank 超时时使用融合分数。
- LLM 超时时返回检索结果和引用。
- 任何降级都写入响应和审计。
- 查询缓存命中后仍必须执行 access block 和引用有效性轻量校验。
- 严格引用模式下，流式响应不能先输出未经引用校验的最终答案。

## 7. 导入链路

```text
1. 用户或系统提交导入请求
2. API 校验权限、文件大小、幂等键
3. 创建 import_job 和 document 记录
4. 原始文件写入对象存储
5. Worker 领取任务
6. parse -> clean -> chunk -> embed -> index_vector(draft) -> index_keyword(draft)
7. 校验索引数量、向量维度、权限字段和 draft 可见性
8. 原子发布 active index version，并将旧版本 archived 或 pending_delete
9. 更新任务状态和审计日志
```

导入任务必须幂等，正式状态以 PostgreSQL 业务表为准。Worker 崩溃后依靠 `locked_until` 释放任务，下一 Worker 可继续领取。

## 8. 权限架构

权限由 RBAC 和文档可见性共同构成：

- RBAC：普通员工、部门管理员、知识库管理员、安全审计员，用于管理操作授权。
- 组织可见性：组织单位收敛为企业、部门、成员三层。
- 文档可见性：只支持 `department` 和 `enterprise`。成员上传文档默认部门可见，可选企业公开。

系统不是单管理员模型。首次初始化只创建第一个 `system_admin`，后续系统管理员、部门管理员、知识库管理员、安全审计员和普通用户的角色定义与分配由 Permission Service 管理，并由具备相应管理权限的操作者执行。Auth Service 只负责账号生命周期，不直接决定角色授权。

部门不建模上下级递归关系。文档默认只对归属部门 `owner_department_id` 的成员可见；选择企业公开后，对当前企业内所有部门成员可见。文档权限设置界面只展示“部门可见”和“企业公开”，不支持上级部门、子部门、项目组、指定用户或自定义组织 ID 共享。

每个 chunk 进入索引时必须带 `owner_department_id`、`visibility`、权限快照版本和索引可见性状态。查询时由 Permission Service 生成统一过滤条件，并同时用于向量检索和关键词检索；发现索引权限版本落后必须拒绝候选或回源确认，不能直接放行。

## 9. 配置架构

配置分层：

| 层级 | 来源 | 用途 |
| --- | --- | --- |
| L0 | 启动配置 | 数据库连接配置 |
| L1 | active config | Redis、Secret Provider、MinIO、Qdrant、模型网关和普通业务运行配置 |
| L2 | active config + Secret Provider | 密钥、token、连接凭证 |
| L3 | active config + 审批 | 高风险策略配置 |
| L4 | 代码默认值 | 安全兜底和 schema 默认值 |

业务模块只通过 Config Service 读取配置。`initialized = true` 但 active config 不可用，或 active config 加载后任一关键服务模块初始化失败时，系统进入拒绝模式并告警，不能回退到环境变量业务配置。

## 10. 数据存储

| 存储 | 职责 |
| --- | --- |
| PostgreSQL | 用户、组织、权限、文档元数据、任务状态、配置、审计、关键词索引 |
| Redis | 权限缓存、查询缓存、限流计数、分布式锁、配置变更通知 |
| MinIO | 原始文件、解析结果、清洗结果、大文本 |
| Qdrant | chunk 向量和权限 payload |
| Model Gateway 后端 | vLLM、TEI embedding、TEI rerank |

所有外部依赖必须有 adapter 和集成测试，替换基础设施时尽量只改 adapter 和配置。

核心数据模型详见 `docs/modules/14-核心数据模型设计实现文档.md`。PostgreSQL 中的 `documents`、`chunks`、`permission_snapshots`、`index_versions`、`chunk_index_refs` 和 `access_blocks` 是查询可见性的事实源。

## 11. 高并发策略

- API Gateway 层做全局限流、请求大小限制和连接超时。
- 应用层按用户、部门、知识库、API Key、IP、模型池做细粒度限流。
- 查询链路设置总超时预算，各阶段必须有独立 timeout。
- 导入 Worker 设置全局、部门、用户、任务内并发上限。
- Embedding 在线池和批处理池分离，导入不能挤占查询资源。
- Redis 缓存权限上下文、query embedding、热点查询结果和配置版本。查询结果缓存 key 必须包含 `enterprise_id`、`user_id` 或 `permission_filter_hash`、请求过滤条件、`config_version`、active index 版本、模型路由和 Prompt 版本；权限刷新中、引用校验失败或降级不完整的响应不缓存最终答案。

## 12. 可观测与审计

必须记录：

- `request_id`
- `trace_id`
- `user_id`
- `enterprise_id`
- `config_version`
- `permission_version`
- `index_version`
- `job_id`
- `stage`
- `latency_ms`
- `degraded`
- `error_code`

审计事件覆盖查询、导入、权限变更、配置发布、初始化、登录、管理员操作和模型高风险调用。

## 13. 部署架构

最小生产 Docker Compose 服务：

```text
api
import-worker
scheduled-worker
postgres
redis
minio
qdrant
model-gateway
tei-embedding
tei-rerank
vllm-llm
log-collector，可选
```

最小生产要求：

- PostgreSQL、Redis、MinIO、Qdrant 使用持久化 volume。
- 所有服务配置 healthcheck。
- API、Worker、模型服务设置 CPU、内存、GPU 资源限制。
- PostgreSQL 和 MinIO 配置定时备份。
- 日志输出 JSON 到 stdout。
- `.env` 或等价启动配置只保存数据库连接内容。
- Redis、Secret Provider、MinIO、Qdrant、模型服务地址和策略通过初始化接口写入数据库配置中心。

## 14. 演进路线

阶段 1：内部 MVP

- 模块化单体 + 单类 Worker。
- PostgreSQL Full Text + Qdrant 单节点。
- 本地账号、本地管理员配置中心。
- 结构化日志。

阶段 2：生产 Beta

- Worker 按阶段拆分。
- 增加 Prometheus、Grafana、OpenTelemetry。
- 增加配置审批和回滚。
- 增加离线评测集。

阶段 3：企业生产

- API 多副本。
- Redis Sentinel/Cluster。
- PostgreSQL 主从。
- Qdrant 集群。
- OpenSearch 可选。

阶段 4：集团级扩展

- Kubernetes。
- Celery/RabbitMQ 或 Kafka。
- Temporal 可选。
- 多地域、多租户、模型池治理。
