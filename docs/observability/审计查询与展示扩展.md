# 审计查询与展示扩展

## 1. 文档目标

本文定义 Little Bear RAG 后端 P1 阶段的审计查询、详情展示、导出边界、留存策略和脱敏规则扩展，作为管理后台审计中心、后端查询接口、数据留存任务和安全评审的共同输入。

本文基于以下 P0 契约扩展，不改变 P0 编码门禁：

- `docs/contracts/审计事件字典.md`
- `docs/admin/管理后台交互契约.md`
- `docs/contracts/openapi.yaml`
- `docs/contracts/database-schema.md`
- `docs/contracts/权限矩阵.md`
- `docs/testing/测试计划.md`

## 2. 范围边界

P1 要解决：

- 审计列表、查询日志列表和模型调用日志列表的展示字段与筛选规则。
- 审计详情页的 before / after / changed_fields 展示规范。
- 高风险事件的关联跳转和排障上下文。
- 审计导出的权限、字段、脱敏、异步任务和审计要求。
- `audit_logs`、`query_logs`、`model_call_logs` 的默认留存和清理策略。
- 脱敏策略在查询、详情、导出和归档中的一致执行。

P1 不解决：

- 不把审计导出反向提升为 P0 必须实现能力。
- 不保存完整 prompt、未脱敏查询原文、未脱敏文档原文、token 或 secret value。
- 不实现完整 SIEM、告警中心、取证平台或合规报表系统。
- 不允许前端绕过后端脱敏直接读取数据库或普通应用日志。
- 不以 LLM 判断审计风险、权限或脱敏结果。

## 3. 设计原则

- 审计查询是安全能力，不是普通业务搜索；所有接口必须要求 `audit:read` 和 `audit_admin` 资源校验。
- 展示字段必须从后端返回的脱敏结果中读取，前端只做布局，不承担安全脱敏。
- `audit_logs` 关注高风险动作、拒绝、失败和异常结果；`query_logs` 关注查询链路；`model_call_logs` 关注模型调用摘要。
- before / after 只记录结构化摘要，不记录可逆敏感值。
- 导出默认异步执行，必须限字段、限时间范围、限行数、限权限，并记录导出审计。
- 留存策略必须可配置，但不能通过普通业务配置关闭安全审计。
- 所有审计查询自身也必须写审计，避免审计数据被无痕查看或批量枚举。

## 4. 数据源与职责

| 数据源 | 主用途 | 详情页定位 | 禁止用途 |
| --- | --- | --- | --- |
| `audit_logs` | 高风险写操作、拒绝、失败、降级和安全事件 | 追责、操作回放、风险排障 | 保存完整业务明文、替代查询日志 |
| `query_logs` | 每次问答查询链路摘要 | 检索、权限过滤、缓存、citation 排障 | 保存 query 原文、prompt、模型输出全文 |
| `model_call_logs` | embedding / rerank / chat 调用摘要 | 模型路由、耗时、token、降级排障 | 保存 API key、完整 prompt、输入输出全文 |
| 普通应用日志 | 工程排障 | 仅通过 request_id / trace_id 关联 | 管理后台直接展示或导出 |

## 8. before / after 摘要规范

### 8.1 通用结构

`summary_json.before` 和 `summary_json.after` 必须是 object 或 null。

推荐结构：

```json
{
  "before": {
    "status": "active",
    "visibility": "department",
    "owner_department_id": "dept_sales",
    "permission_version": 12
  },
  "after": {
    "status": "active",
    "visibility": "enterprise",
    "owner_department_id": "dept_sales",
    "permission_version": 13
  },
  "changed_fields": [
    "visibility",
    "permission_version"
  ]
}
```

规则：

- `changed_fields` 必须只包含实际变化字段。
- before / after 只放业务排障需要的字段，不做完整对象快照。
- 字段名使用数据库或 OpenAPI 中已有命名。
- 秘密值、密码、token、完整 prompt、query 原文、文档正文不得进入 before / after。
- 删除类事件的 `after` 可以是 `{"status":"deleted"}` 或 `null`，但必须有 `reason` 和安全补偿摘要。
- 创建类事件的 `before` 固定为 `null`。
- 失败类事件如果没有状态变更，before / after 可以省略，必须提供 `stage`、`error_code` 和补偿结果。

### 8.2 事件级 before / after 建议

| `membership.replaced` | `department_ids_hash`, `primary_department_id`, `org_version` | `department_ids_hash`, `primary_department_id`, `org_version`, `permission_version` |
| `role_binding.replaced` | `role_codes`, `scope_summary`, `permission_version` | `role_codes`, `scope_summary`, `permission_version`, `high_risk_binding_present` |
| `department.updated` | `name`, `code`, `status`, `org_version` | `name`, `code`, `status`, `org_version` |
| `knowledge_base.updated` | `name`, `status`, `default_visibility`, `owner_department_id` | `name`, `status`, `default_visibility`, `owner_department_id` |
| `document.updated` | `title`, `status`, `visibility`, `owner_department_id`, `permission_snapshot_id` | `title`, `status`, `visibility`, `owner_department_id`, `permission_snapshot_id` |

### 8.3 配置 diff 摘要

配置变更不得把完整 active config 直接塞入 `summary_json`。

推荐字段：

| 字段 | 说明 |
| --- | --- |
| `config_hash` | 脱敏后的配置整体 hash |
| `changed_fields` | 点路径字段，例如 `retrieval.vector.top_k` |
| `risk_level` | 配置变更风险 |
| `high_risk_keys` | 高风险 key 列表，不含 secret value |
| `secret_ref_changed` | 是否变更 Secret ref |
| `service_targets` | 影响 API / Worker / Model Gateway 等目标 |
| `reload_strategy` | `hot_reload` / `drain_and_restart` / `manual_restart` |
| `active_pointer_unchanged` | 发布失败或 reload 失败时必须明确 |

## 10. 导出边界

### 10.1 权限与确认

审计导出是 P1 受控能力，不属于 P0 门禁。启用前必须满足：

- 需要 `audit:read`。
- 推荐新增 `audit:export` scope；未新增前不得复用普通 `audit:read` 直接开放批量导出。
- 必须经过 `audit_admin` resource_check。
- 导出时间范围默认最长 7 天，超过需要强确认或更高权限。
- 每次导出必须写 `audit.export_requested`、`audit.export_succeeded` 或 `audit.export_failed` 审计事件。

### 10.2 导出方式

推荐异步任务：

1. 提交导出请求，后端校验 scope、筛选条件、时间范围、行数上限和字段集合。
2. 创建导出任务，返回 `export_job_id`。
3. Worker 生成 CSV 或 JSONL 文件，写入对象存储受控 bucket。
4. 下载链接短期有效，默认 15 分钟。
5. 下载动作写审计，过期后不可继续访问。

P1 最小实现可只支持后台生成后由运维取件，但仍必须记录导出请求和结果。

### 10.3 导出字段

默认允许导出：

- `id`
- `created_at`
- `event_name`
- `actor_type`
- `actor_id`
- `resource_type`
- `resource_id`
- `action`
- `result`
- `risk_level`
- `error_code`
- `config_version`
- `permission_version`
- `index_version_hash`
- `request_id`
- `trace_id`
- 脱敏后的 `summary_json`

默认禁止导出：

- 完整普通应用日志。
- 未脱敏 `summary_json`。
- query 原文。
- prompt 原文。
- 模型输入输出全文。
- 文档原文或大段 chunk。
- token、secret value、密码或 API key。

### 10.4 导出限制

| 限制项 | 默认值 |
| --- | --- |
| 最大时间范围 | 7 天 |
| 最大行数 | 100000 行 |
| 最大文件大小 | 100 MB |
| 下载链接有效期 | 15 分钟 |
| 导出文件保留 | 24 小时 |
| 格式 | CSV 或 JSONL |

超过限制时返回结构化错误，例如 `AUDIT_EXPORT_TOO_LARGE`、`AUDIT_EXPORT_RANGE_TOO_WIDE`。

## 11. 留存策略

以下为默认建议，实际生产值应由安全、法务和业务方确认后固化到部署配置。P0 本地开发可以使用更短留存。

| 数据 | 热存储默认 | 归档默认 | 删除默认 | 说明 |
| --- | --- | --- | --- | --- |
| `audit_logs` | 180 天 | 3 年 | 3 年后删除或按企业策略延长 | 高风险操作和安全追责优先保留 |
| `query_logs` | 90 天 | 180 天 | 1 年后删除 | 只存 hash 和链路摘要，仍按敏感行为数据管理 |
| `model_call_logs` | 30 天 | 180 天 | 1 年后删除 | 成本和性能排障为主 |
| 审计导出文件 | 24 小时 | 不归档 | 24 小时后删除 | 导出结果本身属于敏感文件 |
| 聚合指标 | 180 天 | 1 年 | 1 年后删除 | 只保留不可反查个人明细的聚合值 |

留存任务要求：

- 清理任务必须按企业分区或时间分区执行，避免长事务。
- 删除前可先归档到对象存储或冷库，归档文件必须加密。
- 清理任务本身必须写审计摘要，包含删除范围、数量、数据类型和执行结果。
- 清理失败必须告警，不得静默跳过。
- 对处于安全调查 hold 状态的数据，禁止自动清理。

## 12. 脱敏策略

### 12.1 分层脱敏

| 层级 | 责任 |
| --- | --- |
| 写入层 | 业务服务写入 `summary_json` 前必须脱敏 |
| 查询层 | 后端返回审计数据前再次执行 denylist 扫描和字段裁剪 |
| 展示层 | 前端只展示后端返回字段，不本地恢复或拼接敏感值 |
| 导出层 | 使用导出专用字段白名单和二次脱敏 |
| 归档层 | 归档前保留脱敏结果，不额外保存明文副本 |

### 12.2 字段处理规则

| 类型 | 允许展示 | 禁止展示 |
| --- | --- | --- |
| token | `jti`, token 类型, hash, 过期时间 | access / refresh / setup / service token 明文 |
| secret | `secret_ref`, provider, path hash | secret value, API key, 对象存储 access key |
| 密码 | 操作类型, 策略校验结果 | 明文密码, 密码摘要材料, 重置密码明文 |
| query | `query_hash`, 长度, 状态, 降级原因 | 查询原文 |
| prompt | `prompt_template_id`, 版本, hash, 变量摘要 | 完整 prompt |
| 文档内容 | `content_hash`, page, chunk_id, 限长 `text_preview` | 原文、大段 chunk |
| 个人信息 | 用户 ID, 角色, 部门, 掩码邮箱/手机号 | 完整个人敏感信息 |
| 配置 | `config_hash`, changed field path, secret ref | secret value, 连接密钥 |

### 12.3 自动扫描规则

后端返回或导出前必须执行基础 denylist 扫描：

- 字段名命中 `password`、`token`、`secret`、`api_key`、`authorization`、`credential` 时默认拒绝返回，除非字段在允许白名单中，例如 `token_jti`、`secret_ref`。
- 字符串疑似 Bearer token、JWT、AK/SK、私钥块时拒绝返回并记录安全事件。
- `summary_json` 单字段长度超限时截断并标记 `truncated=true`。
- 数组超过上限时只返回前 N 项摘要和 `total_count`。

## 13. 失败处理

| 场景 | 处理 |
| --- | --- |
| 审计查询超时 | 返回 `AUDIT_QUERY_TIMEOUT`，提示缩小时间范围 |
| 无权限访问审计 | 返回 `FORBIDDEN`，写 `auth.token_scope_denied` 或等价审计 |
| 脱敏扫描失败 | fail closed，返回 `AUDIT_REDACTION_FAILED` |
| 导出任务过大 | 拒绝创建任务，返回 `AUDIT_EXPORT_TOO_LARGE` |
| 导出文件生成失败 | 标记任务失败，写 `audit.export_failed` |
| 留存清理失败 | 保留数据，写失败审计并告警 |

## 14. 建议新增审计事件

这些事件用于审计中心自身，不要求进入 P0：

| event_name | 触发点 | actor_type | resource_type | action | result | risk_level | `summary_json` 最小字段 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| `audit.viewed` | 查看审计、查询日志或模型调用日志详情 | `user` | `audit` | `view` | success | medium | `request_path`, `target_log_type`, `target_log_id`, `reason` |
| `audit.search_executed` | 执行审计中心列表查询 | `user` | `audit` | `search` | success | medium | `request_path`, `filters_hash`, `time_range`, `result_count` |
| `audit.export_requested` | 提交审计导出任务 | `user` | `audit` | `export` | success | high | `request_path`, `filters_hash`, `time_range`, `field_set`, `export_job_id` |
| `audit.export_failed` | 导出失败 | `service` | `audit` | `export` | failure | high | `export_job_id`, `error_code`, `stage` |
| `audit.export_succeeded` | 导出文件生成成功 | `service` | `audit` | `export` | success | high | `export_job_id`, `row_count`, `file_size_bytes`, `expires_at` |
| `audit.export_downloaded` | 下载导出文件 | `user` | `audit` | `download_export` | success | high | `export_job_id`, `file_id`, `expires_at` |
| `audit.retention_cleanup_succeeded` | 留存清理成功 | `system` | `audit` | `retention_cleanup` | success | medium | `data_type`, `time_range`, `deleted_count`, `archive_target` |
| `audit.retention_cleanup_failed` | 留存清理失败 | `system` | `audit` | `retention_cleanup` | failure | high | `data_type`, `time_range`, `error_code`, `retry_scheduled` |

如 `resource_type` 枚举暂未包含 `audit`，P1 实现前必须同步更新 `docs/contracts/审计事件字典.md`、`docs/contracts/database-schema.md`、`docs/contracts/openapi.yaml` 和权限矩阵。

## 15. 测试要求

P1 自动化测试至少覆盖：

- `audit:read` 用户可以查询审计列表和详情。
- 普通用户不能访问审计中心接口。
- 筛选条件按 `enterprise_id`、时间范围、风险、结果和资源约束生效。
- before / after 不包含禁止字段。
- 查询日志详情不返回 query 原文、prompt 或模型输出全文。
- 模型调用日志不返回 API key、prompt、输入输出全文。
- 导出字段白名单生效。
- 导出超过时间范围或行数限制被拒绝。
- 审计查询、导出和下载动作自身写审计。
- 留存清理任务不会删除 investigation hold 数据。
- 脱敏扫描失败时 fail closed。

手工验收至少覆盖：

1. 使用审计管理员账号查看 `config.reload_failed` 详情，确认能看到 reload 阶段、目标、错误码和 active 指针未变。
2. 使用审计管理员账号查看 `document.permission_tightened` 详情，确认 before / after、access block、permission version 和 refresh job 可追踪。
3. 使用普通用户访问审计中心，确认被拒绝且返回结构化错误。
4. 尝试导出 30 天范围数据，确认被限制或要求更高权限。

## 16. 落地步骤

1. 后端补充审计查询 repository 的时间范围、事件名、错误码、版本和 trace 筛选。
2. 后端实现统一 `AuditRedactor`，用于详情、列表和导出。
3. 前端按本文分区展示审计详情，不展示原始全量 JSON 作为默认视图。
4. P1 如启用导出，先新增 `audit:export` scope、导出任务表或复用受控 job 表，再补 OpenAPI。
5. 增加 retention cleanup 任务，先 dry-run 输出数量，再允许正式删除。
6. 将新增审计中心自身事件回填到 `docs/contracts/审计事件字典.md` 和测试计划。
