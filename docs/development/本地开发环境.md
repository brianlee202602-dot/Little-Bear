# 本地开发环境

## 1. 目标

本文定义 Little Bear RAG 后端 P0 阶段的本地开发环境和最小依赖启动方式，用于支持初始化、登录、导入、索引、权限过滤查询、citation 和模型 mock 降级测试。

本地环境只解决基础设施启动和依赖连通性，不把业务运行配置写入环境变量。业务配置必须通过 `PUT /internal/v1/setup-initialization` 写入数据库并发布为 `active_config v1`。

## 2. 文件清单

| 文件 | 用途 |
| --- | --- |
| `.env.example` | 本地进程启动配置模板，只包含数据库连接和进程参数 |
| `docker-compose.yml` | PostgreSQL（含 `zhparser`）、Redis、MinIO、Qdrant 和模型网关 mock |
| `Makefile` | 本地启动、停止、查看日志、健康检查和重置数据入口 |
| `tools/mock_model_gateway.py` | P0 模型网关 mock，提供 embedding、rerank、LLM 和 health API |
| `docs/examples/setup-initialization.local.p0.json` | 本地 P0 初始化请求样例，覆盖 `setup` 和完整 `active_config v1` |

## 3. 前置条件

- Docker Engine 或 Docker Desktop。
- Docker Compose v2。
- `make`。
- `curl`。

可选：

- `psql`，用于人工连接 PostgreSQL。
- MinIO Client `mc`，用于人工检查对象存储；compose 内已使用 `minio-init` 自动创建 bucket。

## 4. 启动方式

首次启动基础依赖：

```bash
make env
make up
make ps
make health
```

当前 `Makefile` 只负责本地基础设施生命周期。后端 API、Worker 和受控 CLI 实现后，必须补齐以下等价入口，名称可以按工程脚手架调整，但文档和 CI 必须保持一致：

```bash
# 启动 API Service，只允许读取数据库连接配置和进程参数
make api

# 启动 Worker，只允许读取数据库连接配置和进程参数
make worker

# 签发一次性 setup JWT；JWT 明文只允许在本命令输出中出现一次
rag-admin setup-token issue --ttl 30m

# 轮换或吊销 setup JWT
rag-admin setup-token rotate --ttl 30m
rag-admin setup-token revoke --jti <setup_jwt_jti>
```

`make api` 当前封装的 FastAPI 异步服务启动命令为：

```bash
PYTHONPATH=apps/api python3 -m uvicorn app.main:app --host ${API_HOST:-0.0.0.0} --port ${API_PORT:-8000} --reload
```

`uvicorn` 是 ASGI Server，负责运行 FastAPI 应用和 asyncio 事件循环；它不改变配置边界，API 启动阶段仍只读取 PostgreSQL 连接配置和进程参数。

实现约束：

- `make api` 和 `make worker` 不得从环境变量读取 Redis、MinIO、Qdrant、模型网关、检索、权限、缓存、限流或审计业务配置。
- API 和 Worker 在系统未初始化时只能完成数据库连接、migration 检查、setup 生命周期门禁和 healthcheck。
- 初始化完成后，API 和 Worker 必须从数据库中的 `active_config v1` 执行 ServiceBootstrap。
- `rag-admin setup-token issue/rotate/revoke` 必须写入 `setup_token.issued`、`setup_token.rotated` 或 `setup_token.revoked` 审计事件。

停止：

```bash
make down
```

查看日志：

```bash
make logs
make logs-model
```

重置所有本地数据：

```bash
make reset
```

`make reset` 会删除 PostgreSQL、Redis、MinIO 和 Qdrant 的 Docker volume。执行前需要确认本地数据可以丢弃。

## 6. 启动配置边界

`.env` 只允许保存：

- 数据库连接配置。
- API host、port、服务名、日志级别等进程启动参数。

`docker-compose.yml` 可以为本地基础设施提供默认端口、默认 bucket 和容器初始化账号，但这些值只是本地容器启动默认值，不得被 API、Worker 或业务模块当作运行配置读取。

`.env` 不得保存：

- Redis URL。
- MinIO endpoint、bucket、access key、secret key。
- Qdrant URL。
- 模型网关地址。
- JWT 签名密钥。
- 检索策略、权限策略、缓存策略、降级策略、审计策略。

这些配置必须通过初始化接口写入 active config。

## 7. 本地初始化配置

初始化请求必须使用 `docs/contracts/config-schema.md` 和 `docs/contracts/config.schema.json` 校验。完整本地 P0 初始化 payload 已提供：

```text
docs/examples/setup-initialization.local.p0.json
```

该文件包含 `setup` 和 `config` 两部分，可作为 `POST /internal/v1/setup-config-validations` 与 `PUT /internal/v1/setup-initialization` 的请求体基线。

本地 P0 依赖地址固定为：

| 配置分组 | 本地值 |
| --- | --- |
| `redis.url` | `redis://redis:6379/0` |
| `storage.minio_endpoint` | `http://minio:9000` |
| `storage.bucket` | `little-bear-rag` |
| `vector_store.qdrant_base_url` | `http://qdrant:6333` |
| `vector_store.collection_prefix` | `little_bear_p0` |
| `keyword_search.provider` | `postgres_full_text` |
| `keyword_search.keyword_analyzer` | `zhparser` |
| `model_gateway.base_url` | `http://model-gateway:8080` |
| `model.embedding_dimension` | `1024` |

本地初始化 payload 必须覆盖以下 `active_config v1` 顶层分组：

```text
schema_version, config_version, scope, secret_provider, redis, storage,
vector_store, keyword_search, model_gateway, model, llm, auth, retrieval,
chunk, import, permission, security, cache, rate_limit, timeout, degrade,
audit, observability
```

缺少任一分组时，`setup-config-validations` 必须返回 `CONFIG_SCHEMA_INVALID`，不得写入 active config。

Secret value 不得写入 active config。P0 本地可使用 PostgreSQL 加密 `secrets` 表作为 Secret Store，并预置以下 Secret 引用：

- `secret://rag/minio/access-key`
- `secret://rag/minio/secret-key`
- `secret://rag/auth/jwt-signing-key`

本地 Secret Store 预置约束：

- `secret://rag/minio/access-key` 的 value 对应 compose 中 MinIO 默认 root user。
- `secret://rag/minio/secret-key` 的 value 对应 compose 中 MinIO 默认 root password。
- `secret://rag/auth/jwt-signing-key` 必须由初始化前的受控本地命令生成，不得硬编码进 active config、日志或审计摘要。
- API 响应、审计摘要和测试输出只能展示 Secret ref 或脱敏摘要。

## 8. 初始化生命周期要求

未初始化时：

- 只允许 `GET /internal/v1/setup-state`。
- 只允许使用 setup JWT 调用 `POST /internal/v1/setup-config-validations`。
- 只允许使用 setup JWT 调用 `PUT /internal/v1/setup-initialization`。
- 普通业务、管理后台和服务间 API 必须返回 `SETUP_REQUIRED`。
- `/health/live` 可以返回 live。
- `/health/ready` 必须返回 not ready 或未初始化摘要。

初始化完成后：

- `POST /internal/v1/setup-config-validations` 和 `PUT /internal/v1/setup-initialization` 必须返回 `SETUP_CLOSED`。
- API Service 和 Worker 只能从数据库 active config 初始化 Redis、Secret Store、MinIO、Qdrant、KeywordSearch、Model Gateway、Audit Sink 和 Rate Limiter。
- 不得回退读取环境变量中的业务配置。

## 9. ready 检查要求

`GET /health/ready` 必须覆盖：

- PostgreSQL 可连接。
- migration 版本符合当前代码要求。
- 初始化状态为 initialized。
- active config 存在且 schema 有效。
- ServiceBootstrap 已完成。
- Secret Provider 可读取必要 Secret ref。
- Redis ping 成功。
- MinIO bucket 可访问，测试对象可写入和删除。
- Qdrant 可连接；ServiceBootstrap 必须创建或校验 P0 collection，collection 维度与 `model.embedding_dimension` 一致。
- PostgreSQL Full Text 最小查询可执行。
- `zhparser` 扩展和 `little_bear_zh` text search configuration 可用。
- Model Gateway mock `/internal/v1/model-health` ready。
- `audit_logs` 可写入。
- Rate Limiter 可用或已进入明确降级策略。

Qdrant ready 语义：

- 初始化完成且尚未导入文档时，`GET /health/ready` 也必须可以返回 ready。
- 如果 P0 collection 不存在，ServiceBootstrap 应创建空 collection 后再标记 ready。
- 如果 collection 已存在但维度、distance 或 payload schema 与 active config 不兼容，ready 必须失败并返回结构化原因。
- 不得因为 collection 暂无 point、chunk 或文档数据而判定 not ready。

PostgreSQL Full Text / 中文分词 ready 语义：

- 本地 compose 提供的 PostgreSQL 必须内置 `zhparser` 扩展。
- Alembic `0001_extensions` 迁移必须成功创建 `little_bear_zh` text search configuration。
- 如果 `zhparser` 扩展不存在，初始化依赖检查必须失败，并返回结构化原因。


```bash
curl -fsS http://localhost:8080/internal/v1/model-embeddings \
```

## 11. P0 本地验收路径

后端 API 和 Worker 实现后，本地环境应支持以下路径：

1. `make up` 启动依赖。
2. API 进程只读取数据库连接配置启动。
3. `GET /internal/v1/setup-state` 返回未初始化。
4. 受控 CLI 签发 setup JWT。
5. `POST /internal/v1/setup-config-validations` 校验初始化配置、Secret ref 和依赖连通性。
6. `PUT /internal/v1/setup-initialization` 创建首个管理员、默认企业、默认部门、内置角色和 `active_config v1`。
7. `GET /health/ready` 返回 ready。
8. 管理员登录。
9. 创建销售部门、Alice、Bob、角色绑定和销售知识库。
10. 上传销售政策文档并创建 import_job。
11. Worker 完成解析、切块、embedding、draft 索引写入和 active index 发布。
12. Alice 查询得到带 citation 的 mock 答案。
13. Bob 在 `department` 可见性下查不到销售文档。
14. 权限扩大为 `enterprise` 后 Bob 可查到。
15. 权限收紧或删除后，Bob/Alice 不能通过旧缓存或旧索引命中。
16. embedding、rerank 或 LLM mock 故障时按测试计划降级并写审计。

最小初始化命令序列：

```bash
make up
make api

curl -fsS http://localhost:8000/internal/v1/setup-state

SETUP_JWT="$(rag-admin setup-token issue --ttl 30m --plain)"

curl -fsS http://localhost:8000/internal/v1/setup-config-validations \
  -H "authorization: Bearer ${SETUP_JWT}" \
  -H "content-type: application/json" \
  --data-binary @docs/examples/setup-initialization.local.p0.json

curl -fsS -X PUT http://localhost:8000/internal/v1/setup-initialization \
  -H "authorization: Bearer ${SETUP_JWT}" \
  -H "content-type: application/json" \
  --data-binary @docs/examples/setup-initialization.local.p0.json

curl -fsS http://localhost:8000/health/ready
make worker
```

初始化后的最小登录和查询 smoke test：

```bash
curl -fsS http://localhost:8000/internal/v1/sessions \
  -H "content-type: application/json" \
  -d '{"username":"admin","password":"ChangeMe_123456"}'

curl -fsS http://localhost:8000/internal/v1/queries \
  -H "authorization: Bearer <access_jwt>" \
  -H "content-type: application/json" \
  -d '{"kb_ids":["<sales_kb_id>"],"query":"销售政策是什么？","mode":"answer","top_k":8,"include_sources":true}'
```

`<sales_kb_id>` 必须由管理员配置验收路径创建，不能假定系统内置业务知识库。

## 12. 故障排查

查看容器状态：

```bash
make ps
```

查看全部日志：

```bash
make logs
```

检查 PostgreSQL：

```bash
docker compose exec postgres pg_isready -U little_bear -d little_bear
```

检查 Redis：

```bash
docker compose exec redis redis-cli ping
```

检查 MinIO：

```bash
curl -fsS http://localhost:9000/minio/health/live
```

检查 Qdrant：

```bash
curl -fsS http://localhost:6333/readyz
```

检查模型网关 mock：

```bash
curl -fsS http://localhost:8080/internal/v1/model-health
```

常见问题：

| 现象 | 可能原因 | 处理方式 |
| --- | --- | --- |
| `make health` 中 MinIO 失败 | bucket 初始化尚未完成 | 查看 `docker compose logs minio-init`，重跑 `make up` |
| Qdrant ready 失败 | 容器仍在启动或端口被占用 | 查看 `docker compose logs qdrant`，检查本机 `6333` 端口 |
| Model Gateway mock 失败 | 本地仓库未挂载或脚本异常 | 查看 `make logs-model` |
| API ready=false | 未初始化、active config 缺失或 ServiceBootstrap 失败 | 先检查 setup 状态和 active config |
| 端口冲突 | 本机已有服务占用默认端口 | 修改 `docker-compose.yml` 中本地端口映射后重启；不得把业务依赖地址加入 `.env` |

## 13. 完成标准

- 新开发者可以通过 `make env && make up && make health` 启动最小环境。
- 本地环境包含 PostgreSQL、Redis、MinIO、Qdrant、模型网关 mock、Embedding mock、Rerank mock 和 LLM mock。
- 启动配置只包含数据库连接和进程参数，业务配置通过初始化接口写入 active config。
- 未初始化和初始化后的 API 生命周期门禁有明确约束。
- ready 检查覆盖数据库、初始化状态、active config、ServiceBootstrap 状态和 migration 版本。
- 不需要真实大模型即可跑通 P0 自动化测试。
