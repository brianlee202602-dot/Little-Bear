# API Gateway 与高并发设计实现文档

## 1. 模块目标

API Gateway 与并发控制负责保护后端服务免受突发流量、恶意请求和资源争抢影响，同时为查询和导入链路提供可预测的超时、限流、熔断、缓存和降级策略。

## 2. 职责划分

API Gateway / Ingress：

- TLS。
- WAF。
- 请求大小限制。
- 全局限流。
- IP 限流。
- 路由。
- 访问日志。

FastAPI Middleware：

- request_id / trace_id 注入。
- setup guard。
- Auth 依赖。
- 应用层限流。
- 超时上下文。
- 错误响应标准化。

业务模块：

- 用户、部门、知识库、模型池维度限流。
- 查询和导入降级。
- 缓存命中。

## 3. 请求入口流程

```text
1. Gateway 接收请求
2. TLS/WAF/请求大小限制
3. 全局限流
4. 转发到 FastAPI
5. RequestContextMiddleware 生成 request_id、trace_id
6. SetupGuardMiddleware 检查 initialized
7. Auth dependency 鉴权
8. RateLimit dependency 应用层限流
9. Route handler 执行业务
10. 统一错误处理和结构化日志
```

## 4. 限流维度

| 维度 | 用途 |
| --- | --- |
| IP | 防止异常客户端 |
| user_id | 防止单用户占满资源 |
| department_id | 防止单部门导入或查询打满 |
| kb_id | 防止热点知识库拖垮索引 |
| api_key | 内部系统调用隔离 |
| model_pool | 防止 GPU 资源耗尽 |
| import_job | 控制导入并发 |

限流实现：

- 最小阶段使用 Redis 计数器或滑动窗口。
- 对导入使用数据库任务并发控制。
- 高并发阶段可迁移到 API Gateway 或服务网格统一限流。

## 5. 超时预算

查询默认总预算建议 8s：

| 阶段 | 建议 |
| --- | --- |
| auth + permission | 100ms |
| rewrite | 300ms |
| embedding | 300ms |
| vector search | 300ms |
| keyword search | 300ms |
| fusion | 50ms |
| rerank | 800ms |
| context | 100ms |
| LLM first token | 3000ms |
| response postprocess | 100ms |

导入不走 HTTP 长连接等待结果，HTTP 请求只创建任务。

## 6. 降级策略

| 异常 | 降级 |
| --- | --- |
| rewrite 超时 | 使用原始查询 |
| embedding 超时 | 关键词检索 |
| 向量索引不可用 | 关键词检索 + 元数据 |
| 关键词索引不可用 | 向量检索 + 元数据 |
| rerank 超时 | 融合分数排序 |
| LLM 超时 | 返回检索结果和引用 |
| 模型池满载 | 路由小模型或返回排队提示 |
| 导入积压 | 降低导入速度，不影响查询 |

降级响应必须包含：

```json
{
  "degraded": true,
  "degrade_reason": "rerank_timeout",
  "fallback": "fusion_score"
}
```

## 7. 缓存策略

| 缓存 | key 组成 | 失效 |
| --- | --- | --- |
| 权限上下文 | enterprise_id + user_id + permission_version + org_version | 权限或组织变更 |
| query embedding | enterprise_id + normalized_query + embedding_model + model_version + dictionary_version + input_type | 模型或词库变更 |
| 检索中间结果 | enterprise_id + query_hash + kb_ids_hash + request_filter_hash + permission_filter_hash + config_version + index_version_hash + retrieval_strategy_hash | 权限、配置、索引变更 |
| 最终答案 | enterprise_id + user_id + query_hash + kb_ids_hash + request_filter_hash + permission_filter_hash + config_version + index_version_hash + model_route_hash + prompt_template_version | 权限、配置、索引、模型路由、Prompt 变更 |
| active config | config_version | 配置发布 |
| 知识库元数据 | kb_id + version | 知识库更新 |

查询结果缓存只适合稳定、可解释的结果。权限刷新中、引用校验失败或降级不完整的最终答案不得缓存。缓存命中后仍必须执行 access block 和 citation 有效性轻量校验。

## 8. Backpressure

需要反压的资源：

- LLM GPU 池。
- Embedding batch 队列。
- 导入 Worker。
- Qdrant 写入。
- PostgreSQL 连接池。

策略：

- 超过阈值时返回 `429` 或排队状态。
- 导入任务降速。
- 查询优先级高于导入 embedding。
- 模型池过载时按优先级路由小模型或拒绝低优先级请求。

## 9. 连接池

建议：

- API PostgreSQL pool 和 Worker PostgreSQL pool 分开配置。
- 导入 Worker 不占用查询 API 的连接池。
- Redis 连接池设置合理上限。
- 模型 HTTP client 使用连接池和 per-host 限制。

## 10. 错误响应

统一结构：

```json
{
  "request_id": "req_001",
  "error_code": "QUERY_RATE_LIMITED",
  "message": "rate limit exceeded",
  "retryable": true,
  "retry_after_ms": 1000
}
```

## 11. 测试点

- 未初始化时业务 API 被 setup guard 拦截。
- 单用户超过 QPS 返回 429。
- 导入积压不影响查询接口。
- rerank 超时返回降级响应。
- request_id 出现在响应和日志中。
- 缓存 key 包含用户、租户、权限过滤、请求过滤、配置版本、索引版本、模型路由和 Prompt 版本，权限变更后不会复用旧结果。
- 权限刷新中、引用校验失败或降级不完整的查询不会缓存最终答案。
- 缓存命中后仍会执行 access block 和 citation 有效性检查。
