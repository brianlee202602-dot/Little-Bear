# Model Gateway 设计实现文档

## 1. 模块目标

Model Gateway 是 Embedding、Rerank、LLM 的统一模型入口，屏蔽底层 vLLM、TEI 或其他模型供应商差异，为业务服务提供稳定的模型调用接口。

## 2. 模块职责

- 提供统一 HTTP/gRPC API。
- 模型路由和版本选择。
- 在线 embedding 和离线 embedding 分池。
- Rerank 调用。
- LLM chat/completions 调用。
- 超时、重试、熔断、降级。
- token usage 和 latency 统计。
- 模型服务健康检查。
- 模型版本与索引兼容校验。
- 返回可审计的 `model_route_hash`，用于查询日志和缓存 key。

## 3. 非职责

- 不负责权限判断。
- 不持久化业务文档。
- 不决定是否引用资料。
- 不直接暴露公网。

## 4. 部署形态

最小阶段：

```text
model-gateway: FastAPI
tei-embedding: TEI embedding 服务
tei-rerank: TEI rerank 服务
vllm-llm: vLLM LLM 服务
```

后续可拆：

- embedding-online。
- embedding-batch。
- rerank。
- llm-large。
- llm-small。
- high-security llm pool。

## 5. 建议包结构

```text
app/modules/models/
  gateway_client.py
  schemas.py

model_gateway/
  api/
  routing.py
  providers/
    tei_embedding.py
    tei_rerank.py
    vllm_chat.py
  circuit_breaker.py
  metrics.py
  schemas.py
  errors.py
```

如果最小阶段 Model Gateway 内置在 FastAPI 应用中，仍应保持上述逻辑边界。

## 6. 内部 API

```http
POST /internal/v1/models/embeddings
POST /internal/v1/models/rerank
POST /internal/v1/models/chat/completions
GET  /internal/v1/models
GET  /internal/v1/models/health
```

Embedding 请求：

```json
{
  "model": "bge-m3-online-v1",
  "input_type": "query",
  "input": ["销售合同审批需要哪些材料？"],
  "normalize": true,
  "trace_id": "trace_001"
}
```

Rerank 请求：

```json
{
  "model": "bge-reranker-v2",
  "query": "销售合同审批需要哪些材料？",
  "documents": [
    {"id": "chunk_001", "text": "销售合同审批需提交合同正文..."}
  ],
  "top_n": 8,
  "trace_id": "trace_001"
}
```

LLM 请求建议兼容 Chat Completions：

```json
{
  "model": "qwen-enterprise-72b",
  "messages": [
    {"role": "system", "content": "你是企业内部知识检索助手。"},
    {"role": "user", "content": "销售合同审批需要哪些材料？"}
  ],
  "temperature": 0.1,
  "max_tokens": 800,
  "stream": true,
  "trace_id": "trace_001"
}
```

## 7. 路由配置

示例：

```json
{
  "embedding": {
    "online_default": "bge-m3-online-v1",
    "batch_default": "bge-m3-batch-v1"
  },
  "rerank": {
    "default": "bge-reranker-v2"
  },
  "llm": {
    "default": "qwen-enterprise-72b",
    "high_security": "private-qwen-72b",
    "fallback": "qwen-enterprise-14b"
  }
}
```

路由因素：

- 模型类型。
- 文档可见性和知识库策略。
- 用户部门。
- 请求优先级。
- 当前模型池负载。
- 灰度策略。

每次路由决策必须输出 `model_route_hash` 和可审计摘要，包含 embedding、rerank、LLM 模型名、模型版本、路由池和灰度版本；不得把底层凭证或完整 prompt 写入普通日志。

## 8. 超时和重试

建议：

| 调用 | 超时 | 重试 |
| --- | --- | --- |
| query embedding | 300ms 到 800ms | 1 次 |
| batch embedding | 5s 到 30s | 可多次 |
| rerank | 500ms 到 900ms | 0 到 1 次 |
| LLM 首 token | 3s | 不盲目重试 |
| LLM 总超时 | 20s 到 60s | 按业务模式 |

重试必须带幂等 request id，并避免放大模型池压力。

## 9. 熔断和降级

| 服务 | 异常 | 降级 |
| --- | --- | --- |
| Embedding Online | 超时或不可用 | Query Service 降级关键词检索 |
| Embedding Batch | 积压过高 | 降低导入速度 |
| Rerank | 超时 | 使用融合分数 |
| LLM | 超时 | 返回检索结果和引用 |
| 大模型池 | GPU 满载 | 路由小模型或排队 |

Model Gateway 返回结构化降级原因，业务响应也要体现 `degraded=true`。

## 10. 模型版本与索引兼容

Embedding 配置必须记录：

- model name。
- model version。
- dimension。
- normalize。
- tokenizer/version。

升级策略：

- 相同维度但语义空间变化：建议双索引评测后切换。
- 维度变化：必须新建 collection 或全量重建。
- 灰度期间 query embedding 和 document embedding 必须匹配。

## 11. 安全要求

- Model Gateway 做服务间认证。
- 外部模型路由必须符合企业数据安全策略。
- 请求和响应日志脱敏。
- 不记录完整 prompt，除非审计策略明确允许且做访问控制。
- 普通日志只记录 prompt template/version/hash、变量摘要和模型版本。
- 模型错误不得返回底层凭证、路径或内部拓扑。

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `MODEL_EMBEDDING_FAILED` | embedding 调用失败 |
| `MODEL_RERANK_FAILED` | rerank 调用失败 |
| `MODEL_LLM_TIMEOUT` | LLM 超时 |
| `MODEL_ROUTE_NOT_FOUND` | 找不到模型路由 |
| `MODEL_POOL_OVERLOADED` | 模型池过载 |
| `MODEL_VERSION_INCOMPATIBLE` | 模型版本和索引不兼容 |

## 13. 测试点

- 业务模块只调用 Model Gateway。
- 未授权外部模型不会被路由。
- embedding 维度不匹配会失败。
- rerank 超时返回可降级错误。
- LLM 流式输出支持取消。
- token usage 和 latency 正确记录。
