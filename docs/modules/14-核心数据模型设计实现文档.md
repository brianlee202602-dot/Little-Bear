# 核心数据模型设计实现文档

## 1. 模块目标

本文定义 Little Bear RAG 系统的核心业务数据模型，补齐知识库、文件夹、文档、文档版本、chunk、权限快照、索引版本、删除阻断和查询缓存之间的统一契约，避免各模块各自定义字段导致权限不一致、半成品索引可见、缓存串权或引用不可回溯。

## 2. 设计原则

- PostgreSQL 是业务事实源，Qdrant、关键词索引和缓存都只能作为派生数据。
- 查询可见性必须同时满足业务状态、索引状态、权限快照和删除阻断。
- 文档内容、权限快照和索引版本都必须可追溯。
- 任何索引写入先进入 draft/staging 状态，发布成功后才允许查询命中。
- 权限收紧和文档删除必须 fail closed，不能等待异步物理删除完成后才生效。
- 缓存 key 必须绑定租户、用户可见范围、显式过滤条件、配置版本、索引版本和模型/Prompt 版本。

## 3. 实体关系

```text
Enterprise
  -> User / Department / Group
  -> KnowledgeBase
      -> Folder
      -> Document
          -> DocumentVersion
              -> Chunk
                  -> ChunkIndexRef
                  -> PermissionSnapshot

IndexVersion
  -> ChunkIndexRef

ResourcePolicy
  -> PermissionSnapshot

AccessBlock / Tombstone
  -> Document / Chunk
```

## 4. 知识库与文件夹

`knowledge_bases`：

| 字段 | 说明 |
| --- | --- |
| id | 知识库 ID |
| enterprise_id | 企业 ID |
| name | 名称 |
| status | active、disabled、archived |
| owner_department_id | 归属部门 |
| default_visibility | 默认可见范围，department 或 enterprise |
| policy_version | 知识库权限策略版本 |
| config_scope_id | 配置作用域 |
| created_at / updated_at | 时间戳 |

`folders`：

| 字段 | 说明 |
| --- | --- |
| id | 文件夹 ID |
| enterprise_id | 企业 ID |
| kb_id | 知识库 ID |
| parent_id | 父文件夹 |
| path | 层级路径 |
| policy_inherit_mode | inherit、override、merge |
| status | active、disabled、archived |

约束：

- `kb_id` 必须和 `enterprise_id` 一起参与唯一和查询条件。
- 文件夹循环引用必须被数据库约束或 service policy 阻止。
- 文件夹权限继承结果不能在查询时临时猜测，必须在权限快照中落地。

- 文档删除先写 `lifecycle_status = deleted` 和阻断记录，再异步清理索引。

## 5.1 文档可见性策略

`document_visibility_policies`：

| 字段 | 说明 |
| --- | --- |
| id | 可见性策略 ID |
| enterprise_id | 企业 ID |
| document_id | 文档 ID |
| owner_department_id | 文档归属部门 |
| visibility | department、enterprise |
| policy_hash | 可见性策略哈希 |
| status | draft、active、archived |
| created_by / created_at | 创建信息 |

可见性约束：

- 默认 `visibility = department`，仅 `owner_department_id` 对应部门成员可见。
- `visibility = enterprise` 表示企业公开，当前企业内所有部门成员可见。
- 文档权限不支持上级部门、子部门递归、自定义组织 ID、指定用户或项目组共享。
- 查询和索引只使用 `visibility`、`owner_department_id` 和 `enterprise_id`，不在查询时递归组织关系，也不先全量召回后遍历数据库筛选可见文档。
- `owner_department_id` 或 `visibility` 变化后，必须生成新的权限快照并刷新索引 payload。

## 6. Chunk 与引用

`chunks`：

| 字段 | 说明 |
| --- | --- |
| id | chunk ID |
| enterprise_id | 企业 ID |
| kb_id | 知识库 ID |
| document_id | 文档 ID |
| document_version_id | 文档版本 ID |
| ordinal | 文档内序号 |
| text_object_key | 大文本对象 key，可选 |
| text_preview | 限长预览 |
| heading_path | 标题路径 |
| page_start / page_end | 页码范围 |
| source_offsets | 原文偏移 |
| content_hash | chunk 内容哈希 |
| token_count | token 数 |
| quality_flags | OCR 低置信、表格、代码等标记 |
| status | draft、active、archived、deleted |
| permission_snapshot_id | 权限快照 |
| created_at / updated_at | 时间戳 |

引用约束：

- 对外引用使用 `chunk_id + document_version_id + page_start/page_end`。
- 引用校验必须确认 chunk 仍为 active、文档未删除、权限快照未过期。
- Prompt 中的 `source_id` 必须映射到真实 chunk，不能由 LLM 自由生成。

## 7. 权限策略与快照

`resource_policies`：

| 字段 | 说明 |
| --- | --- |
| id | 策略 ID |
| enterprise_id | 企业 ID |
| resource_type | knowledge_base、folder、document、chunk |
| resource_id | 资源 ID |
| policy_json | 文档可见性、管理操作授权或其他资源策略 |
| version | 策略版本 |
| status | active、archived |
| created_by / created_at | 创建信息 |

`permission_snapshots`：

| 字段 | 说明 |
| --- | --- |
| id | 权限快照 ID |
| enterprise_id | 企业 ID |
| resource_type | document、chunk |
| resource_id | 资源 ID |
| permission_version | 全局权限版本 |
| policy_version | 资源策略版本 |
| payload_json | 写入索引的权限 payload |
| payload_hash | payload 哈希 |
| owner_department_id | 文档归属部门 |
| visibility | department、enterprise |
| created_at | 创建时间 |

约束：

- 索引中只写权限快照结果，不在查询时重新推导文件夹继承或组织关系。
- 向量索引和关键词索引 payload 必须写入 `owner_department_id` 和 `visibility`，查询时通过 `visibility = enterprise OR owner_department_id IN user.departments` 做前置过滤。
- 权限收紧先创建阻断记录并递增 `permission_version`，再异步刷新索引 payload。
- 索引权限版本落后于资源当前权限版本时必须拒绝候选或回源确认，不允许直接放行。

查询可见性条件：

```text
enterprise_id = current_enterprise
AND kb_id IN requested_kbs
AND document.lifecycle_status = active
AND document.index_status = indexed
AND chunk.status = active
AND chunk_index_ref.visibility_state = active
AND index_version.status = active
AND is_deleted = false
AND access_block NOT EXISTS
AND permission_filter MATCHES
```

## 9. 删除与阻断

`access_blocks`：

| 字段 | 说明 |
| --- | --- |
| id | 阻断 ID |
| enterprise_id | 企业 ID |
| resource_type | document、chunk、knowledge_base |
| resource_id | 资源 ID |
| reason | deleted、permission_tightened、legal_hold、security_incident |
| block_level | query、citation、all |
| created_by | 操作者 |
| expires_at | 过期时间，可为空 |
| created_at | 创建时间 |

删除流程：

```text
1. PostgreSQL 事务内写 access_block
2. documents.lifecycle_status = deleted
3. chunks.status = deleted
4. chunk_index_refs.visibility_state = blocked
5. 提交后发送索引物理删除任务
6. Qdrant 和关键词索引异步删除或 payload 标记 deleted
```

要求：

- 查询候选进入上下文前必须做 access block gate。
- 物理索引删除失败只影响存储成本，不能影响查询阻断。
- 物理索引删除持续失败必须告警并进入重试队列。

## 10. 查询缓存模型

`query_cache_entries`：

| 字段 | 说明 |
| --- | --- |
| cache_key | 完整缓存 key hash |
| enterprise_id | 企业 ID |
| user_id | 用户 ID，最终答案缓存默认按用户隔离 |
| permission_filter_hash | 权限过滤条件哈希 |
| request_filter_hash | 用户显式过滤条件哈希 |
| kb_ids_hash | 请求知识库集合哈希 |
| config_version | 配置版本 |
| index_version_hash | active index 版本集合哈希 |
| model_route_hash | embedding、rerank、LLM 路由哈希 |
| prompt_template_version | Prompt 模板版本 |
| value_json | 缓存值 |
| expires_at | 过期时间 |

缓存约束：

- 查询结果缓存默认按用户隔离。
- 如需跨用户共享，只能缓存检索中间结果，并且必须绑定 `permission_filter_hash`。
- 权限版本刷新中、引用校验失败或降级不完整的响应不得缓存。
- 缓存命中后仍需执行轻量 access block gate 和引用有效性检查。

## 11. 必须保持的跨模块不变量

- `documents.current_version_id` 指向的版本必须存在且状态为 active。
- active `index_versions` 的 embedding 模型、维度必须和查询 embedding 模型匹配。
- active `chunk_index_refs` 必须指向 active chunk 和 active index version。
- `permission_snapshots.payload_hash` 必须和索引 payload hash 可对账。
- 查询响应中的每个 citation 必须能回溯到 active chunk、document version 和 object storage 原文。
- 所有缓存和审计事件必须记录 `enterprise_id`、`config_version`、`permission_version` 和必要的索引版本信息。

## 12. 测试点

- draft index 写入后查询不可见。
- index publish 后新版本可见，旧版本按策略 archived。
- 文档删除后即使 Qdrant 物理删除失败也不能被查询命中。
- 权限收紧后旧权限用户不能通过缓存或旧索引访问。
- 查询缓存不会跨用户、跨租户、跨权限过滤条件复用。
- citation 可以回溯到文档版本、页码、chunk 和对象存储原文。
