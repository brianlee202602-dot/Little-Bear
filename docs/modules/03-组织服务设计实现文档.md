# Org Service 设计实现文档

## 1. 模块目标

Org Service 负责维护企业、部门、成员三层组织关系，为权限上下文构建提供稳定的数据来源。

## 2. 模块职责

- 企业、部门、成员基础数据管理。
- 用户部门关系管理。
- 组织版本号维护。
- 组织结构同步任务。
- 为 Permission Service 提供 subject 集合。

## 3. 非职责

- 不决定文档是否可访问。
- 不直接写向量索引权限字段。
- 不处理登录和会话。
- 不负责知识库权限策略。
- 不创建角色定义，不直接分配角色；部门角色绑定由 Permission Service 管理。

## 5. 核心实体

```text
Enterprise
  -> Department

User
  -> Department memberships
```

`departments`：

| 字段 | 说明 |
| --- | --- |
| id | 部门 ID |
| enterprise_id | 企业 ID |
| name | 部门名称 |
| status | active、disabled |
| version | 组织版本 |

组织约束：

- 最小生产阶段组织单位收敛为三层：企业、部门、成员。
- 部门不建模上下级关系，不维护 `parent_id`、`path`、`department_closure` 或递归展开。
- 文档权限只使用成员直接所属部门和企业 ID；不存在上级部门可见、子部门可见或包含下级部门的语义。
- 组织变更只需要递增 `org_version` 并失效用户 subject 缓存，不触发文档权限展开刷新。

`user_department_memberships`：

| 字段 | 说明 |
| --- | --- |
| user_id | 用户 ID |
| department_id | 部门 ID |
| relation_type | primary、secondary、temporary |
| valid_from / valid_to | 有效期 |

## 6. 组织同步流程

最小阶段可由管理员手动维护。后续接入外部源时使用同步任务：

```text
1. scheduled-worker 拉取外部组织数据
2. 转换为内部标准结构
3. 校验重复部门、无效用户、无效部门关系
4. 写入 staging 表
5. 生成 diff
6. 管理员确认或自动发布
7. 更新 departments、memberships
8. 增加 org_version
9. 触发权限缓存失效
```

## 7. 对 Permission Service 的接口

```python
class OrgService:
    def get_user_subjects(user_id: str) -> UserSubjects:
        ...

    def get_department(department_id: str) -> Department:
        ...
```

`UserSubjects` 示例：

```json
{
  "user_id": "user_123",
  "departments": ["dept_rnd"],
  "primary_department": "dept_rnd",
  "org_version": 17
}
```

语义约束：

- `UserSubjects.departments` 表示用户直接所属的 active 部门，用于 `owner_department_id IN user.departments` 精确匹配。
- 查询链路不计算祖先部门、后代部门或组织树展开。
- 文档权限只有部门可见和企业公开：部门可见匹配用户直接部门，企业公开匹配 `enterprise_id`。

## 8. 缓存策略

- 用户 subject 集合可缓存到 Redis。
- 缓存 key 包含 `user_id` 和 `org_version`。
- 组织版本更新后失效相关用户缓存。
- 不缓存部门祖先或后代关系。

## 9. 管理 API

```http
GET    /internal/v1/admin/departments
POST   /internal/v1/admin/departments
GET    /internal/v1/admin/departments/{department_id}
PATCH  /internal/v1/admin/departments/{department_id}
DELETE /internal/v1/admin/departments/{department_id}

GET  /internal/v1/admin/users/{user_id}/departments
PUT  /internal/v1/admin/users/{user_id}/departments
```

所有写接口必须校验管理员权限并记录审计。

部门管理边界：

- 组织结构由系统管理员或具备组织管理权限的管理员维护。
- 部门管理员身份本身不是 Org Service 决定的，而是 Permission Service 的 `department_admin` 角色绑定结果。
- 部门管理员可以在被授权部门范围内维护成员关系和发起用户创建，但角色分配必须调用 Permission Service。
- 部门管理员不得修改其他部门或全局角色定义。

## 10. 错误码

| 错误码 | 场景 |
| --- | --- |
| `ORG_DEPARTMENT_NOT_FOUND` | 部门不存在 |
| `ORG_MEMBERSHIP_INVALID` | 用户部门关系无效 |
| `ORG_SYNC_FAILED` | 外部组织同步失败 |

## 11. 审计事件

- `org.department.created`
- `org.department.updated`
- `org.department.disabled`
- `org.membership.changed`
- `org.sync.published`

## 12. 测试点

- 一个用户可属于一个主部门，也可按业务需要属于多个部门。
- 组织版本变更会失效权限缓存。
- 部门没有上下级递归关系。
- 普通用户不能修改组织结构。
