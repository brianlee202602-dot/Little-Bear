# Setup Service 设计实现文档

## 1. 模块目标

Setup Service 负责 API 服务启动后的首次初始化流程，让系统实现最小启动配置：

- 环境变量只提供 API 进程启动和基础连接能力。
- 系统未初始化时生成临时 `system_token`。
- 临时 token 只允许访问初始化接口。
- 初始化接口一次性接收完整业务配置。
- 校验通过后发布 `active_config v1`，创建首个管理员，并开放业务 API。

## 2. 模块职责

- 维护 `system_state`。
- 生成、展示、校验、失效临时 `system_token`。
- 拦截未初始化状态下的普通业务 API。
- 提供 setup status、validate、initialize 接口。
- 执行初始化配置 schema 校验和依赖连通性测试。
- 创建第一个系统管理员、默认角色、默认部门和基础权限。
- 发布初始化配置版本。
- 记录初始化审计事件。

## 3. 非职责

- 不负责后续管理员配置变更审批。
- 不负责普通用户登录。
- 不负责业务查询、导入、索引。
- 不通过 HTTP 返回 token 明文。

## 4. 建议包结构

```text
app/modules/setup/
  service.py
  token_service.py
  state_repository.py
  initialize_service.py
  validators.py
  dependency_checks.py
  schemas.py
  errors.py
  events.py
```

## 5. 核心状态

`system_state` 建议用 key-value 表承载：

| key | 说明 |
| --- | --- |
| `initialized` | 是否完成初始化 |
| `setup_status` | 初始化状态机状态 |
| `active_config_version` | 当前 active config 版本 |
| `system_token_hash` | 临时 token hash |
| `system_token_expires_at` | token 过期时间 |
| `system_token_used` | token 是否已使用 |
| `setup_attempt_count` | 初始化失败次数 |
| `setup_locked_until` | 失败过多后的锁定时间 |

状态机：

```text
not_initialized
  -> setup_required
  -> validating_config
  -> testing_dependencies
  -> creating_admin
  -> publishing_config
  -> initialized

异常状态：
  -> validation_failed
  -> dependency_test_failed
  -> initialization_failed
```

## 6. 启动流程

```text
1. API 进程读取最小环境变量
2. 连接 PostgreSQL、Redis、Secret Provider
3. SetupService.load_state()
4. initialized = true:
   - ConfigService.load_active_config()
   - 开放业务 API
5. initialized = false:
   - SetupTokenService.ensure_token()
   - 进入 setup_required
   - 普通业务 API 返回 SETUP_REQUIRED
```

## 7. 临时 system token

生成规则：

- 仅在 `initialized = false` 时可生成。
- 如果已有未过期且未使用 token，则复用。
- 使用 256-bit 安全随机数。
- 数据库只保存 hash，明文只展示一次。
- 默认有效期 30 到 60 分钟。
- token 具备临时 `system` 等级和 `setup:*` scope。
- 初始化成功后立即失效。

获取方式：

```text
docker compose exec api rag-admin setup-token show
docker compose exec api rag-admin setup-token rotate
```

也可以写入受限临时文件：

```text
/run/rag/setup_token
```

初始化完成后必须删除临时文件。

## 8. API 设计

```http
GET  /internal/v1/setup/status
POST /internal/v1/setup/validate
POST /internal/v1/setup/initialize
```

鉴权：

```http
GET  /internal/v1/setup/status      不需要 token，只返回有限状态
POST /internal/v1/setup/validate    Authorization: System <system_token>
POST /internal/v1/setup/initialize  Authorization: System <system_token>
```

`status` 响应示例：

```json
{
  "initialized": false,
  "setup_status": "setup_required",
  "system_token_expires_at": "2026-04-29T20:00:00+08:00"
}
```

## 9. 初始化请求

初始化请求必须覆盖业务最小运行配置：

- 首个管理员账号。
- 本地认证策略。
- 对象存储。
- 向量库。
- 关键词检索。
- 模型网关。
- 检索策略。
- 中文分词、企业词库和同义词策略。
- 导入策略。
- 权限策略。
- 查询缓存策略。
- 安全策略。
- 限流策略。
- 审计策略。

敏感配置优先传 Secret 引用，例如：

```json
{
  "storage": {
    "minio_endpoint": "http://minio:9000",
    "access_key_ref": "secret://rag/minio/access-key",
    "secret_key_ref": "secret://rag/minio/secret-key"
  }
}
```

## 10. 校验流程

`validate` 和 `initialize` 共用校验逻辑：

```text
1. 校验 token、状态和失败次数
2. 校验 JSON schema
3. 校验 Secret 引用存在且可访问
4. 校验管理员账号密码强度
5. 测试 PostgreSQL 可读写
6. 测试 Redis 可读写和锁
7. 测试 MinIO bucket
8. 测试 Qdrant collection 创建、写入、删除
9. 测试 PostgreSQL Full Text 和中文分词器
10. 测试 Model Gateway、vLLM、TEI embedding、TEI rerank
11. 校验最终答案缓存策略不会用于高密级或 strict permission mode 查询
12. 执行端到端自检
13. 返回配置校验结果
```

`validate` 不写 active config。`initialize` 在校验通过后进入事务。

## 11. 初始化事务

```text
1. 开启数据库事务
2. 创建系统管理员用户
3. 创建默认部门、默认角色、基础权限
4. 写入 system_configs draft
5. 创建 config_versions v1
6. 标记 active_config_version = 1
7. 标记 initialized = true
8. 标记 system_token_used = true
9. 提交事务
10. 触发运行时配置加载
```

要求：

- 管理员账号、默认权限、配置版本、初始化状态要么全部成功，要么全部回滚。
- 外部测试 collection、测试对象和测试数据必须清理。
- active config 加载成功后才开放业务 API。

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `SETUP_REQUIRED` | 系统未初始化，普通业务 API 被拦截 |
| `SETUP_ALREADY_INITIALIZED` | 初始化完成后再次调用 initialize |
| `SETUP_TOKEN_INVALID` | token 不存在或 hash 不匹配 |
| `SETUP_TOKEN_EXPIRED` | token 已过期 |
| `SETUP_TOKEN_USED` | token 已使用 |
| `SETUP_LOCKED` | 初始化失败过多被锁定 |
| `SETUP_CONFIG_INVALID` | 配置 schema 校验失败 |
| `SETUP_DEPENDENCY_FAILED` | 依赖连通性测试失败 |
| `SETUP_INITIALIZE_FAILED` | 初始化事务失败 |

## 13. 审计事件

- `setup.token.created`
- `setup.token.rotated`
- `setup.validate.started`
- `setup.validate.failed`
- `setup.initialize.started`
- `setup.initialize.succeeded`
- `setup.initialize.failed`

审计不能记录 token 明文、密码明文或 secret value。

## 14. 测试点

- 未初始化时普通业务 API 返回 `SETUP_REQUIRED`。
- status 不返回 token 明文。
- validate 缺 token 返回 `SETUP_TOKEN_INVALID`。
- 初始化成功后 token 失效。
- 初始化失败不产生半初始化状态。
- 依赖测试失败会清理测试数据。
- initialized=true 但 active config 缺失时拒绝服务并告警。
- 初始化配置禁止开启高密级最终答案缓存。
- 中文关键词检索配置缺失时，validate 必须明确提示召回质量风险。
