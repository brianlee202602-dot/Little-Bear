# Setup Service 设计实现文档

## 1. 模块目标

Setup Service 负责 API 服务启动后的首次初始化流程，让系统实现最小启动配置：

- 启动配置只包含数据库连接内容。
- Redis、Secret Provider、对象存储、向量库和模型服务等都来自数据库配置。
- 系统未初始化，或系统已初始化但配置损坏并进入恢复初始化流程时，生成临时 `system_token`。
- 临时 token 只允许访问初始化接口。
- 初始化接口一次性接收完整业务配置。
- 校验通过后发布 `active_config v1`，创建首个系统管理员和内置默认角色，并在 ServiceBootstrap 成功后开放业务 API。

## 2. 模块职责

- 维护 `system_state`。
- 生成、展示、校验、失效临时 `system_token`。
- 拦截未初始化状态下的普通业务 API。
- 提供 setup-state、setup-config-validations、setup-initialization 接口。
- 执行初始化配置 schema 校验和依赖连通性测试。
- 创建第一个系统管理员、内置默认角色、默认部门和基础权限。
- 发布初始化配置版本。
- 记录初始化审计事件。

## 3. 非职责

- 不负责后续管理员配置变更审批。
- 不负责普通用户登录。
- 不负责业务查询、导入、索引。
- 不通过 HTTP 返回 token 明文。

## 4. 建议包结构

```text
app/modules/setup/

  token_service.py
  state_repository.py

```

## 5. 核心状态

`system_state` 建议用 key-value 表承载：

| key | 说明 |
| --- | --- |
| `initialized` | 是否完成初始化 |
| `setup_status` | 初始化状态机状态 |
| `active_config_version` | 当前 active config 版本 |
| `permission_version` | 当前权限版本 |
| `schema_migration_version` | 当前核心 Schema migration 版本 |
| `recovery_setup_allowed` | initialized=true 但配置不可用时是否允许恢复初始化 |
| `recovery_reason` | 恢复初始化原因，例如 active_config_missing、config_corrupted、bootstrap_failed |
| `setup_attempt_count` | 初始化失败次数 |
| `setup_locked_until` | 失败过多后的锁定时间 |

setup JWT 的签发、失效、过期和一次性使用状态不放在 `system_state` 中，统一写入 `jwt_tokens` 和 `setup_tokens`。每次签发新 setup JWT 时，必须在同一事务内吊销旧的 active setup token。

状态机：

```text
not_initialized
  -> setup_required
  -> validating_config
  -> testing_dependencies
  -> creating_admin
  -> publishing_config
  -> initialized

异常状态：
  -> validation_failed
  -> dependency_test_failed
  -> initialization_failed
  -> recovery_required
  -> recovery_validating_config
  -> recovery_publishing_config
```

## 6. 启动流程

```text
1. API 进程读取数据库连接启动配置
2. 校验 `DATABASE_URL` 等数据库连接配置
3. 数据库连接配置缺失或 PostgreSQL 连接失败:
   - 启动失败或 readiness=false
   - 不生成 system_token
   - 不进入 setup_required
   - 不开放 setup 初始化写接口或业务 API
4. PostgreSQL 连接成功后 SetupService.load_state()
5. system_state 不存在、system_state 为空、initialized = false，或数据库内无任何配置数据:
   - SetupTokenService.issue_token()
   - 进入 setup_required
   - 仅开放 setup-state / setup-config-validations / setup-initialization
   - 不开放普通业务 API；只有初始化成功并完成 ServiceBootstrap 后才开放业务 API
6. initialized = true 且 active config 存在:
   - ConfigService.load_active_config()
   - ServiceBootstrap.initialize_modules(active_config)
   - 初始化 Redis、Secret Provider、ObjectStorage、VectorStore、KeywordSearch、模型 Provider Adapter、Audit Sink、Rate Limiter
   - 全部成功后开放业务 API
7. initialized = true 但 active_config_version 缺失、active config 记录损坏，或任一关键模块初始化失败:
   - 进入拒绝模式
   - readiness=false
   - 写告警和结构化错误
   - setup_status = recovery_required
   - 返回管理员提示：当前系统配置不可用，是否进入恢复初始化流程；确认后设置 recovery_setup_allowed=true，生成或轮换 system_token，并走 setup-config-validations / setup-initialization 流程
```

ServiceBootstrap 要求：

- 输入只能是数据库中的 active config，不读取 Redis、Secret Provider 或其他业务依赖的环境变量。
- Redis、Secret Provider、ObjectStorage、VectorStore、KeywordSearch、模型 Provider Adapter、Audit Sink、Rate Limiter 都必须显式初始化并通过健康检查。
- 任一关键模块初始化失败时，系统保持 live 但 not ready，普通业务 API 返回结构化错误。
- 未初始化状态下不执行业务模块 ServiceBootstrap，只允许 Setup Service 使用 PostgreSQL 生成和校验 system token。

## 7. 临时 system token

临时 `system_token` 采用 JWT，token_type 固定为 `setup`，用于和普通用户 access token、refresh token、service token 保持统一鉴权机制。

生成规则：

- 允许生成 token 的状态：
  - `initialized = false` 且 `setup_status` 为 `setup_required`。
  - `initialized = true` 且 `setup_status = recovery_required`，并且管理员已确认 `recovery_setup_allowed = true`。
- 每次请求签发临时 token 都必须生成新 token，不复用已有 token。
- 签发新 token 前必须将已有未过期、未使用的 setup token 标记为失效，且只允许最新 `jti` 通过校验。
- 使用 256-bit 安全随机数。
- JWT 必须包含 `jti`、`token_type=setup`、`scopes=["setup:validate","setup:initialize"]`、`setup_status`、`iat`、`exp`。
- 数据库只保存 `jti`、token hash 和状态，JWT 明文只展示一次；校验时只允许最新 active setup token 通过。
- 默认有效期 30 到 60 分钟。
- 初始化成功后立即失效。
- 恢复初始化 token 只能用于 `setup:validate` 和 `setup:initialize`，不能访问普通业务 API 或管理 API。
- Setup Service 和 Auth Service 共享 JWT 校验组件、签名算法和 key 轮换策略。

获取方式：

```text
docker compose exec api rag-admin setup-token issue
docker compose exec api rag-admin setup-token rotate
```

`issue` 和 `rotate` 都会签发新的 setup JWT，并使旧 setup JWT 立即失效。

也可以写入受限临时文件：

```text
/run/rag/setup_token
```

初始化完成后必须删除临时文件。

## 8. API 设计

```http
GET  /internal/v1/setup-state
POST /internal/v1/setup-config-validations
PUT  /internal/v1/setup-initialization
```

鉴权：

```http
GET  /internal/v1/setup-state                不需要 token，只返回有限状态
POST /internal/v1/setup-config-validations   Authorization: Bearer <setup_jwt>
PUT  /internal/v1/setup-initialization       Authorization: Bearer <setup_jwt>
```

`setup-state` 响应示例：

```json
{
  "initialized": false,
  "setup_status": "setup_required",
  "system_token_expires_at": "2026-04-29T20:00:00+08:00"
}
```

## 9. 初始化请求

初始化请求必须一次性提交系统首次可运行所需的完整配置。请求分为两类：

- 初始化实体数据：首个管理员账号、默认组织、内置默认角色、基础权限和首个 `system_admin` 绑定。这些数据写入用户、组织和权限相关业务表，不写入 active config。
- 初始化运行配置：业务依赖地址、模型路由、检索、导入、权限、安全、缓存、限流、审计等配置。这些配置必须写入 `system_configs`，生成 `config_versions v1`，并标记为 `active_config v1`。

初始化运行配置可以在业务启动后由具备权限的管理员通过 Config Service 修改，但首次初始化必须提供一个完整初始版本，不能依赖环境变量或隐式代码默认值补齐关键依赖。

初始化请求最小清单：

| 分组 | 必填内容 | 落库位置 | 后续是否可改 |
| --- | --- | --- | --- |
| `setup.admin` | 首个管理员用户名、显示名、初始密码或密码设置凭证、联系方式 | `users`、`user_identities`、角色绑定表 | 账号可改；不是 active config |
| `setup.organization` | 默认企业、初始部门清单 | 组织表 | 可改，部门不建模上下级递归 |
| `setup.roles` | 内置默认角色、普通用户默认角色、首个管理员绑定 `system_admin` | 角色和权限表；默认角色策略写入 active config | 角色定义由 Permission Service 管理 |
| `secret_provider` | Secret Store 类型、endpoint、访问方式、密钥引用解析规则 | active config；敏感值存 Secret Store | 可改，需依赖校验 |
| `redis` | Redis URL、连接池、锁、限流和配置通知使用方式 | active config | 可改，需连通性测试 |
| `storage` | 对象存储类型、MinIO endpoint、bucket、region、TLS、access key ref、secret key ref、对象 key 前缀 | active config + Secret 引用 | 可改，需 bucket 测试 |
| `vector_store` | 向量库类型、Qdrant base URL、collection 前缀、API key ref、距离度量、写入和删除测试策略 | active config + Secret 引用 | 可改，需索引兼容性检查 |
| `keyword_search` | PostgreSQL Full Text 配置、中文分词器、企业词库版本、同义词版本、停用词版本、字段 boost | active config | 可改，大规模变更需审批 |
| `model_gateway` | 外部 embedding/rerank/LLM provider URL、认证方式、模型路由和健康检查 | active config + Secret 引用 | 可改，外部模型路由需审批 |
| `model` | embedding/rerank/LLM 默认模型，embedding model name、version、dimension、normalize、tokenizer/version | active config | 可改，embedding 维度或语义空间变化需索引兼容检查 |
| `llm` | temperature、max tokens、首 token 超时、总超时、重试策略 | active config | 可改 |
| `auth` | 本地密码策略、登录失败锁定、access token TTL、refresh token TTL、JWT issuer/audience、JWT signing key ref | active config + Secret 引用 | 可改，token/key 变更需兼容已有会话 |
| `retrieval` | vector topK、keyword topK、fusion method、RRF 参数、rerank topK、final context topK、max context tokens、rewrite/expansion 开关 | active config | 可改 |
| `chunk` | 默认切块策略、chunk size、overlap、表格/FAQ/合同条款拆分策略 | active config | 可改，仅影响后续导入或重建索引 |
| `import` | 最大文件大小、允许文件类型、全局/部门/用户/任务文件并发、embedding batch size、index batch size、重试和 dead-letter 策略 | active config | 可改 |
| `permission` | 默认可见性 `department`、企业公开策略、默认角色分配策略、权限缓存 TTL、权限收紧阻断策略 | active config | 可改，企业公开策略可配置审批 |
| `security` | require citation、Prompt 防泄露策略、敏感信息脱敏策略 | active config | 可改，高风险策略需审批 |
| `cache` | 权限上下文缓存、query embedding 缓存、检索中间结果缓存、最终答案缓存开关、TTL、跨用户共享限制 | active config | 可改，最终答案缓存策略需审批 |
| `rate_limit` | IP、user、department、kb、api_key、model_pool、import_job 维度限流阈值和窗口 | active config | 可改 |
| `timeout` | 查询总预算和 auth、permission、rewrite、embedding、vector search、keyword search、fusion、rerank、context、LLM、postprocess 阶段预算 | active config | 可改 |
| `degrade` | rewrite、embedding、vector search、keyword search、rerank、context compression、LLM、模型池满载、导入积压的降级策略 | active config | 可改 |
| `audit` | 审计 sink、保留天数、查询原文 hash/明文策略、完整 prompt 记录策略、snippet 限长、PII 脱敏策略 | active config | 可改，记录原文/完整 prompt 或缩短保留周期需审批 |
| `observability` | 指标、Trace、告警阈值，例如 active config 加载失败、缓存串权、索引权限落后、导入失败率、队列积压、LLM 超时率 | active config | 可改 |

初始化请求建议结构：

```json
{
  "setup": {
    "admin": {},
    "organization": {},
    "roles": {}
  },
  "config": {
    "secret_provider": {},
    "redis": {},
    "storage": {},
    "vector_store": {},
    "keyword_search": {},
    "model_gateway": {},
    "model": {},
    "llm": {},
    "auth": {},
    "retrieval": {},
    "chunk": {},
    "import": {},
    "permission": {},
    "security": {},
    "cache": {},
    "rate_limit": {},
    "timeout": {},
    "degrade": {},
    "audit": {},
    "observability": {}
  }

```

Secret Provider / Secret Store 是密钥管理抽象，不是启动依赖。初始化请求可以选择：

- 使用 PostgreSQL 加密 `secrets` 表作为最小 Secret Store。
- 配置外部 Vault、云 KMS、Kubernetes Secret、Docker Secret 或企业内部密钥管理服务。

无论选择哪种方式，active config 中都应优先保存 Secret 引用，而不是保存密钥明文。

敏感配置优先传 Secret 引用，例如：

```json
{
  "storage": {
    "minio_endpoint": "http://minio:9000",
    "access_key_ref": "secret://rag/minio/access-key",
    "secret_key_ref": "secret://rag/minio/secret-key"
  }
}
```

## 10. 校验流程

`validate` 和 `initialize` 共用校验逻辑：

```text
1. 校验 token、状态和失败次数
2. 校验 JSON schema
3. 按初始化请求中的配置校验 Secret Provider 配置、Secret 引用存在且可访问
4. 校验管理员账号密码强度
5. 测试 PostgreSQL 可读写
6. 使用初始化请求中的 Redis 配置测试可读写和锁
7. 使用初始化请求中的 MinIO 配置测试 bucket
8. 使用初始化请求中的 Qdrant 配置测试 collection 创建、写入、删除
9. 测试 PostgreSQL Full Text 和中文分词器
10. 测试 TEI embedding、TEI rerank 和 OpenAI-compatible LLM provider
11. 校验最终答案缓存策略不会跨用户、跨租户或跨权限过滤条件复用
12. 执行端到端自检
13. 返回配置校验结果
```

`validate` 不写 active config。`initialize` 在校验通过后进入事务。

`initialize` 支持两种模式：

- 首次初始化：`initialized = false`，创建首个系统管理员、内置默认角色、默认组织和 `active_config v1`。
- 恢复初始化：`initialized = true` 且 `setup_status = recovery_required`，只修复或重新发布配置，不重复创建首个系统管理员、内置默认角色和默认组织。

内置默认角色：

| 角色 | 创建者 | 说明 |
| --- | --- | --- |
| `system_admin` | Setup Service | 系统管理员，可管理全局配置、用户、角色和恢复初始化；初始化时只创建首个账号，后续可有多个 system_admin |
| `security_admin` | Setup Service | 安全管理员，可管理高风险权限策略和安全审批 |
| `audit_admin` | Setup Service | 审计管理员，可查看审计、评测和高风险操作记录，默认不具备业务数据修改权限 |
| `department_admin` | Setup Service | 部门管理员角色模板，只能在授权部门范围内管理成员和部门级授权 |
| `knowledge_base_admin` | Setup Service | 知识库管理员角色模板，只能在授权知识库范围内管理文档、文件夹和导入 |
| `employee` | Setup Service | 普通员工默认角色，普通用户创建或同步时默认分配 |

要求：

- “首个管理员”不是单管理员限制，只是系统首次初始化入口。
- 默认角色由 Setup Service 初始化创建，后续角色定义和角色分配由 Permission Service 管理。
- 普通用户账号创建或外部身份同步时默认分配 `employee`，具体角色绑定必须经过 Permission Service 校验。

## 11. 初始化事务

首次初始化事务：

```text
1. 开启数据库事务
2. 创建首个系统管理员用户
3. 创建默认部门、内置默认角色、基础权限和首个 system_admin 角色绑定
4. 写入 system_configs draft
5. 创建 config_versions v1
6. 标记 active_config_version = 1
7. 标记 initialized = true
8. 标记当前 `setup_tokens.status = used`，并同步更新对应 `jwt_tokens.status = used`
9. 提交事务
10. 触发运行时配置加载和 ServiceBootstrap
```

恢复初始化事务：

```text
1. 开启数据库事务
2. 校验 initialized = true
3. 校验 setup_status = recovery_required
4. 校验 recovery_setup_allowed = true
5. 写入 system_configs draft
6. 创建新的 config_versions
7. 标记 active_config_version = 新版本
8. 清理 recovery_setup_allowed 和 recovery_reason
9. 标记 setup_status = initialized
10. 标记当前 `setup_tokens.status = used`，并同步更新对应 `jwt_tokens.status = used`
11. 提交事务
12. 触发运行时配置加载和 ServiceBootstrap
```

要求：

- 管理员账号、默认权限、配置版本、初始化状态要么全部成功，要么全部回滚。
- 恢复初始化不得重复创建首个系统管理员、默认部门、内置默认角色或基础权限。
- 外部测试 collection、测试对象和测试数据必须清理。
- active config 加载成功且所有关键服务模块初始化成功后才开放业务 API。
- ServiceBootstrap 失败时不得标记系统 ready，也不得回退到环境变量或代码默认业务配置。

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `SETUP_REQUIRED` | 系统未初始化，普通业务 API 被拦截 |
| `SETUP_ALREADY_INITIALIZED` | 初始化完成后再次调用 initialize，且当前不处于恢复初始化流程 |
| `SETUP_TOKEN_INVALID` | setup JWT 无效、jti 不存在或 jti hash 不匹配 |
| `SETUP_TOKEN_EXPIRED` | setup JWT 已过期 |
| `SETUP_TOKEN_USED` | setup JWT 已使用 |
| `SETUP_LOCKED` | 初始化失败过多被锁定 |
| `SETUP_DATABASE_UNAVAILABLE` | 数据库连接配置缺失或 PostgreSQL 启动连接失败 |
| `SETUP_RECOVERY_NOT_ALLOWED` | initialized=true 但未进入恢复初始化流程 |
| `SETUP_CONFIG_INVALID` | 配置 schema 校验失败 |
| `SETUP_DEPENDENCY_FAILED` | 依赖连通性测试失败 |
| `SETUP_INITIALIZE_FAILED` | 初始化事务失败 |

## 13. 审计事件

- `setup.token.created`
- `setup.token.rotated`
- `setup.validate.started`
- `setup.validate.failed`
- `setup.initialize.started`
- `setup.initialize.succeeded`
- `setup.initialize.failed`

审计不能记录 token 明文、密码明文或 secret value。

## 14. 测试点

- 未初始化时普通业务 API 返回 `SETUP_REQUIRED`。
- 数据库连接配置缺失或 PostgreSQL 连接失败时不进入 setup mode。
- status 不返回 token 明文。
- validate 缺 token 返回 `SETUP_TOKEN_INVALID`。
- 初始化成功后 token 失效。
- initialized=true 且配置损坏进入恢复初始化流程时，可以生成或轮换 system token。
- 恢复初始化不会重复创建管理员、内置默认角色和默认组织。
- 首次初始化会创建内置默认角色，并给首个管理员绑定 `system_admin`。
- 初始化失败不产生半初始化状态。
- 依赖测试失败会清理测试数据。
- initialized=true 但 active config 缺失时拒绝服务并告警。
- 初始化配置禁止开启跨用户最终答案缓存。
- 中文关键词检索配置缺失时，validate 必须明确提示召回质量风险。
