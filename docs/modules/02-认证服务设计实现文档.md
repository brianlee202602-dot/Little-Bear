# Auth Service 设计实现文档

## 1. 模块目标

Auth Service 负责企业内部最小生产阶段的本地账号认证，并为后续接入 LDAP、OIDC、SAML、企业微信、飞书等身份源预留扩展点。

## 2. 模块职责

- 本地账号登录。
- 密码强度校验和密码哈希。
- JWT 签发、校验、吊销和刷新。
- 当前用户上下文加载。
- 用户账号创建、禁用、重置账号。
- 服务间调用鉴权。
- 登录审计和账号操作审计。

## 3. 非职责

- 不计算资源访问权限，资源权限由 Permission Service 负责。
- 不管理业务配置。
- 不直接读取组织权限索引。
- 不决定用户能检索哪些 chunk。
- 不创建角色定义，不直接分配角色；角色定义和角色绑定由 Permission Service 负责。

## 5. 核心表

`users`：

| 字段 | 说明 |
| --- | --- |
| id | 用户 ID |
| enterprise_id | 企业 ID |
| username | 登录账号 |
| password_hash | 密码哈希 |
| name | 姓名 |
| status | active、disabled、locked、deleted |
| attributes | 地区、岗位、职级等扩展属性 |
| created_at / updated_at | 时间戳 |

`jwt_tokens`，用于 JWT 吊销、刷新和审计：

| 字段 | 说明 |
| --- | --- |
| id | 记录 ID |
| jti | JWT ID |
| user_id | 用户 ID |
| token_type | access、refresh、service、setup |
| status | active、revoked、used、expired |
| expires_at | 过期时间 |
| created_ip | 登录 IP |
| user_agent | User-Agent 摘要 |

`external_identities`，后续扩展：

| 字段 | 说明 |
| --- | --- |
| provider | ldap、oidc、saml、wecom、feishu |
| external_subject | 外部身份唯一标识 |
| user_id | 本地用户 ID |

## 6. 用户创建与角色边界

账号创建流程：

```text
1. 操作者调用用户创建 API
2. Auth Service 鉴别操作者身份
3. Permission Service 校验操作者是否具备用户管理权限
4. Auth Service 创建 users 记录和认证凭证
5. Permission Service 为新用户绑定默认 `employee` 角色
6. 如请求包含额外角色，Permission Service 按操作者管理范围逐项校验后绑定
7. 写账号审计和角色绑定审计
```

职责边界：

- Auth Service 只负责账号、密码、状态、会话和外部身份映射。
- Permission Service 负责角色定义、角色绑定、角色解绑和权限版本递增。
- 系统管理员账号不是唯一的，首个系统管理员由 Setup Service 创建，后续系统管理员由已有 `system_admin` 或经审批的安全流程创建和授权。
- 部门管理员可以在被授权部门范围内创建或维护用户账号，但只能申请或分配自己有权管理的部门级角色。
- 普通用户账号创建或外部身份同步时默认绑定 `employee` 角色。

## 7. 登录流程

```text
1. 接收 username/password
2. 校验系统 initialized
3. 读取 active auth config
4. 查询用户
5. 校验用户状态
6. 校验密码 hash
7. 检查登录失败次数和锁定状态
8. 签发 JWT access token 和 refresh token
9. 写登录审计事件
10. 返回 access token
```

## 8. 密码策略

配置项：

- `auth.password_min_length`
- `auth.password_require_uppercase`
- `auth.password_require_lowercase`
- `auth.password_require_digit`
- `auth.password_require_symbol`
- `auth.login_failure_limit`
- `auth.lock_minutes`
- `auth.access_token_ttl_minutes`
- `auth.refresh_token_ttl_minutes`

实现要求：

- 密码使用 Argon2id 或 bcrypt。
- 密码明文只存在于请求生命周期内。
- 重置密码必须强制下次登录修改，或按企业策略立即生效。
- 登录失败次数写入 Redis 或数据库，必须有过期时间。

## 9. JWT Token

JWT claims 建议：

```json
{
  "sub": "user_123",
  "enterprise_id": "ent_001",
  "jti": "jwt_001",
  "token_type": "access",
  "auth_type": "local",
  "scopes": ["auth:user"],
  "iat": 1777468800,
  "exp": 1777497600
}
```

JWT 中不放完整权限上下文。权限上下文由 Permission Service 按用户、组织、角色和权限版本实时构建或从缓存读取。

JWT 类型：

| token_type | 用途 | 说明 |
| --- | --- | --- |
| `access` | 用户访问业务 API | 短 TTL，普通登录签发 |
| `refresh` | 刷新 access token | 长 TTL，可吊销 |
| `service` | 服务间调用 | 不映射普通用户，必须带 `service_name` 和 scopes |
| `setup` | 初始化和恢复初始化 | 由 Setup Service 签发，只允许访问 setup-config-validations / setup-initialization |

要求：

- 所有接口统一使用 `Authorization: Bearer <jwt>`。
- Auth Service 和 Setup Service 共享 JWT 校验组件、签名算法和 key 轮换策略。
- JWT 必须包含 `jti`，服务端保存 `jti` 状态用于吊销、一次性使用和审计。
- setup JWT 不能访问普通业务 API、管理员 API 或服务间 API。

## 10. 管理 API

```http
POST   /internal/v1/sessions
DELETE /internal/v1/sessions/current
POST   /internal/v1/token-refreshes
GET    /internal/v1/users/me
PUT    /internal/v1/users/me/password

GET    /internal/v1/admin/users
POST   /internal/v1/admin/users
GET    /internal/v1/admin/users/{user_id}
PATCH  /internal/v1/admin/users/{user_id}
DELETE /internal/v1/admin/users/{user_id}
PUT    /internal/v1/admin/users/{user_id}/password
DELETE /internal/v1/admin/users/{user_id}/lock
```

管理员 API 必须先由 Auth Service 鉴别身份，再由 Permission Service 校验管理员权限。

用户 API 只负责账号生命周期。涉及角色变更时，Auth Service 必须调用 Permission Service 的角色绑定接口，不得直接写角色表。

`POST /internal/v1/token-refreshes` 必须校验 refresh token 的 `jti` 状态。建议采用 refresh token rotation，签发新 refresh token 后将旧 refresh token 标记为 used 或 revoked。

## 11. 服务间认证

内部服务调用建议使用：

- mTLS，后续生产增强。
- service JWT，最小阶段可用。
- API Key，仅限内部系统调用。

服务 token 不映射为普通用户，必须带明确 `service_name`、`scopes`、`expires_at`。

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `AUTH_INVALID_CREDENTIALS` | 用户名或密码错误 |
| `AUTH_USER_DISABLED` | 用户禁用 |
| `AUTH_TOKEN_EXPIRED` | JWT 过期 |
| `AUTH_TOKEN_INVALID` | token 无效 |
| `AUTH_PASSWORD_WEAK` | 密码强度不足 |
| `AUTH_ACCOUNT_LOCKED` | 失败过多被锁定 |
| `AUTH_MFA_REQUIRED` | 后续扩展 MFA |

## 13. 审计事件

- `auth.login.succeeded`
- `auth.login.failed`
- `auth.logout`
- `auth.password.changed`
- `auth.password.reset`
- `auth.user.created`
- `auth.user.disabled`

说明：角色变更审计由 Permission Service 记录，Auth Service 可记录关联的账号操作摘要。

审计日志记录账号、操作者、来源 IP、结果和原因，不记录密码。

## 14. 测试点

- 正确账号密码登录成功。
- 错误密码增加失败次数。
- 达到失败限制后锁定。
- disabled 用户不能登录。
- JWT 过期不能访问业务 API。
- 管理员创建账号必须触发审计。
- 普通用户不能访问管理员账号 API。
- 创建普通用户后默认绑定 `employee` 角色。
- Auth Service 不直接写角色绑定表。
