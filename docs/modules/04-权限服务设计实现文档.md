# Permission Service 设计实现文档

## 1. 模块目标

Permission Service 是企业内部 RAG 的安全核心，负责把用户身份、组织关系、角色和文档可见性转换为统一权限上下文和索引过滤条件，保证用户只能检索自己有权访问的文档和 chunk。

## 2. 模块职责

- 构建权限上下文。
- 维护角色、权限、资源策略。
- 管理角色定义、角色绑定、角色解绑和角色分配审计。
- 生成向量检索和关键词检索可复用的过滤条件。
- 处理知识库、文件夹、文档、chunk 权限变更。
- 管理权限版本和权限缓存。
- 写入权限审计事件。

## 3. 非职责

- 不负责用户登录。
- 不负责组织数据源同步。
- 不调用 LLM 判断权限。
- 不在召回后才过滤权限。

## 5. 权限模型

采用收敛后的组合模型：

- RBAC：角色权限，例如普通员工、部门管理员、知识库管理员、安全审计员，用于管理操作授权。
- 组织可见性：企业、部门、成员三层；文档只支持部门可见和企业公开。
- 文档可见性不再使用自定义组织 ID 列表、上级部门、子部门递归、项目组共享或按用户 allow/deny 的复杂规则。

管理员不是单人模型。初始化阶段只创建第一个 `system_admin` 账号，后续可以存在多个系统管理员。角色定义和角色分配都由 Permission Service 管理，Auth Service 只负责账号身份。

内置默认角色：

| 角色 | 作用域 | 说明 |
| --- | --- | --- |
| `system_admin` | enterprise | 系统管理员，可管理全局配置、用户、角色、权限和恢复初始化 |
| `security_admin` | enterprise | 安全管理员，可管理高风险权限策略和审批 |
| `audit_admin` | enterprise | 审计管理员，可查看审计、评测和高风险操作记录 |
| `department_admin` | department | 部门管理员，只能在授权部门范围内管理成员和部门级授权 |
| `knowledge_base_admin` | knowledge_base | 知识库管理员，只能在授权知识库范围内管理文档、导入和知识库权限 |
| `employee` | enterprise | 普通员工默认角色，普通用户创建或同步时默认绑定 |

角色创建和分配规则：

- 内置默认角色由 Setup Service 初始化创建。
- 自定义角色、部门角色模板和知识库角色模板由 `system_admin` 或具备 `role:manage` scope 的安全管理员创建。
- `department_admin` 不能创建全局角色，只能在自己管理的部门范围内分配部门级角色。
- `knowledge_base_admin` 不能创建全局角色，只能在授权知识库范围内分配知识库级权限。
- 普通用户账号由 Auth Service 创建或外部身份源同步，默认 `employee` 角色由 Permission Service 绑定。
- 任何角色绑定变更都必须递增 `permission_version`，失效权限缓存并记录审计。

核心表建议：

`roles`：

| 字段 | 说明 |
| --- | --- |
| id | 角色 ID |
| enterprise_id | 企业 ID |
| code | 角色编码，例如 system_admin、employee |
| name | 角色名称 |
| scope_type | enterprise、department、knowledge_base |
| is_builtin | 是否内置角色 |
| status | active、disabled、archived |
| created_by / created_at | 创建信息 |

`role_bindings`：

| 字段 | 说明 |
| --- | --- |
| id | 绑定 ID |
| enterprise_id | 企业 ID |
| role_id | 角色 ID |
| subject_type | user、department |
| subject_id | 用户或部门 ID |
| scope_type | enterprise、department、knowledge_base |
| scope_id | 作用域 ID |
| valid_from / valid_to | 有效期 |
| created_by / created_at | 创建信息 |

## 5.1 文档可见性权限

文档权限只保留两种可见性：

| visibility | 说明 |
| --- | --- |
| `department` | 部门可见。仅 `owner_department_id` 对应部门成员可检索，是成员上传文档的默认值 |
| `enterprise` | 企业公开。当前企业内所有部门成员可检索 |

默认语义：

- 文档必须有 `owner_department_id`，默认取上传成员的主部门。
- 成员上传文档默认 `visibility = department`。
- 用户界面只展示“部门可见”和“企业公开”，不展示上级部门、子部门、递归共享、自定义组织 ID、指定用户或项目组共享。
- 查询链路不得递归组织关系，也不得先全量召回后遍历数据库筛文档；必须将 `visibility` 和 `owner_department_id` 下推到向量索引和关键词索引过滤。
- 企业公开属于扩大可见范围，必须记录审计；如业务需要，可通过配置要求管理员审批。

示例：

```text
企业
  研发部
  人事部
```

研发部成员上传文档时：

- 默认 `visibility=department`：只有研发部成员可查。
- 选择 `visibility=enterprise`：企业内所有部门成员可查。

权限快照：

```json
{
  "owner_department_id": "dept_rnd",
  "visibility": "department",
  "permission_version": 42
}
```

安全约束：

- 用户必须属于文档 `owner_department_id` 对应部门，或文档为 `visibility=enterprise`，才可访问。
- 部门管理员只能管理自己部门内的文档可见性；将文档改为企业公开需要 `system_admin`、`department_admin` 或显式授权。
- 权限收紧，例如 `enterprise -> department`，必须递增 `permission_version`，刷新索引 payload，并失效相关查询缓存。

## 6. 权限上下文

示例：

```json
{
  "enterprise_id": "ent_001",
  "user_id": "user_123",
  "departments": ["dept_rnd"],
  "roles": ["employee"],
  "scopes": ["rag:query", "document:read"],
  "permission_version": 42,
  "request_id": "req_001"
}
```

构建流程：

```text
1. Auth Service 校验用户身份
2. Org Service 返回用户所属部门
3. 读取用户角色和资源授权
4. 计算 scopes
5. 读取 permission_version
6. 生成 permission_filter_hash
7. 写入 Redis 缓存
8. 返回 PermissionContext
```

## 7. 索引权限字段

每个 chunk 写入索引时必须包含：

```json
{
  "enterprise_id": "ent_001",
  "kb_id": "kb_sales",
  "doc_id": "doc_001",
  "chunk_id": "chunk_009",
  "owner_department_id": "dept_sales",
  "visibility": "department",
  "permission_snapshot_id": "ps_001",
  "permission_version": 42,
  "resource_policy_version": 8,
  "is_deleted": false
}
```

## 8. 过滤条件生成

逻辑语义：

```text
enterprise_id = current_enterprise
AND kb_id IN allowed_kbs
AND is_deleted = false
AND visibility_state = 'active'
AND index_version_id IN active_index_versions
AND document_status = 'active'
AND document_index_status = 'indexed'
AND chunk_status = 'active'
AND (
  visibility = 'enterprise'
  OR (visibility = 'department' AND owner_department_id IN user.departments)
)
AND access_block NOT EXISTS
```

输出应同时支持：

- Qdrant payload filter。
- PostgreSQL Full Text where 条件。
- 元数据召回查询条件。

推荐接口：

```python
class PermissionService:
    def build_context(user_id: str, request_id: str) -> PermissionContext:
        ...

    def build_filter(
        context: PermissionContext,
        kb_ids: list[str],
        resource_action: str,
        active_index_versions: list[str]
    ) -> PermissionFilter:
        ...
```

## 9. 权限变更流程

```text
1. 管理员提交权限变更
2. Auth Service 鉴别操作者
3. Permission Service 校验操作者管理权限
4. 写 resource_policy 新版本
5. 增加 permission_version
6. 失效 Redis 权限缓存
7. 对权限收紧资源先写 access block 或阻断旧 chunk_index_refs
8. 创建权限索引刷新任务
9. Worker 更新 Qdrant payload 和关键词索引权限字段
10. 刷新成功后解除必要的临时阻断
11. 写审计事件
```

要求：

- 权限索引刷新失败必须告警。
- 查询发现索引权限版本落后必须拒绝候选或回源确认，不能直接放行。
- 权限收紧期间必须清理或失效相关查询缓存，避免旧答案复用。
- 文档 `visibility` 或 `owner_department_id` 变更必须创建新的 resource_policy 版本、permission_snapshot，并刷新索引 payload。

## 10. 缓存策略

缓存 key：

```text
permission_context:{enterprise_id}:{user_id}:{permission_version}:{org_version}
```

失效条件：

- 用户角色变更。
- 用户部门变更。
- 资源授权策略变更。
- 用户状态变为 disabled/left。

## 11. API

```http
PUT /internal/v1/knowledge-bases/{kb_id}/permissions
PUT /internal/v1/documents/{doc_id}/permissions
GET /internal/v1/permissions/effective?resource_id=doc_001&user_id=user_123

POST /internal/v1/admin/roles
GET  /internal/v1/admin/roles
PATCH /internal/v1/admin/roles/{role_id}
POST /internal/v1/admin/roles/{role_id}:disable

PUT  /internal/v1/admin/users/{user_id}/roles
POST /internal/v1/admin/users/{user_id}/roles:grant
POST /internal/v1/admin/users/{user_id}/roles:revoke
```

角色 API 要求：

- 创建或修改 enterprise 级角色必须具备 `role:manage`。
- 分配 `system_admin`、`security_admin` 等高风险角色必须触发高风险审计，生产阶段建议审批或双人确认。
- 部门管理员只能在自己管理的部门 `scope_id` 内分配部门级角色。
- 普通用户默认 `employee` 角色不得被删除；如需禁用用户，应由 Auth Service 修改用户状态。

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `PERM_DENIED` | 无权限访问资源 |
| `PERM_SCOPE_MISSING` | 缺少操作 scope |
| `PERM_POLICY_INVALID` | 权限策略非法 |
| `PERM_VERSION_STALE` | 权限版本过期 |
| `PERM_INDEX_REFRESH_FAILED` | 权限索引刷新失败 |
| `PERM_ACCESS_BLOCKED` | 资源处于删除、权限收紧或安全阻断状态 |
| `PERM_ROLE_SCOPE_DENIED` | 操作者无权在目标作用域创建或分配角色 |
| `PERM_ROLE_BUILTIN_PROTECTED` | 内置角色不允许删除或非法修改 |
| `PERM_VISIBILITY_SCOPE_DENIED` | 操作者无权将文档设为目标可见性 |
| `PERM_VISIBILITY_INVALID` | 文档可见性非法 |

## 13. 审计事件

- `permission.context.built`
- `permission.query.filtered`
- `permission.resource.updated`
- `permission.role.created`
- `permission.role.updated`
- `permission.role.granted`
- `permission.role.revoked`
- `permission.visibility.updated`
- `permission.cache.invalidated`
- `permission.index.refresh.started`
- `permission.index.refresh.failed`

审计需要记录被过滤候选数量，但不记录敏感原文。

## 14. 测试点

- 向量召回和关键词召回使用同一权限语义。
- 权限变更后缓存失效。
- 权限收紧后立即阻断旧访问。
- 索引权限版本落后时不能直接放行。
- 权限收紧后旧查询缓存不能复用。
- 初始化后存在内置默认角色。
- 创建普通用户时默认绑定 `employee`。
- 部门管理员不能越权分配其他部门角色。
- Auth Service 不直接写角色绑定表。
- 成员上传文档默认仅 owner_department 对应部门可见。
- `visibility=enterprise` 后企业内所有部门成员可见。
- 文档权限设置不支持上级部门、子部门递归、自定义组织 ID 或指定用户共享。
- 查询过滤使用 `visibility` 和 `owner_department_id` 下推到索引层，不在查询时递归组织关系或遍历数据库筛文档。
- 部门管理员不能越权修改其他部门文档可见性。
