# Permission Service 设计实现文档

## 1. 模块目标

Permission Service 是企业内部 RAG 的安全核心，负责把用户身份、组织关系、角色、资源 ACL 和数据密级转换为统一权限上下文和索引过滤条件，保证用户只能检索自己有权访问的文档和 chunk。

## 2. 模块职责

- 构建权限上下文。
- 维护角色、权限、资源策略。
- 生成向量检索和关键词检索可复用的过滤条件。
- 处理知识库、文件夹、文档、chunk 权限变更。
- 管理权限版本和权限缓存。
- 写入权限审计事件。

## 3. 非职责

- 不负责用户登录。
- 不负责组织数据源同步。
- 不调用 LLM 判断权限。
- 不在召回后才过滤权限。

## 5. 权限模型

采用组合模型：

- RBAC：角色权限，例如普通员工、部门管理员、知识库管理员、安全审计员。
- ABAC：部门、岗位、职级、地区、项目、密级。
- ACL：资源级 allow/deny。
- 数据密级：公开、内部、部门内、项目内、机密、绝密。

## 6. 权限上下文

示例：

```json
{
  "enterprise_id": "ent_001",
  "user_id": "user_123",
  "departments": ["dept_rnd_1", "dept_arch"],
  "roles": ["employee", "project_reviewer"],
  "groups": ["project_alpha", "temp_ipo_review"],
  "clearance_level": 3,
  "scopes": ["rag:query", "document:read"],
  "permission_version": 42,
  "request_id": "req_001"
}
```

构建流程：

```text
1. Auth Service 校验用户身份
2. Org Service 返回用户部门和组
3. 读取用户角色和资源授权
4. 计算 scopes
5. 读取 permission_version 和 strict permission mode
6. 生成 permission_filter_hash
7. 写入 Redis 缓存
8. 返回 PermissionContext
```

## 7. 索引权限字段

每个 chunk 写入索引时必须包含：

```json
{
  "enterprise_id": "ent_001",
  "kb_id": "kb_sales",
  "doc_id": "doc_001",
  "chunk_id": "chunk_009",
  "owner_department_id": "dept_sales",
  "allowed_departments": ["dept_sales", "dept_mgmt"],
  "allowed_groups": ["project_alpha"],
  "allowed_users": [],
  "denied_users": [],
  "visibility": "department",
  "security_level": 2,
  "permission_snapshot_id": "ps_001",
  "permission_version": 42,
  "resource_policy_version": 8,
  "strict_mode": true,
  "is_deleted": false
}
```

## 8. 过滤条件生成

逻辑语义：

```text
enterprise_id = current_enterprise
AND kb_id IN allowed_kbs
AND is_deleted = false
AND visibility_state = 'active'
AND index_version_id IN active_index_versions
AND document_status = 'active'
AND document_index_status = 'indexed'
AND chunk_status = 'active'
AND security_level <= user.clearance_level
AND (
  visibility = 'enterprise'
  OR owner_department_id IN user.departments
  OR allowed_departments INTERSECTS user.departments
  OR allowed_groups INTERSECTS user.groups
  OR allowed_users CONTAINS user.user_id
)
AND denied_users NOT CONTAINS user.user_id
AND access_block NOT EXISTS
```

输出应同时支持：

- Qdrant payload filter。
- PostgreSQL Full Text where 条件。
- 元数据召回查询条件。

推荐接口：

```python
class PermissionService:
    def build_context(user_id: str, request_id: str) -> PermissionContext:
        ...

    def build_filter(
        context: PermissionContext,
        kb_ids: list[str],
        resource_action: str,
        active_index_versions: list[str]
    ) -> PermissionFilter:
        ...
```

## 9. 权限变更流程

```text
1. 管理员提交权限变更
2. Auth Service 鉴别操作者
3. Permission Service 校验操作者管理权限
4. 写 resource_policy 新版本
5. 增加 permission_version
6. 失效 Redis 权限缓存
7. 对高密级、strict mode 或权限收紧资源先写 access block 或阻断旧 chunk_index_refs
8. 创建权限索引刷新任务
9. Worker 更新 Qdrant payload 和关键词索引权限字段
10. 刷新成功后解除必要的临时阻断
11. 写审计事件
```

要求：

- 高密级文档权限变更必须先阻断旧权限访问。
- 权限索引刷新失败必须告警。
- strict permission mode 下，查询发现索引权限版本落后必须拒绝候选或回源确认，不能直接放行。
- 权限收紧期间必须清理或失效相关查询缓存，避免旧答案复用。

## 10. 缓存策略

缓存 key：

```text
permission_context:{enterprise_id}:{user_id}:{permission_version}:{org_version}
```

失效条件：

- 用户角色变更。
- 用户部门或组变更。
- 资源授权策略变更。
- 用户密级变更。
- 用户状态变为 disabled/left。

## 11. API

```http
PUT /internal/v1/knowledge-bases/{kb_id}/permissions
PUT /internal/v1/documents/{doc_id}/permissions
GET /internal/v1/permissions/effective?resource_id=doc_001&user_id=user_123
```

## 12. 错误码

| 错误码 | 场景 |
| --- | --- |
| `PERM_DENIED` | 无权限访问资源 |
| `PERM_SCOPE_MISSING` | 缺少操作 scope |
| `PERM_POLICY_INVALID` | 权限策略非法 |
| `PERM_VERSION_STALE` | 权限版本过期 |
| `PERM_INDEX_REFRESH_FAILED` | 权限索引刷新失败 |
| `PERM_ACCESS_BLOCKED` | 资源处于删除、权限收紧或安全阻断状态 |

## 13. 审计事件

- `permission.context.built`
- `permission.query.filtered`
- `permission.resource.updated`
- `permission.cache.invalidated`
- `permission.index.refresh.started`
- `permission.index.refresh.failed`

审计需要记录被过滤候选数量，但不记录敏感原文。

## 14. 测试点

- 向量召回和关键词召回使用同一权限语义。
- denied_users 优先级高于 allowed_users。
- 用户密级不足时不能检索高密级文档。
- 部门继承策略正确。
- 权限变更后缓存失效。
- 高密级权限收紧后立即阻断旧访问。
- strict permission mode 下索引权限版本落后时不能直接放行。
- 权限收紧后旧查询缓存不能复用。
