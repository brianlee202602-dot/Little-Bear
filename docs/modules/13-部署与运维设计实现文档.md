# 部署与运维设计实现文档

## 1. 模块目标

本文定义最小生产阶段的 Docker Compose 部署、配置边界、数据持久化、备份恢复、容量规划、上线检查和后续演进方式。

## 2. 最小生产部署

服务清单：

```text
api
  - FastAPI
  - setup API、查询 API、导入 API、权限 API、配置 API

import-worker
  - 轮询 PostgreSQL import_jobs
  - 执行 validate、parse、clean、chunk、embed、index、publish

scheduled-worker
  - 定时任务
  - 组织同步、失败任务扫描、索引健康检查

postgres
  - 用户、组织、权限、配置、任务、审计、PostgreSQL Full Text

redis
  - 缓存、锁、限流、配置通知

minio
  - 原始文件和中间产物

qdrant
  - 向量检索

model-gateway
  - 统一模型调用入口

tei-embedding
  - embedding 模型服务

tei-rerank
  - rerank 模型服务

vllm-llm
  - LLM 推理服务
```

## 3. 配置边界

API `.env` 只保存启动配置：

```text
APP_ENV=prod
SERVICE_NAME=rag-api
SERVICE_PORT=8080
LOG_LEVEL=INFO
DATABASE_URL=postgresql://...
REDIS_URL=redis://redis:6379/0
SECRET_PROVIDER=docker_secret
CONFIG_ENCRYPTION_KEY_REF=kms/rag-config-key
KMS_KEY_ID=rag-prod-key
```

不放入 API `.env`：

- MinIO endpoint。
- Qdrant URL。
- vLLM URL。
- TEI URL。
- 检索参数。
- 导入参数。
- 权限策略。
- 安全策略。
- 审计策略。

这些由初始化接口写入 active config。

## 4. 持久化 volume

必须持久化：

- PostgreSQL data。
- Redis data，视限流和锁要求可选持久化，但生产建议开启。
- MinIO data。
- Qdrant storage。
- 模型缓存目录，按模型部署需要。

## 5. Healthcheck

API：

- `/health/live`：进程存活。
- `/health/ready`：数据库、Redis、初始化状态、active config、核心数据模型迁移版本。

Worker：

- 能连接数据库。
- 能领取或扫描任务。
- 当前队列积压未超过阈值。

模型服务：

- Model Gateway health。
- TEI embedding health。
- TEI rerank health。
- vLLM health。

依赖：

- PostgreSQL ready。
- Redis ping。
- MinIO bucket。
- Qdrant collections。
- active index version 和 draft index 隔离检查。

## 6. 备份与恢复

PostgreSQL：

- 每日全量备份。
- WAL 或增量备份，按 RPO 要求。
- 定期恢复演练。

MinIO：

- bucket 版本化可选。
- 原始文件和中间产物按生命周期管理。
- 关键 bucket 定期备份。

Qdrant：

- 快照。
- 索引可从 PostgreSQL + MinIO 重建，但重建成本高，生产建议备份。

配置：

- active config 和 config_versions 在 PostgreSQL。
- Secret Provider 需要单独备份或有恢复方案。

## 7. 容量估算

粗略估算：

```text
文档数: 100,000
平均 chunk: 10
总 chunk: 1,000,000
embedding dimension: 1024
float32 向量大小: 4KB/chunk
纯向量约: 4GB
加索引和 payload: 约 8GB 到 16GB
```

需要预留：

- PostgreSQL 元数据和审计增长。
- MinIO 原始文件容量。
- Qdrant 索引膨胀。
- 模型显存。
- Worker 临时文件空间。

## 8. 资源隔离

建议：

- API 和 Worker 分开进程。
- 查询 API 和导入 Worker 使用不同数据库连接池。
- Embedding 在线池和 batch 池分离。
- LLM GPU 服务和普通业务服务分离节点。
- 导入任务限速，避免影响查询。

## 9. 上线流程

```text
1. 部署基础组件 PostgreSQL、Redis、MinIO、Qdrant
2. 部署模型服务和 Model Gateway
3. 部署 API 和 Worker
4. API 读取最小环境变量启动
5. 获取临时 system_token
6. 调用 setup validate
7. 调用 setup initialize
8. 创建首个管理员和 active config v1
9. 登录管理员后台
10. 创建普通用户、部门、知识库
11. 导入测试文档
12. 执行带权限查询
13. 检查审计、日志、指标
```

## 10. 上线检查清单

- API `.env` 不包含业务依赖地址和密钥明文。
- system token 不出现在普通日志。
- 初始化完成后 setup initialize 关闭。
- active config v1 存在且可回滚。
- PostgreSQL、MinIO、Qdrant 已持久化。
- 所有服务有 healthcheck。
- 查询、导入、权限变更有审计。
- 权限过滤下推到 Qdrant 和关键词索引。
- 查询过滤包含 active index version、`visibility_state=active`、`is_deleted=false` 和 access block gate。
- 查询结果缓存 key 包含用户、租户、权限过滤、请求过滤、配置版本、索引版本、模型路由和 Prompt 版本。
- 高密级和 strict permission mode 查询不缓存最终答案。
- LLM 不可用时可以降级。
- 导入失败不会发布未完成或未校验的索引。
- draft 索引和已删除文档不可被查询命中。
- 备份任务已配置并演练。

## 11. 运维命令建议

```text
rag-admin setup-token show
rag-admin setup-token rotate
rag-admin config active
rag-admin config validate <file>
rag-admin import retry <job_id>
rag-admin index check <kb_id>
rag-admin permission refresh <resource_id>
rag-admin health check
```

命令输出不得包含 secret value。

## 12. 演进路线

最小生产：

- Docker Compose。
- 单 API 实例。
- 单 Qdrant。
- PostgreSQL Full Text。
- 结构化日志。

标准生产：

- API 多副本。
- Redis Sentinel/Cluster。
- PostgreSQL 主从。
- Qdrant 集群。
- Prometheus + Grafana + OpenTelemetry。
- Worker 按阶段拆分。

大规模：

- Kubernetes。
- Celery/RabbitMQ 或 Kafka。
- OpenSearch。
- Temporal 可选。
- 模型多池治理。

## 13. 故障处理

| 故障 | 处理 |
| --- | --- |
| API 无法加载 active config | 拒绝业务请求，告警，检查 PostgreSQL 和配置版本 |
| Qdrant 不可用 | 查询降级关键词，导入索引阶段暂停 |
| PostgreSQL 不可用 | API readiness 失败，禁止写入 |
| Redis 不可用 | 权限缓存失效，限流降级为本地保守策略 |
| LLM 不可用 | 返回检索结果和引用 |
| Worker 积压 | 降低导入入口速率，扩 Worker 或模型 batch 池 |
| 审计写入失败 | 高风险操作应失败或进入阻断队列 |
| draft index 被查询命中 | 立即阻断查询入口，回滚相关 index_version，清理缓存并告警 |
| 缓存串权 | 立即关闭最终答案缓存，清理 Redis，审计影响范围并修复 key 生成 |

## 14. 测试点

- 新环境从零初始化成功。
- 重启 API 后能加载 active config。
- 初始化完成后 system token 不可用。
- 备份可恢复。
- Qdrant 故障时查询降级。
- Worker 崩溃后任务可恢复。
- draft 索引、已删除文档和 access block 资源不可被查询命中。
- 查询缓存不会跨用户或跨权限过滤条件复用。
