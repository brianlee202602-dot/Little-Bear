# 公共实现约束

## 1. 目标

本文定义所有功能模块必须遵守的工程约束，避免后续实现时出现跨层调用、配置散落、权限绕过、不可观测和难以替换基础设施的问题。

## 2. 代码分层

推荐结构：

```text
app/
  api/
    routes/
    dependencies/
    schemas/
  modules/
    <module_name>/
      service.py
      models.py
      policies.py
      repository.py
      schemas.py
      errors.py
      events.py
  ports/
  adapters/
  shared/
```

每个模块内部建议：

- `service.py`：应用服务，承载用例编排。
- `policies.py`：纯业务规则，尽量无外部 IO。
- `repository.py`：模块内部数据库访问封装。
- `schemas.py`：内部 DTO 和校验 schema。
- `events.py`：领域事件或审计事件定义。
- `errors.py`：模块错误码和异常。

## 3. 代码注释与语言约定

项目代码中的注释、docstring 和面向开发者的说明默认使用中文。

要求：

- 注释应优先解释设计意图、业务约束、权限边界、事务边界、幂等策略、降级策略和安全限制。
- 避免为显而易见的代码行为添加重复注释，例如“给变量赋值”“调用函数”等。
- 涉及外部协议、标准字段、第三方库参数或 OpenAPI / SQL / HTTP 固定术语时，可以保留英文原文。
- 错误码、API 字段名、数据库字段名、配置 key、scope、枚举值和日志字段保持英文或契约中定义的原始名称。
- Alembic migration、ORM model、service、policy、repository、worker、adapter、前端组件和测试中的新增注释均遵守本约定。
- 注释不得包含 secret、token、密码、完整 prompt、文档原文或未脱敏个人信息。

代码评审时需要检查注释是否使用中文、是否解释关键约束，以及是否引入无效、过期或误导性描述。

## 4. 依赖方向

允许：

```text
api -> module service
module service -> repository
module service -> ports
ports -> adapters
module service -> audit/event sink
```

禁止：

```text
api -> database session raw SQL
api -> qdrant/minio/model sdk
module -> another module database table without service/repository boundary
worker -> FastAPI route function
permission -> LLM
LLM -> permission decision
```

## 5. 配置读取

所有后端进程的启动层只允许读取数据库连接配置和进程启动参数：

- `DATABASE_URL`
- 数据库连接超时、连接池、SSL mode 等数据库连接参数
- `APP_ENV`、`SERVICE_NAME`、`API_HOST`、`API_PORT`、`LOG_LEVEL` 等不参与业务决策的运行参数

禁止从启动配置读取：

- Redis 连接。
- Secret Provider 连接。
- MinIO、Qdrant、embedding/rerank/LLM provider 地址。
- 检索、权限、缓存、限流、审计等业务策略。

业务模块读取配置必须走 `ConfigService`：

```text
service -> ConfigService.get_effective_config(scope) -> active config cache/db
```

实现要求：

- 数据库连接配置缺失或 PostgreSQL 连接失败时，进程启动失败或 readiness=false，不进入 setup mode。
- Redis、Secret Provider、ObjectStorage、VectorStore、KeywordSearch、模型 Provider Adapter、Audit Sink、Rate Limiter 等服务模块必须在 active config 加载后由 ServiceBootstrap 初始化。
- active config 加载成功且关键模块初始化成功前，业务 API 不得开放。
- 每次查询和导入任务记录 `config_version`。
- 配置读取失败时使用本地最近一次 active config 只适用于已成功启动后的运行期热更新失败；冷启动时数据库不可用必须失败。
- 如果系统未初始化，返回 `SETUP_REQUIRED`。
- 如果系统已初始化但没有 active config，进入拒绝模式并告警。

## 6. Secret 处理

Secret Provider / Secret Store 是密钥管理抽象，用于保存和读取 MinIO access key、模型服务 API key、外部系统 token、服务间调用密钥等敏感值。它不是启动依赖，必须由数据库 active config 指定并在 ServiceBootstrap 阶段初始化。

最小生产阶段可以使用 PostgreSQL 加密 `secrets` 表作为 Secret Store；后续可通过 adapter 替换为 Vault、云 KMS、Kubernetes Secret、Docker Secret 或企业内部密钥管理服务。

- 日志禁止输出 secret value。
- active config 优先保存 Secret 引用，例如 `secret://rag/minio/access-key`。
- 必须保存敏感值时，先写入由 active config 指定的 Secret Provider，或使用数据库/应用侧受控加密方案加密；加密方案本身不得要求额外启动配置。
- API 响应不得返回密钥明文。
- 审计日志只记录 Secret key、引用路径、变更摘要和操作者。

## 7. 事务边界

事务应由应用服务开启，不在 API route 或 adapter 内散落开启。

建议：

- 一个用户请求对应一个主要业务事务。
- 外部 IO 不应长时间占用数据库事务。
- 初始化、配置发布、权限变更必须具备明确原子边界。
- 导入 Worker 每个阶段独立提交状态，便于失败重试。
- 索引写入和元数据写入需要补偿逻辑，不能假设分布式事务。

## 8. 幂等

以下操作必须幂等：

- 初始化配置 validate。
- 导入任务创建，基于 `idempotency_key`。
- Worker 阶段执行。
- 索引写入，基于确定性 `vector_id`、`keyword_id`。
- 权限索引刷新。
- 配置变更通知消费。

幂等实现建议：

- 使用唯一约束。
- 使用状态机防止重复推进。
- 使用内容哈希识别重复文档或 chunk。
- Worker 更新任务时检查 `locked_by` 和 `locked_until`。

## 9. 查询可见性与缓存安全

查询候选可进入上下文前必须同时满足：

- 权限过滤条件已下推到向量检索和关键词检索。
- `document.lifecycle_status = active`。
- `document.index_status = indexed`。
- `chunk.status = active`。
- `index_version.status = active`。
- `chunk_index_ref.visibility_state = active`。
- 不存在有效 `access_block`。
- 索引权限版本不落后，或已完成回源确认。

缓存要求：

- 查询结果缓存默认按 `enterprise_id + user_id` 隔离。
- 如需跨用户复用，只能复用检索中间结果，且缓存 key 必须包含 `permission_filter_hash`。
- 最终答案缓存 key 至少包含 `enterprise_id`、`user_id`、`query_hash`、`kb_ids_hash`、`request_filter_hash`、`permission_filter_hash`、`config_version`、`index_version_hash`、`model_route_hash` 和 `prompt_template_version`。
- 权限刷新中、引用校验失败或降级不完整的响应不得缓存。
- 缓存命中后仍需做轻量 access block 校验和引用有效性检查。

## 10. 错误码

错误响应统一结构：

```json
{
  "request_id": "req_001",
  "error_code": "SETUP_REQUIRED",
  "message": "system is not initialized",
  "stage": "setup_guard",
  "retryable": false,
  "details": {}
}
```

错误码分组：

| 前缀 | 模块 |
| --- | --- |
| `SETUP_` | 初始化 |
| `AUTH_` | 认证 |
| `PERM_` | 权限 |
| `CONFIG_` | 配置 |
| `IMPORT_` | 导入 |
| `INDEX_` | 索引 |
| `QUERY_` | 查询 |
| `MODEL_` | 模型 |
| `RAG_` | 检索生成 |
| `AUDIT_` | 审计 |

## 11. JWT 鉴权约束

所有用户、服务间和初始化 token 统一使用 JWT，并通过 `Authorization: Bearer <jwt>` 传递。

JWT 类型：

- `access`：普通用户业务 API 访问。
- `refresh`：刷新 access token。
- `service`：服务间调用。
- `setup`：初始化和恢复初始化，只能访问 setup-config-validations / setup-initialization。

要求：

- JWT 必须包含 `jti`、`token_type`、`iat`、`exp`，并按类型包含 `sub`、`enterprise_id`、`service_name` 或 setup 状态。
- 服务端必须保存 `jti` 状态，用于吊销、一次性使用、恢复初始化 token 失效和审计。
- JWT 中不得放完整权限上下文，权限上下文必须由 Permission Service 构建。
- setup JWT 不能访问普通业务 API、管理员 API 或服务间 API。

## 12. 日志与 Trace

所有结构化日志必须包含：

- `timestamp`
- `level`
- `service`
- `module`
- `request_id`
- `trace_id`
- `user_id`，如有
- `enterprise_id`，如有
- `job_id`，如有
- `stage`
- `latency_ms`
- `status`
- `error_code`，如有

文档原文、完整 prompt、密钥、token、密码和未脱敏个人信息禁止进入普通日志。

Prompt 相关日志只记录 `prompt_template_id`、`prompt_template_version`、`prompt_hash`、变量摘要、模型版本和输出摘要；确需保存完整 prompt 时必须走受控审计存储、脱敏、权限隔离和保留期配置。

## 13. 测试分层

每个模块至少覆盖：

- policy 单元测试。
- service 用例测试。
- repository 数据库测试。
- adapter 集成测试。
- API contract 测试。
- 权限绕过测试。
- 降级和超时测试。

关键链路必须有端到端测试：

- 首次初始化。
- 本地账号登录。
- 文档导入到可检索。
- 权限变更后不可越权查询。
- draft 索引和已删除文档不可被查询命中。
- 查询缓存不会跨用户、跨租户、跨权限过滤条件复用。
- 权限版本落后时 fail closed 或回源确认。
- 流式响应不会输出未经引用校验的最终答案。
- 查询返回引用。
- 模型服务不可用时降级。

## 14. 代码评审重点

- 是否绕过 Config Service 读取业务配置。
- 是否绕过 Permission Service 构建权限过滤。
- 是否在 API 层直接访问外部 SDK。
- 新增注释和 docstring 是否默认使用中文，且是否解释了关键约束而不是重复代码表面行为。
- 是否把 secret、prompt、文档原文写入日志。
- 是否缺少幂等键或唯一约束。
- 查询缓存 key 是否缺少用户、权限过滤、索引版本或请求过滤条件。
- 查询候选是否绕过 active index、access block 或引用有效性准入校验。
- 是否缺少超时、重试、熔断和降级。
- 是否缺少审计事件。
