# 安装部署操作手册

## 1. 文档状态

本文用于指导 Little Bear 的安装、部署和首次初始化。当前版本先覆盖本地或单机内网部署下的初始化闭环，后续在 Auth、Worker、导入、索引、查询和运维能力完成后，继续补充完整生产部署、升级、备份恢复和回滚步骤。

当前初始化目标：

- 安装后端、前端和数据库迁移所需依赖。
- 启动 PostgreSQL、Redis、MinIO、Qdrant、TEI embedding 和 TEI rerank。
- 执行 Alembic 数据库迁移。
- 写入基础设施初始化所需 Secret。
- 启动 API 和管理后台初始化页面。
- 使用 setup JWT 完成首次初始化，按需在页面输入外部模型访问密钥，并发布 `active_config v1`。

## 2. 部署模式

当前手册默认采用“应用进程运行在宿主机，基础设施运行在 Docker Compose”的本地或内网单机模式：

| 组件 | 默认运行方式 |
| --- | --- |
| API Service | 宿主机 Python 进程，`make api` 启动 |
| Admin 前端 | 宿主机 Vite 进程，`make admin` 启动 |
| PostgreSQL | Docker Compose |
| Redis | Docker Compose |
| MinIO | Docker Compose |
| Qdrant | Docker Compose |
| TEI embedding / rerank | Docker Compose，演示用途，可替换为外部服务 |
| LLM Provider | 外部 OpenAI-compatible 服务，当前 compose 不内置 |

如果后续改为 API 也运行在 Docker Compose 网络内，初始化页面中的依赖地址可以使用 compose service name，例如 `redis://redis:6379/0`。如果 API 运行在宿主机，则必须填写 API 进程可访问的主机地址，例如 `localhost` 或内网服务器 IP。

## 3. 前置条件

### 3.1 系统依赖

必须安装：

- Git。
- Docker Engine 或 Docker Desktop。
- Docker Compose v2。
- Python 3.12 或更高版本。
- Node.js 20 或更高版本。
- `make`。
- `curl`。
- `openssl`。

Ubuntu 24.04 示例：

```bash
sudo apt update
sudo apt install -y git make curl openssl python3 python3-venv python3-pip
docker --version
docker compose version
python3 --version
```

Node.js 建议使用 nvm、NodeSource 或企业内部软件源安装 20 或更高版本。安装后确认：

```bash
node --version
npm --version
```

macOS Homebrew 示例：

```bash
brew install python node make openssl
docker --version
docker compose version
python3 --version
node --version
npm --version
```

Docker 的安装方式和权限配置与操作系统有关。执行后续命令前，当前用户必须能够正常运行 `docker compose ps`。

### 3.2 获取代码

```bash
git clone <repository-url> Little-Bear
cd Little-Bear
```

如果已经在仓库根目录，可以跳过本节。

## 4. 安装项目依赖

### 4.1 Python 依赖

从仓库根目录创建并启用虚拟环境：

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -e ".[dev]"
```

确认 Alembic、FastAPI 和项目包可以被当前解释器加载：

```bash
.venv/bin/python -m alembic.config --version
PYTHONPATH=apps/api .venv/bin/python -c "import app; print('api package ok')"
```

### 4.2 Node.js 依赖

仓库使用 npm workspace。首次启动前需要在仓库根目录安装前端依赖：

```bash
npm ci
```

确认管理后台脚本存在：

```bash
npm --workspace @little-bear/admin run typecheck
```

如果只是先跑初始化页面，可以暂时跳过 typecheck；但正式交付前必须执行。

## 5. 创建环境变量文件

生成 `.env`：

```bash
make env
```

至少需要确认以下变量存在：

```dotenv
DATABASE_URL=postgresql+psycopg://little_bear:little_bear_dev@localhost:5432/little_bear
API_HOST=0.0.0.0
API_PORT=8000
ADMIN_SETUP_URL=http://localhost:5174/admin/setup-initialization
SETUP_TOKEN_LOG_ENABLED=true
SETUP_TOKEN_SIGNING_SECRET=
SECRET_STORE_MASTER_KEY=
SECRET_INIT_MINIO_ACCESS_KEY=minioadmin
SECRET_INIT_MINIO_SECRET_KEY=minioadmin
SECRET_INIT_JWT_SIGNING_KEY=
```

生成并填入本地随机密钥：

```bash
openssl rand -base64 64
```

建议分别生成并填入：

- `SETUP_TOKEN_SIGNING_SECRET`：用于签发和校验初始化 setup JWT。
- `SECRET_STORE_MASTER_KEY`：用于加密 PostgreSQL `secrets` 表中的密钥值，初始化后仍必须保留。
- `SECRET_INIT_JWT_SIGNING_KEY`：业务登录 JWT 的签名密钥，写入 secrets 表后应从 `.env` 删除。

MinIO 初始化临时变量必须与 `docker-compose.yml` 中的 MinIO 容器配置一致：

| `.env` 变量 | 必须匹配的 compose 变量 |
| --- | --- |
| `SECRET_INIT_MINIO_ACCESS_KEY` | `MINIO_ROOT_USER` |
| `SECRET_INIT_MINIO_SECRET_KEY` | `MINIO_ROOT_PASSWORD` |

当前默认值均为 `minioadmin`。

安全约束：

- 不要提交 `.env`。
- 不要把真实 secret value 放入 active config。
- 初始化成功并写入 secrets 表后，删除 `.env` 中的 `SECRET_INIT_*` 明文变量。
- 不要删除或丢失 `SECRET_STORE_MASTER_KEY`，否则已写入的 secrets 表密文无法解密。

## 6. 启动基础设施

首次启动：

```bash
make up
make ps
```

查看日志：

```bash
make logs
```

确认关键服务可用：

```bash
docker compose --env-file .env ps
docker compose --env-file .env exec postgres pg_isready -U little_bear -d little_bear
docker compose --env-file .env exec redis redis-cli ping
curl -fsS http://localhost:9000/minio/health/live && echo "MinIO live OK"
curl -fsS http://localhost:6333/readyz || curl -fsS http://localhost:6333/
```

TEI embedding 和 rerank 镜像首次启动可能需要下载模型，耗时较长。可查看：

```bash
docker compose --env-file .env logs -f tei-embedding tei-rerank
```

当前 `docker-compose.yml` 中的 TEI 服务仅用于演示和本地联调。实际部署时可以删除这两个 service，并在初始化页面中填写企业内部模型代理或其他外部 embedding/rerank 服务地址。

## 7. 执行数据库迁移

PostgreSQL healthy 后执行 Alembic 迁移：

```bash
make PYTHON=.venv/bin/python db-upgrade
make PYTHON=.venv/bin/python db-current
```

预期 `db-current` 返回当前代码要求的 Alembic head，例如：

```text
0005_jobs_audit_cache (head)
```

不要跳过迁移。初始化接口依赖 `system_state`、`system_configs`、`config_versions`、`users`、`roles`、`audit_logs`、`setup_tokens` 和 `secrets` 等基础表。

## 8. 写入初始化 Secret

初始化 payload 中的 active config 只保存 Secret ref，不保存 Secret value。正式初始化前必须先写入以下基础 Secret：

| Secret ref | 用途 |
| --- | --- |
| `secret://rag/minio/access-key` | 访问 MinIO 的 access key |
| `secret://rag/minio/secret-key` | 访问 MinIO 的 secret key |
| `secret://rag/auth/jwt-signing-key` | 业务登录 JWT 签名密钥 |
| `secret://rag/qdrant/api-key` | 可选，Qdrant 开启 API Key 鉴权时使用 |

执行写入：

```bash
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets put secret://rag/minio/access-key --value-env SECRET_INIT_MINIO_ACCESS_KEY
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets put secret://rag/minio/secret-key --value-env SECRET_INIT_MINIO_SECRET_KEY
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets put secret://rag/auth/jwt-signing-key --value-env SECRET_INIT_JWT_SIGNING_KEY
```

如果 Qdrant 开启了 API Key 鉴权，可以沿用通用 Secret 写入命令导入标准 Secret ref：

```bash
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets put secret://rag/qdrant/api-key --value-env SECRET_INIT_QDRANT_API_KEY
```

随后在初始化页面的“Qdrant API Key 引用”中填写 `secret://rag/qdrant/api-key`；如果 Qdrant 未开启鉴权，可以留空。

如果外部 embedding、rerank 或 LLM provider 需要 API Key，不需要在本节提前写入。初始化页面提供明文输入框，后端接收后会在初始化事务内加密写入 Secret Store，并把 active config 中的 provider `auth_token_ref` 自动设置为 `secret://rag/model/*-api-key`。

检查 Secret 状态：

```bash
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets list
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets verify secret://rag/minio/access-key
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets verify secret://rag/minio/secret-key
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets verify secret://rag/auth/jwt-signing-key
PYTHONPATH=apps/api .venv/bin/python -m app.cli.secrets verify secret://rag/qdrant/api-key
```

确认写入成功后，从 `.env` 删除：

```dotenv
SECRET_INIT_MINIO_ACCESS_KEY=
SECRET_INIT_MINIO_SECRET_KEY=
SECRET_INIT_JWT_SIGNING_KEY=
```

或者直接移除这三行。保留 `SECRET_STORE_MASTER_KEY`。

## 9. 启动 API Service

另开一个终端，从仓库根目录启动 API：

```bash
make PYTHON=.venv/bin/python api
```

当前实现会在 API 启动时执行数据库连接检查和 setup 状态检查：

- 数据库不可连接：API 启动失败并报错。
- 数据库可连接但迁移缺失：提示先执行 Alembic 迁移。
- 数据库已迁移但未初始化：签发一次性 setup JWT，并在日志中打印初始化页面地址。

日志中应出现类似内容：

```text
setup required: visit http://localhost:5174/admin/setup-initialization
setup JWT token: <setup-jwt>
setup token expires at: <timestamp>
```

复制最新的 `<setup-jwt>`。当前代码尚未提供正式的 `rag-admin setup-token` 命令，初始化测试以 API 启动日志中的 setup JWT 为准。

基础检查：

```bash
curl -fsS http://localhost:8000/internal/v1/setup-state
curl -fsS http://localhost:8000/health/live
```

系统未初始化时，`/health/ready` 不应返回 ready。

## 10. 启动管理后台初始化页面

另开一个终端：

```bash
make admin
```

访问：

```text
http://localhost:5174/admin/setup-initialization
```

将 API 日志中的 setup JWT 填入“初始化访问凭证”。

## 11. 填写初始化配置

### 11.1 管理员与组织

填写首个管理员、默认企业和默认部门。管理员密码必须满足页面提示和后端密码策略，并且确认密码必须一致。

### 11.2 基础设施地址

这些地址由 API Service 在初始化校验和初始化提交时访问，因此必须填写 API 进程可访问的地址。

API 运行在宿主机，基础设施运行在同一台机器的 Docker Compose 时：

| 配置项 | 推荐填写 |
| --- | --- |
| Redis URL | `redis://localhost:6379/0` |
| MinIO 地址 | `http://localhost:9000` |
| Qdrant 地址 | `http://localhost:6333` |
| Embedding 服务地址 | `http://localhost:8081` |
| Rerank 服务地址 | `http://localhost:8082` |

API 运行在内网服务器 `192.168.31.129`，基础设施也暴露在该服务器端口时：

| 配置项 | 推荐填写 |
| --- | --- |
| Redis URL | `redis://192.168.31.129:6379/0` |
| MinIO 地址 | `http://192.168.31.129:9000` |
| Qdrant 地址 | `http://192.168.31.129:6333` |
| Embedding 服务地址 | `http://192.168.31.129:8081` |
| Rerank 服务地址 | `http://192.168.31.129:8082` |

API 运行在 Docker Compose 网络内时：

| 配置项 | 推荐填写 |
| --- | --- |
| Redis URL | `redis://redis:6379/0` |
| MinIO 地址 | `http://minio:9000` |
| Qdrant 地址 | `http://qdrant:6333` |
| Embedding 服务地址 | `http://tei-embedding:80` |
| Rerank 服务地址 | `http://tei-rerank:80` |

如果 Qdrant 开启了 API Key 鉴权，还需要在初始化页面“基础设施”分组中填写：

| 配置项 | 建议值 |
| --- | --- |
| Qdrant API Key 引用 | `secret://rag/qdrant/api-key` |

### 11.3 LLM Provider

当前 compose 不创建 LLM 容器。初始化页面中的 LLM Provider 必须填写真实可访问的 OpenAI-compatible 服务地址，并且该服务需要提供初始化配置中声明的健康检查路径，默认是：

```text
GET /health
```

如果 LLM Provider 不可访问，最终 `PUT /internal/v1/setup-initialization` 会因为 ServiceBootstrap 失败而拒绝初始化。

### 11.4 Secret 引用

保持默认 Secret ref：

```text
secret://rag/minio/access-key
secret://rag/minio/secret-key
secret://rag/auth/jwt-signing-key
```

这些 ref 必须已经通过第 8 节写入 secrets 表。

如果 embedding、rerank 或 LLM provider 需要鉴权，在初始化页面直接填写对应 provider 的明文访问密钥。后端会加密写入 Secret Store，并自动生成以下 provider 级 Secret ref：

```text
secret://rag/model/embedding-api-key
secret://rag/model/rerank-api-key
secret://rag/model/llm-api-key
```

页面不会在请求预览中回显明文 API Key，active config 也只保存 Secret ref。后端在 ServiceBootstrap 和模型调用时从 Secret Store 读取对应 token，并以 `Authorization: Bearer <token>` 访问 provider。

## 12. 执行初始化

在初始化页面中按顺序执行：

1. 获取 setup 状态。
2. 检查本地表单提示，修正必填项和格式问题。
3. 执行配置校验。
4. 提交初始化。

也可以使用 curl 执行接口级 smoke test。注意：下面命令依赖 `docs/examples/setup-initialization.local.p0.json` 中的地址能被 API 进程访问；如果 API 运行在宿主机，请先将样例中的 Docker service name 改为 `localhost` 或内网 IP。

```bash
curl -fsS http://localhost:8000/internal/v1/setup-state

export SETUP_JWT="<复制 API 日志中的 setup JWT>"

curl -fsS http://localhost:8000/internal/v1/setup-config-validations \
  -H "authorization: Bearer ${SETUP_JWT}" \
  -H "content-type: application/json" \
  --data-binary @docs/examples/setup-initialization.local.p0.json

curl -fsS -X PUT http://localhost:8000/internal/v1/setup-initialization \
  -H "authorization: Bearer ${SETUP_JWT}" \
  -H "content-type: application/json" \
  -H "x-setup-confirm: initialize" \
  --data-binary @docs/examples/setup-initialization.local.p0.json
```

## 13. 初始化后验证

确认 setup 已关闭：

```bash
curl -fsS http://localhost:8000/internal/v1/setup-state
```

预期关键字段：

```json
{
  "initialized": true,
  "setup_required": false,
  "active_config_version": 1,
  "active_config_present": true
}
```

确认 ready：

```bash
curl -fsS http://localhost:8000/health/ready
```

再次调用初始化校验或初始化提交时，预期返回 `SETUP_CLOSED`：

```bash
curl -i -sS http://localhost:8000/internal/v1/setup-config-validations \
  -H "authorization: Bearer ${SETUP_JWT}" \
  -H "content-type: application/json" \
  --data-binary @docs/examples/setup-initialization.local.p0.json
```

## 14. 常见问题

| 现象 | 可能原因 | 处理方式 |
| --- | --- | --- |
| API 启动时报数据库不可连接 | PostgreSQL 未启动、端口被占用、`.env` 中 `DATABASE_URL` 错误 | 执行 `make ps`，检查 `DATABASE_URL` 和容器健康状态 |
| API 提示迁移缺失 | 尚未执行 Alembic migration | 执行 `make PYTHON=.venv/bin/python db-upgrade` |
| 初始化页面无法打开 | 管理后台 Vite 未启动或端口不是 5174 | 执行 `make admin`，检查终端输出 |
| 页面能打开但接口失败 | API 未启动或 Vite proxy 无法访问 `127.0.0.1:8000` | 执行 `make PYTHON=.venv/bin/python api` |
| setup JWT 校验失败 | token 过期、复制错误、API reload 后临时签名密钥变化 | 设置固定 `SETUP_TOKEN_SIGNING_SECRET`，重启 API 后复制最新 token |
| Secret 校验失败 | 未写入 secrets 表、`SECRET_STORE_MASTER_KEY` 不一致或 Secret ref 填错 | 重新执行第 8 节，并确认 ref 与初始化页面一致 |
| MinIO 校验失败 | MinIO 未 ready、endpoint 填错、MinIO 密钥与 compose 不一致 | 检查 `curl http://localhost:9000/minio/health/live` 和 `.env` 临时密钥 |
| Redis/Qdrant 校验失败 | 初始化页面填了 API 无法访问的地址 | API 在宿主机时使用 `localhost` 或内网 IP，不要使用 compose service name |
| LLM 校验失败 | 当前 compose 不提供 LLM 服务 | 填写真实 OpenAI-compatible LLM 服务地址，并确认 `/health` 可访问 |
| TEI 启动慢 | 首次启动需要下载模型 | 查看 `docker compose --env-file .env logs -f tei-embedding tei-rerank` |

## 15. 重置本地初始化数据

仅在确认可以丢弃本地数据时执行：

```bash
make reset
make up
make PYTHON=.venv/bin/python db-upgrade
```

`make reset` 会删除 PostgreSQL、Redis、MinIO、Qdrant 和 TEI 的本地 Docker volume。重置后需要重新写入 Secret，并重新执行初始化。
