# 配置变更与热更新策略

## 1. 文档目标

本文定义 Little Bear RAG 后端配置变更、风险识别、热更新、失败处理、回滚边界和作用域覆盖策略，作为 Config Service、ServiceBootstrap、API、Worker、管理后台配置页、测试计划和运维手册的共同输入。

核心目标：

- 配置变更可比较、可校验、可审计、可回放。
- 发布失败或热更新失败时不破坏旧的可服务状态。
- 权限、安全、缓存、索引和模型相关配置默认 fail closed。
- P0/P1 不实现完整审批流和通用配置回滚平台，但保留清晰的阶段 2 扩展边界。

## 2. 范围边界

P0 已要求支持：

- active config 的读取、草稿保存、校验和发布。
- 发布失败 active 指针不变。
- `config_version` 写入查询日志、模型调用日志和关键审计摘要。

P1 策略要求支持：

- 配置版本 diff、风险等级、确认要求和审计摘要。
- API、Worker 运行期配置版本感知。
- 可热更新配置的安全 reload。
- 不可热更新配置的受控发布阻断、drain 或重启提示。
- 热更新失败保留进程内 last-good 配置，并按风险进入 not ready 或降级。

P0/P1 不要求：

- 完整配置审批工作台。
- 双人复核流程。
- 通用配置回滚平台。
- 复杂配置 diff 可视化。
- 多租户运营级配置治理。
- department、knowledge_base、model_pool 的完整覆盖实现；本文只定义阶段 2 解析规则。

## 3. 配置分组和变更等级

配置以 `active_config v1` 为基础。P0 最小实现使用 global bundle，顶层分组如下：

```text
secret_provider, redis, storage, vector_store, keyword_search, model_gateway,
model, llm, auth, retrieval, chunk, import, permission, security, cache,
rate_limit, timeout, degrade, audit, observability
```

版本语义：

- `schema_version` 表示配置结构版本，P0/P1 固定为 `1`。
- `config_version` 表示运行配置版本，必须与数据库 `config_versions.version` 一致。
- 首次初始化发布 `config_version=1`；后续每次发布递增。
- 日志、审计、缓存 key 和 Worker job 记录的 `config_version` 均指运行配置版本，不指 schema version。

变更等级：

| 等级 | 含义 | 发布要求 | 热更新默认策略 |
| --- | --- | --- | --- |
| `low` | 低风险展示、阈值或非安全策略 | schema 校验通过即可发布 | 允许热更新 |
| `medium` | 影响性能、成本、召回或并发 | schema、语义校验、依赖校验 | 允许热更新或局部 reload |
| `high` | 影响权限、安全、隐私、缓存、模型路由或索引兼容 | 必须二次确认、写高风险审计 | 视分组决定 reload / drain / 拒绝 |
| `critical` | 可能导致系统不可用、越权、数据泄露或大规模索引不兼容 | 必须强确认；P0/P1 无审批流时仍不得静默发布 | 默认不热更新，需受控切换 |

## 4. 配置版本 Diff 规则

Diff 输入：

- `from_version`：当前 active config version。
- `to_version`：待发布 draft version。
- `from_config` 和 `to_config` 必须都是通过 schema 校验的完整 bundle。

Diff 生成规则：

- 使用规范化 JSON 比较：对象 key 排序，移除无意义空白，不展开 Secret value。
- 用 JSON Pointer 表达路径，例如 `/model/embedding_dimension`。
- Diff operation 只允许 `add`、`remove`、`replace`。
- 数组字段按整体值比较；后续若需要元素级 diff，再为具体字段定义稳定主键。
- Secret 只比较 Secret ref，不读取、不比较、不展示 Secret value。
- 生成 `old_value_summary` 和 `new_value_summary`，敏感字段只保留 hash、长度、ref 或脱敏摘要。

Diff 输出建议：

```json
{
  "from_version": 12,
  "to_version": 13,
  "risk_level": "high",
  "reload_plan": "api_hot_reload_worker_drain",
  "changes": [
    {
      "path": "/permission/default_visibility",
      "op": "replace",
      "old_value_summary": "department",
      "new_value_summary": "enterprise",
      "risk_level": "high",
      "requires_confirmation": true,
      "reload_scope": ["api", "worker"],
      "compatibility": "compatible"
    }
  ]
}
```

风险合并规则：

- 单个变更的最高风险就是整个版本的 `risk_level`。
- 任何 unsafe 变更必须直接拒绝发布，不进入确认流程。
- 多个 medium 变更如果同时影响查询链路和缓存链路，整体风险至少提升为 high。
- 权限、安全、缓存、审计脱敏、模型 embedding 兼容和 vector index 兼容相关变更不得降级为 low。

## 5. 高风险配置列表和确认规则

| 配置路径 | 风险 | 触发条件 | 发布规则 |
| --- | --- | --- | --- |
| `/secret_provider/*` | high/critical | provider、endpoint、auth_method、ref policy 变化 | 必须依赖校验；无法读取必需 Secret ref 时拒绝 |
| `/storage/access_key_ref`、`/storage/secret_key_ref` | high | Secret ref 变化 | 只展示 ref；必须 bucket 读写探测通过 |
| `/storage/bucket`、`/storage/object_key_prefix` | high | 存储位置变化 | 必须验证对象可写读删；已有对象迁移不在 P0/P1 自动处理 |
| `/vector_store/api_key_ref` | high | Qdrant API Key Secret ref 变化，或在无鉴权和有鉴权模式之间切换 | 只展示 ref；Secret ref 必须可读，Qdrant health 和 collection 兼容检查必须通过 |
| `/vector_store/provider`、`/vector_store/collection_prefix` | critical | 向量库或 collection 命名切换 | P0/P1 默认拒绝热更新；需受控索引迁移 |
| `/vector_store/distance` | high | 距离度量变化 | 需要重建索引或新 collection；不得直接覆盖 active index |
| `/model/embedding_model`、`/model/embedding_version` | high | embedding 语义空间变化 | 新导入可用；旧 active index 不自动兼容，需重建计划 |
| `/model/embedding_dimension` | critical | 向量维度变化 | 必须拒绝直接发布，除非新 collection 和索引重建计划已就绪 |
| `/model/embedding_normalize`、`/model/embedding_tokenizer_version` | high | embedding 兼容参数变化 | 需要索引兼容检查 |
| `/model_gateway/providers` | high | 外部模型 provider URL、类型或健康检查变化 | 必须 provider health 通过；外部供应商路由进入后续审批 |
| `/model_gateway/routes` | high | 模型路由变化 | 必须 provider health 通过；外部供应商路由进入后续审批 |
| `/model_gateway/auth_token_ref` | high | 外部模型服务认证 Secret ref 变化 | 必须验证新 ref 可读且模型健康通过 |
| `/auth/jwt_signing_key_ref` | critical | JWT 签名 key 变化 | P0/P1 默认要求重启或双 key 兼容方案；不得导致旧会话静默异常 |
| `/auth/access_token_ttl_minutes`、`/auth/refresh_token_ttl_minutes` | high | TTL 明显放宽或缩短 | 必须确认并记录影响范围 |
| `/permission/default_visibility` | high | `department -> enterprise` | 必须确认；写审计；清理权限和查询缓存 |
| `/permission/tightening_block_policy/*` | critical | 关闭 access block、旧索引阻断或 fail closed | 必须拒绝发布 |
| `/security/require_citation` | high | `true -> false` | P0/P1 必须拒绝发布 |
| `/security/prompt_leakage_policy/*` | high | 关闭 prompt / secret 防泄露 | 必须拒绝发布 |
| `/security/pii_redaction_policy/*` | high | 关闭日志或审计脱敏 | 必须拒绝发布 |
| `/cache/final_answer_enabled` | high | `false -> true` | P0/P1 默认拒绝；若后续启用必须有权限隔离证明 |
| `/cache/cross_user_final_answer_allowed` | critical | `false -> true` | 必须拒绝发布 |
| `/audit/query_text_mode` | high | `hash/none -> plain` | P0/P1 默认拒绝，除非合规策略另行批准 |
| `/audit/record_full_prompt` | critical | `false -> true` | 必须拒绝发布 |
| `/audit/retention_days` | high | 保留周期缩短 | 必须确认并写审计 |
| `/audit/pii_redaction_enabled` | high | `true -> false` | 必须拒绝发布 |
| `/keyword_search/keyword_analyzer`、词库和同义词版本 | high | 大规模召回行为变化 | 必须确认；发布后需要查询质量回归 |
| `/retrieval/*`、`/llm/*`、`/timeout/*` | medium | topK、温度、超时、上下文大小变化 | 可热更新；异常值由 schema 和语义规则拒绝 |
| `/import/*`、`/rate_limit/*` | medium | 并发、大小、限流变化 | 可热更新；超出容量规划时拒绝 |
| `/degrade/*` | medium/high | 降级策略变化 | 可能 fail open 或输出无引用答案时拒绝 |

确认规则：

- `high` 和 `critical` 变更必须由具备 `config:manage` 的管理员二次确认。
- P0/P1 没有完整审批流时，二次确认不能替代后端安全校验；unsafe 配置必须直接拒绝。
- 所有确认都必须写 `config.draft_saved`、`config.validation_started`、`config.validated`、`config.published` 或失败事件。
- 审计摘要不得包含 Secret value、完整 prompt、token、密码或未脱敏原文。

## 6. 配置发布流程

标准流程：

1. 管理员保存配置草稿。
2. Config Service 读取当前 active config，构造完整 candidate bundle。
3. JSON schema 校验。
4. 语义校验：权限、安全、缓存、降级和审计策略不得违反安全边界。
5. 依赖校验：Secret ref 可读、Redis/MinIO/Qdrant/外部模型 provider 等候选依赖可连接。
6. 索引兼容校验：embedding 维度、向量距离、collection 和 active index 兼容。
7. 生成 diff、风险等级和 reload plan。
8. 根据风险要求执行二次确认。
9. 执行 reload precheck：按 reload plan 构建新运行态或确认需要维护窗口，不替换当前运行态。
10. reload precheck 全部通过后，单事务写入 config version、归档旧 active、发布新 active 指针和审计事件。
11. 写配置变更通知；Redis 不可用时进入数据库轮询兜底。
12. API 和 Worker 原子切换到已预检的新运行态，或按维护窗口计划重启。

发布事务规则：

- 校验失败：不得生成 active version。
- reload precheck 失败：不得切换 active 指针，写 `config.reload_failed` 或 `config.publish_failed`。
- 发布事务失败：active 指针保持旧版本。
- 审计写入失败：高风险或 critical 发布必须失败，不得静默成功。
- active 指针切换必须有 compare-and-set 条件，防止并发发布覆盖。
- `config_version` 必须写入查询日志、模型调用日志和关键审计摘要。
- 多进程部署下，P1 至少要求所有关键进程在配置滞后宽限期内完成 reload；影响权限、安全、缓存、auth、embedding、vector_store 或审计脱敏的版本不得在未完成 reload 时继续对外 ready。

## 7. 热更新策略

热更新必须遵循“先构建新运行态，再原子替换”的原则。不得边修改边服务。

| 配置分组 | API 热更新 | Worker 热更新 | 策略 |
| --- | --- | --- | --- |
| `retrieval` | 是 | 否 | 新请求使用新版本；进行中请求继续旧版本 |
| `llm` | 是 | 否 | 新请求使用新参数；已有流式响应不切换 |
| `timeout` | 是 | 是 | 新请求/新任务阶段使用新预算 |
| `degrade` | 是 | 是 | 新请求/新任务使用新降级策略 |
| `rate_limit` | 是 | 部分 | 原子替换限流规则；失败时保留旧规则 |
| `cache` | 是 | 部分 | TTL 和开关可热更新；权限相关 cache key 必须包含 config_version |
| `import` | 否 | 是 | 只影响新领取任务；运行中任务保留任务开始时配置快照 |
| `chunk` | 否 | 是 | 只影响新导入或重建索引任务 |
| `keyword_search` | 部分 | 部分 | boost 可热更新；analyzer/词库变化需要索引或质量回归计划 |
| `model_gateway` | 是 | 是 | 先创建新 client 并 healthcheck，通过后原子替换 |
| `model.rerank_model`、`model.llm_model` | 是 | 否 | healthcheck 通过后新查询使用 |
| `model.embedding_*` | 部分 | 部分 | 查询和导入必须与 active index 兼容；维度变化不得直接热更新 |
| `redis` | 部分 | 部分 | 新连接池预热成功后替换；失败保留旧连接池 |
| `storage` | 否 | 否 | 普通热更新禁用；bucket、prefix 或 Secret ref 变化必须要求 API/Worker 协同维护窗口或受控重启，避免读写分裂 |
| `vector_store` | 部分 | 部分 | endpoint client 可替换；collection/distance/dimension 变化需受控索引切换 |
| `auth` | 部分 | 否 | 密码策略可热更新；JWT issuer/audience/key 变化需要兼容窗口或重启 |
| `permission` | 是 | 是 | 必须失效权限缓存和查询缓存；收紧策略不允许关闭 fail closed |
| `security` | 是 | 是 | 只能变得更严格或等价；放宽策略默认拒绝 |
| `audit` | 部分 | 部分 | sink 不热更新；retention/query_text_mode/redaction 按安全规则更新 |
| `observability` | 是 | 是 | 指标、trace、阈值可热更新 |
| `secret_provider` | 否 | 否 | P0/P1 需要重启或受控切换，不做普通热更新 |

运行时版本规则：

- 每个请求在 request context 中固定 `config_version`。
- 每个 Worker job 在领取时记录 `config_version`，阶段内不切换。
- 查询缓存 key 必须包含 `config_version`。
- 模型调用日志必须记录 `config_version` 和 `model_route_hash`。
- API `/health/ready` 应包含当前进程已加载的 runtime config version 摘要。

## 8. 热更新失败策略

失败分类：

| 阶段 | 失败 | 行为 |
| --- | --- | --- |
| 草稿保存 | schema 不完整或 JSON 非法 | 拒绝保存；若产品需要保留草稿，只能保存为 `draft` 并在 `validation_result_json` 标记 invalid，不得引入隐式状态 |
| 配置校验 | schema、语义、依赖、索引兼容失败 | 返回结构化错误，active 指针不变 |
| 发布事务 | CAS 冲突、数据库错误、审计失败 | 回滚事务，active 指针不变 |
| 通知 | Redis 发布失败 | active 已发布；写告警，服务通过 DB 轮询发现版本 |
| API reload | 新 client 构建或语义应用失败 | 保留 last-good；若影响权限、安全、缓存、模型或查询准入，ready=false |
| Worker reload | 新 client 构建或任务策略应用失败 | 保留 last-good；停止领取受影响的新任务，ready=false 或 worker_degraded |
| 部分进程版本滞后 | 超过配置版本滞后宽限期 | 进程 ready=false，告警并拒绝高风险业务 |

last-good 规则：

- 进程冷启动时必须从数据库读取 active config；读取失败不得使用本地文件或环境变量兜底。
- 进程已成功启动后，运行期 reload 失败可以继续持有 last-good 配置。
- 如果 last-good 版本落后于数据库 active 版本，且变更涉及权限、安全、缓存、auth、模型 embedding、vector_store 或 audit redaction，服务必须 fail closed。
- fail closed 行为包括：`/health/ready=not_ready`、拒绝普通业务 API、Worker 暂停领取新任务，或对查询返回结构化 `CONFIG_RELOAD_FAILED`。

缓存和权限处理：

- 权限、默认可见性、权限收紧策略或安全策略变化时，必须失效权限缓存、查询缓存和最终答案缓存。
- 如果缓存清理失败，相关查询必须绕过缓存或拒绝服务，不得复用旧缓存。
- 最终答案缓存 P0/P1 默认关闭。

## 9. 回滚策略（阶段 2）

P0/P1 不实现通用配置回滚平台。阶段 2 的回滚本质是“验证旧版本并发布一个新的 active version”，不是把数据库状态倒回过去。

阶段 2 回滚流程：

1. 选择目标历史版本 `target_version`。
2. 使用当前 active version 与 target version 生成 reverse diff。
3. 执行同样的 schema、语义、依赖、索引兼容和风险校验。
4. 若兼容，通过强确认发布新版本 `new_version`，并记录 `rollback_from_version`、`rollback_to_version`。
5. 通知 API 和 Worker reload。
6. 保留所有历史版本、审计和失败版本，禁止删除审计链。

不可自动回滚场景：

- `schema_version` 不兼容。
- `embedding_dimension`、`vector_store.distance`、`collection_prefix` 导致 active index 不兼容。
- storage bucket 或 object key prefix 变化后已有对象迁移不完整。
- Secret ref 已不存在或不可读。
- JWT signing key 变化后没有双 key 兼容窗口。
- 权限、安全或审计配置回滚会放宽安全边界。

恢复初始化边界：

- 如果 active config 缺失、损坏或无法驱动 ServiceBootstrap，可进入受控恢复初始化流程。
- 恢复初始化不是普通回滚；必须由运维和管理员确认，并写 critical 审计。
- 恢复流程仍不得从环境变量读取业务配置。

## 10. 作用域覆盖规则

P0 最小实现只要求 global active config。department、knowledge_base、model_pool 和 connector 覆盖属于阶段 2 扩展。

阶段 2 有效配置解析规则：

```text
effective_config = global
effective_config = merge(effective_config, department_override)
effective_config = merge(effective_config, knowledge_base_override)
effective_config = merge(effective_config, model_pool_override after route selected)
```

优先级：

| 作用域 | 适用场景 | 优先级 | P0/P1 状态 |
| --- | --- | --- | --- |
| `global` | 全系统默认配置 | 低 | 必须实现 |
| `department` | 部门级检索、限流、审计策略 | 中 | 阶段 2 |
| `knowledge_base` | 知识库级检索、chunk、导入、可见性默认值 | 高 | 阶段 2 |
| `model_pool` | 模型路由、模型池限流、超时、降级 | 路由后最高 | 阶段 2 |
| `connector` | 外部连接器同步 | 独立 | P2 |

覆盖约束：

- 覆盖只允许出现在配置注册表声明支持的 key 上。
- 覆盖不能改变 `schema_version`、Secret Provider、全局 JWT 签名策略或全局安全红线。
- `knowledge_base` 覆盖不能放宽 `security.require_citation`、PII 脱敏、prompt 防泄露或权限 fail closed。
- `model_pool` 覆盖只在模型路由选定后生效，不影响权限判断。
- 合并后的 effective config 必须再次通过 schema 和语义校验。
- 查询日志和模型调用日志必须记录 effective config 的版本摘要和 scope 摘要。

## 11. 审计、日志和可观测

配置变更必须写入以下信息：

- `request_id`、`trace_id`。
- actor、scope、required_scope。
- from_version、to_version、active_version。
- diff_hash、risk_level、reload_plan。
- validation_result、dependency_check_summary。
- changed_paths。
- reload result：api、worker、model provider adapter、redis notify。

禁止记录：

- Secret value。
- token 明文。
- 密码或密码摘要材料。
- 完整 prompt。
- 未脱敏查询原文或文档原文。

健康检查建议：

- `/health/ready` 返回当前 runtime config version、数据库 active config version 是否一致、last reload result。
- reload 失败时返回 `not_ready` 和结构化 `CONFIG_RELOAD_FAILED` 摘要。
- 配置版本滞后超过宽限期时告警。

## 12. 测试要求

必须覆盖：

- diff 生成稳定：同内容不同 key 顺序不产生 diff。
- Secret ref 只比较 ref，不泄露 Secret value。
- `record_full_prompt=true`、`cross_user_final_answer_allowed=true`、`fail_closed=false` 被拒绝。
- embedding 维度变化在无新 collection / 索引计划时拒绝发布。
- high / critical 变更必须要求确认。
- 配置校验失败时 active 指针不变。
- 发布事务 CAS 冲突时 active 指针不变。
- Redis 通知失败时服务可通过 DB 轮询发现新版本。
- API reload 失败时保留 last-good，并对权限、安全、缓存相关变更 fail closed。
- Worker reload 失败时停止领取受影响的新任务。
- 查询缓存 key 包含 config_version。
- 查询日志、模型调用日志和审计记录 config_version。
- 阶段 2 scope merge 不允许放宽全局安全红线。

## 13. 完成标准

- 每个 `active_config v1` 顶层分组都有明确风险和热更新策略。
- 配置发布流程明确 active 指针何时切换、失败时如何保持旧版本。
- 高风险配置列表能映射到管理后台二次确认和审计事件。
- 热更新失败不会导致权限绕过、缓存串权、无引用答案或未脱敏日志。
- 回滚边界明确：P0/P1 不提供通用回滚平台，阶段 2 通过“验证旧版本并发布新版本”实现。
- department、knowledge_base、model_pool 覆盖规则有清晰优先级，但不作为 P0 编码门禁。
