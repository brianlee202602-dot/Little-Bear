# 部署与发布检查清单

## 1. 文档目标

本文定义 Little Bear RAG 后端 P1 阶段的部署与发布检查清单，覆盖 migration 执行、初始化与恢复初始化、备份恢复、配置恢复/回滚、索引重建、权限收紧以及日志、指标、Trace 验收。

本文用于发布前评审、灰度或最小生产上线演练、故障恢复演练和后续运维 runbook。它不替代 P0 的 OpenAPI、数据库 Schema、权限矩阵、状态机、审计事件字典和测试计划。

## 2. 范围边界

P1 要解决：

- 发布前必须满足的工程、配置、数据和观测条件。
- migration 执行策略和失败处理。
- 首次初始化与恢复初始化的受控流程。
- PostgreSQL、MinIO、Qdrant、Secret Store 和配置数据的备份恢复演练。
- 配置恢复/回滚阶段 2 的演练边界。
- 索引重建和权限收紧的安全演练。
- 日志、指标、Trace、审计和 healthcheck 的验收标准。

P1 不解决：

- 不把 Kubernetes、Terraform、Helm、完整 SIEM、完整告警中心或跨地域容灾作为当前门禁。
- 不允许通过环境变量恢复业务配置。
- 不允许数据库物理回滚绕过审计链、权限阻断或配置版本语义。
- 不把配置回滚做成“覆盖旧记录”；阶段 2 回滚必须是验证旧版本后发布新 active version。
- 不允许为了完成发布而跳过权限、安全、citation、脱敏或审计检查。

## 3. 发布角色与职责

| 角色 | 职责 |
| --- | --- |
| Release Owner | 组织发布窗口、确认版本、执行 go/no-go |
| Backend Owner | 确认 API、Worker、migration、状态机和接口契约 |
| Data Owner | 执行备份、恢复演练、migration dry-run 和数据对账 |
| Security Owner | 确认 setup JWT、权限收紧、secret、审计和脱敏 |
| Ops Owner | 确认部署、healthcheck、日志、指标、Trace 和告警 |
| QA Owner | 确认 P0 自动化、E2E、security 和 release smoke |

任何一方对安全阻断项提出异议时，发布必须暂停。

## 4. 发布分级

| 类型 | 说明 | 必须执行 |
| --- | --- | --- |
| 首次部署 | 空库、首次初始化、创建 active config v1 | 全量检查 |
| 普通应用发布 | API / Worker 代码变更，不改 schema | 测试门禁、健康检查、smoke |
| Schema 发布 | 包含 Alembic migration | migration 策略、备份、回滚预案 |
| 配置发布 | active config 变更 | 配置校验、reload precheck、配置审计 |
| 检索/模型发布 | retrieval、chunk、model、keyword_search 相关变更 | 评测基线、索引兼容、模型健康 |
| 安全敏感发布 | auth、permission、cache、audit redaction 相关变更 | security gate、权限收紧演练、脱敏检查 |

## 5. 发布前总检查

### 5.1 代码与契约

- [ ] 当前发布版本有明确 commit、镜像 tag 或构建号。
- [ ] `docs/contracts/openapi.yaml`、`docs/contracts/database-schema.md`、`docs/contracts/权限矩阵.md`、`docs/contracts/状态机设计.md` 和 `docs/contracts/审计事件字典.md` 已与实现同步。
- [ ] `docs/contracts/config.schema.json` 与 `docs/contracts/config-schema.md` 一致。
- [ ] P0 自动化门禁通过：fast、db、contract、cli-contract、integration、e2e、security。
- [ ] 若涉及 3.4 范围，`eval-smoke` 或 `eval-config-change` 已执行并归档报告。
- [ ] 变更说明列出影响模块、风险等级、回退或补偿策略。

### 5.2 环境与配置边界

- [ ] API、Worker 和 Model Gateway 管理进程启动配置只包含数据库连接和进程参数。
- [ ] `.env` 或部署平台变量不包含 Redis、MinIO、Qdrant、模型网关、检索策略、权限策略、缓存策略、审计策略或 Secret value。
- [ ] Secret Store 可用，必要 Secret ref 可解析。
- [ ] setup JWT 签发、轮换和吊销命令可用，JWT 明文只展示一次。
- [ ] active config 不包含 secret value、token、密码或完整 prompt。
- [ ] `/health/live` 和 `/health/ready` 的返回不泄露内部拓扑、secret 或敏感配置。

### 5.3 数据与依赖

- [ ] PostgreSQL、Redis、MinIO、Qdrant 持久化 volume 已确认。
- [ ] PostgreSQL 备份完成并记录备份 ID。
- [ ] MinIO 关键 bucket 备份或版本策略已确认。
- [ ] Qdrant snapshot 或可重建计划已确认。
- [ ] Secret Store 备份或恢复方案已确认。
- [ ] Model Gateway、Embedding、Rerank、LLM mock 或单一 adapter health 通过。

## 6. Migration 执行策略

### 6.1 原则

- migration 默认 forward-only；生产不得依赖 destructive downgrade 恢复业务。
- destructive 变更必须拆为多阶段：新增字段/表 -> 双写或兼容读 -> 回填 -> 切流量 -> 清理旧字段。
- migration 不得绕过状态机、审计或权限约束直接修正业务状态。
- schema migration 版本必须进入 `/health/ready` 判定。
- migration 脚本必须可重复 dry-run，并能从空库执行到目标版本。

### 6.2 执行前检查

- [ ] 当前数据库 revision 与代码期望 revision 一致或在允许升级路径内。
- [ ] migration 已在空库和带样本数据的库执行通过。
- [ ] migration 执行时间、锁表风险、回填量和索引创建耗时已评估。
- [ ] 大索引创建优先使用并发或分批策略，避免长时间阻塞写入。
- [ ] 发布窗口内禁止并行执行多个 schema 发布。
- [ ] 已完成 PostgreSQL 备份，并验证备份文件可读取。
- [ ] migration 失败时的应用启动策略明确：API/Worker ready=false，不继续处理业务流量。

### 6.3 执行顺序

1. 暂停新 Worker 领取高风险任务，保留已有安全阻断状态。
2. 将 API 入口切到维护或只读策略，具体视 migration 风险决定。
3. 执行数据库备份。
4. 执行 migration dry-run 或预检查。
5. 执行 Alembic upgrade 到目标 revision。
6. 校验关键表、约束、索引和 schema revision。
7. 启动或滚动重启 API。
8. API `/health/ready` 通过后再恢复 Worker 领取任务。
9. 运行 release smoke 和关键安全用例。

### 6.4 失败处理

| 失败点 | 处理 |
| --- | --- |
| migration 预检查失败 | 不执行升级，修复脚本或数据后重排发布 |
| migration 执行失败且未提交 | 修复后重试，保留应用维护状态 |
| migration 部分提交 | 暂停业务，Data Owner 手工评估；不得直接 `downgrade` 破坏数据 |
| 新版本应用不兼容 | 停止新版本，恢复旧应用；若 schema 已前向升级，旧应用必须兼容新 schema |
| revision 不一致 | `/health/ready=not_ready`，阻断业务 API |

## 7. 初始化与恢复初始化流程

### 7.1 首次初始化

首次初始化发布步骤：

1. 部署 PostgreSQL、Redis、MinIO、Qdrant、Model Gateway mock 或单一 adapter。
2. 启动 API，确认只读取数据库连接配置。
3. `GET /internal/v1/setup-state` 返回未初始化。
4. 使用受控 CLI 签发 setup JWT。
5. 使用 setup JWT 调用 `POST /internal/v1/setup-config-validations`。
6. 校验 Secret ref、依赖连通性、active config schema 和 ServiceBootstrap 依赖。
7. 调用 `PUT /internal/v1/setup-initialization`。
8. 确认管理员、默认企业、默认部门、内置角色、active config v1 在同一事务语义内完成。
9. 确认 setup JWT 状态变为 used，初始化写接口关闭。
10. `/health/ready` 返回 ready。

检查项：

- [ ] 初始化失败不会留下半初始化状态。
- [ ] 初始化完成后 setup 写接口返回 `SETUP_CLOSED`。
- [ ] setup JWT 不能访问普通业务 API、管理员 API 或服务间 API。
- [ ] 初始化审计事件完整，且不包含 token、secret value 或密码。

### 7.2 恢复初始化

恢复初始化只用于 initialized=true 但 active config 缺失、损坏或无法驱动 ServiceBootstrap 的情况。

恢复边界：

- 不重复创建首个管理员、默认企业、默认部门、内置角色或基础权限。
- 只修复或重新发布配置版本。
- 必须由运维和系统管理员确认，并写 critical 审计。
- 仍然只能通过 setup JWT 和 setup validation 流程，不读取环境变量业务配置。

恢复步骤：

1. 系统进入 `recovery_required` 或等价拒绝模式。
2. 普通业务 API fail closed，`/health/live` 可 live，`/health/ready` not ready。
3. 运维确认恢复原因和影响范围。
4. 受控 CLI 签发或轮换恢复用 setup JWT。
5. 提交恢复初始化配置校验。
6. 发布新的 config version，并更新 active config pointer。
7. 触发 ServiceBootstrap。
8. ready 通过后恢复业务。
9. 审计记录恢复原因、旧版本、新版本、操作者和结果。

禁止：

- 禁止通过修改 `.env` 临时写入 Redis、MinIO、Qdrant 或模型地址。
- 禁止直接 UPDATE active config 跳过 schema 和依赖校验。
- 禁止删除历史 config_versions 或审计记录。

## 8. 备份与恢复演练

### 8.1 备份范围

| 数据 | 备份要求 | 恢复验证 |
| --- | --- | --- |
| PostgreSQL | 全量备份 + WAL 或增量策略 | schema、配置、用户、权限、任务、审计可读 |
| MinIO | 原始文件、中间产物、导出文件按策略备份 | 文档预览和 citation 原文可回溯 |
| Qdrant | snapshot 或明确可重建策略 | active index 可查询，维度和 payload 兼容 |
| Redis | 可选持久化；缓存可丢弃，锁和限流不能作为事实源 | 清空后系统可重建缓存 |
| Secret Store | 单独备份或恢复方案 | Secret ref 可解析，Secret value 不写日志 |
| 配置版本 | 随 PostgreSQL 备份 | active config 和历史版本可查 |

### 8.2 恢复演练步骤

1. 选择最近一次备份，记录备份 ID、时间点和目标恢复环境。
2. 恢复 PostgreSQL 到隔离环境。
3. 恢复或挂载 MinIO 备份。
4. 恢复 Qdrant snapshot；若选择重建，执行索引重建演练。
5. 恢复 Secret Store 或注入等价 Secret ref。
6. 启动 API 和 Worker，确认只读取数据库连接配置。
7. 执行 ServiceBootstrap，确认 `/health/ready`。
8. 登录管理员，查询审计日志、配置版本、知识库、文档和导入任务。
9. 执行权限过滤查询，确认 citation 可回溯到文档版本、页码和对象存储原文。
10. 执行删除阻断和权限收紧 smoke，确认旧缓存不泄露。

演练通过标准：

- [ ] RPO/RTO 达到当前业务目标。
- [ ] active config 可加载且版本正确。
- [ ] Secret ref 可解析，日志不泄露 Secret value。
- [ ] 文档、chunk、index_version、permission_snapshot 可对账。
- [ ] 查询返回 citation，且无权限用户不能访问受限文档。
- [ ] 审计日志和查询日志可查询。

## 9. 配置恢复/回滚演练（阶段 2）

P0/P1 不提供通用配置回滚平台。阶段 2 回滚是“验证历史版本并发布一个新的 active version”。

演练前提：

- [ ] 历史 config version 可读取。
- [ ] 目标版本 Secret ref 仍存在且可解析。
- [ ] 目标版本 schema_version 与当前代码兼容。
- [ ] 目标版本不会放宽安全红线，例如关闭 citation、关闭 fail closed、允许跨用户最终答案缓存。
- [ ] 若涉及 embedding、vector_store、storage，已有索引兼容或迁移计划。

演练步骤：

1. 选择历史版本 `target_version`。
2. 生成当前 active 与 `target_version` 的 reverse diff。
3. 执行 schema、语义、依赖和索引兼容校验。
4. 执行 reload precheck，确认 API 和 Worker 可加载。
5. 强确认后发布为新的 `config_version`。
6. 通知 API 和 Worker reload。
7. 校验 `/health/ready` 中 runtime config version 与数据库 active version 一致。
8. 执行查询、导入、权限、审计 smoke。
9. 写入配置发布/回滚审计。

失败处理：

- 校验失败：不发布新版本，active 指针不变。
- reload precheck 失败：不切 active pointer，写 `config.reload_failed`。
- 部分进程 reload 失败：影响权限、安全、缓存、auth、embedding、vector_store 或审计脱敏时 fail closed。
- 目标版本不兼容：放弃自动回滚，走恢复初始化或人工修复流程。

## 10. 索引重建演练

索引重建用于文档重新解析、chunk 策略调整、embedding 兼容变更、权限 payload 修复或索引健康问题修复。

安全原则：

- 重建期间旧 active index 继续服务。
- 新索引先写 draft，不得被查询命中。
- 发布前必须校验 chunk 数量、向量维度、payload hash、权限快照、draft 不可见和 citation 来源。
- 发布失败不影响旧 active index。

演练步骤：

1. 选择测试知识库或文档，记录当前 active `index_version_hash`。
2. 创建 index rebuild job。
3. Worker 写入新的 draft index_version、chunk_index_refs 和关键词索引。
4. 校验 draft index 查询不可见。
5. 校验 Qdrant collection 维度、distance 和 payload schema。
6. 校验 chunk 数量、document_version、permission_snapshot 和 indexed_permission_version。
7. 发布新 active index。
8. 旧 index 进入 archived 或 pending_delete。
9. 查询命中新的 active index，citation 可回溯。
10. 清理或保留旧索引按策略执行，并记录审计。

失败演练：

- Qdrant 写入失败：新 index_version failed，旧 active 继续服务。
- 关键词索引写入失败：不发布 active。
- 发布校验失败：不切 active index。
- 物理删除失败：只影响存储成本，access block 或旧 refs 阻断仍保持。

## 11. 权限收紧演练

权限收紧是安全高风险操作，必须验证旧用户无法通过旧缓存、旧索引或旧 permission_version 访问。

演练场景：

- 将文档 visibility 从 `enterprise` 收紧为 `department`。
- Bob 原本可见，收紧后不可见。
- Alice 属于 owner department，仍可见。

最小顺序：

1. 写 active `access_block(reason='permission_tightened')`。
2. 递增 `permission_version`。
3. 创建新 `permission_snapshot`。
4. 阻断旧 active refs 或确保查询准入 gate 不再放行旧 refs。
5. 构建并发布新受限 refs。
6. 更新 `documents.visibility` 和 `permission_snapshot_id`。
7. 失效权限缓存、查询缓存和最终答案缓存。
8. 验证 Bob 查询无结果或被拒绝，且不泄露文档存在性。
9. 验证 Alice 仍可通过新 refs 查询。
10. 仅在旧 refs 不可见且新 refs active 后释放临时 access block。
11. 写 `document.permission_tightened` 和 `access_block.*` 审计。

阻断条件：

- cache invalidation 失败时，相关查询必须绕过缓存或拒绝服务。
- 新受限 refs 发布失败时，`permission_tightened` access block 保持 active。
- 旧 refs 仍可见时，不得释放 access block。
- 评测或 smoke 发现 Bob 命中 forbidden doc，发布失败并进入安全事件处理。

## 12. 日志、指标、Trace 验收

### 12.1 结构化日志

每条关键日志至少包含：

- `timestamp`
- `level`
- `service`
- `env`
- `request_id`
- `trace_id`
- `enterprise_id`
- `user_id` 或脱敏 hash
- `stage`
- `status`
- `latency_ms`
- `error_code`
- `config_version`

禁止记录：

- password。
- access / refresh / setup / service token 明文。
- Secret value、API key。
- 完整 prompt。
- 未脱敏 query 原文。
- 文档原文或大段 chunk。

### 12.2 核心指标

发布验收至少能看到：

| 指标 | 验收 |
| --- | --- |
| API 请求量、错误率、P95 延迟 | dashboard 或日志聚合可查 |
| `/health/ready` 状态 | 可按实例查看 |
| active config version | API / Worker 当前 runtime version 可查 |
| PostgreSQL 连接池 | 使用率和错误可查 |
| Worker 队列积压 | import_jobs 队列、失败和重试可查 |
| Qdrant 查询和写入错误 | 可查 |
| Model Gateway latency / error | embedding、rerank、LLM 分类型可查 |
| query degraded rate | 可查降级原因 |
| audit write failures | 非 0 告警 |
| permission violation / draft exposure | 必须为 0 |

### 12.3 Trace 验收

至少覆盖以下链路：

- setup validation / initialization。
- 登录和 refresh rotation。
- 文档导入：API -> import_job -> Worker -> model embedding -> index publish。
- 查询：API -> permission -> retrieval -> rerank -> context -> LLM -> citation verify。
- 配置发布：validate -> reload precheck -> active pointer switch -> runtime reload。
- 权限收紧：access block -> permission snapshot -> index refs -> cache invalidation。

验收要求：

- [ ] request_id 和 trace_id 能从 API 响应、日志、审计、query_logs、model_call_logs 贯穿。
- [ ] 失败 trace 包含 stage、error_code、retryable 和补偿结果。
- [ ] Trace 不包含 prompt、query 原文、文档正文、token 或 secret。

## 13. 发布中执行顺序

推荐最小顺序：

1. 冻结发布窗口和变更范围。
2. 执行发布前总检查。
3. 执行备份。
4. 如有 migration，先执行 migration。
5. 部署 API 新版本。
6. API ready 通过。
7. 部署 Worker 新版本。
8. Worker ready 或可领取任务检查通过。
9. 如有配置变更，执行配置校验、reload precheck 和发布。
10. 执行 release smoke。
11. 解除维护或灰度限制。
12. 观察指标和错误率。
13. 记录发布结果和遗留风险。

release smoke 至少包含：

- `/health/live` 和 `/health/ready`。
- 管理员登录。
- 读取 active config 摘要。
- 创建或读取测试知识库。
- 查询一条有 citation 的问题。
- Bob 查询销售部门文档被阻断。
- 查询审计日志、query_logs 和 model_call_logs。
- Worker 可领取或扫描任务。

## 14. Go / No-Go 条件

### 14.1 Go 条件

- [ ] 所有 P0 自动化门禁通过。
- [ ] migration 成功且 schema revision 正确。
- [ ] active config 可加载，API 和 Worker runtime config version 一致。
- [ ] ServiceBootstrap ready。
- [ ] release smoke 通过。
- [ ] 审计、日志、指标和 Trace 可查。
- [ ] 备份已完成，恢复路径明确。

### 14.2 No-Go 条件

出现以下任一情况必须暂停或回滚发布：

- 权限泄露、draft index 暴露、deleted/blocked 文档命中。
- setup JWT、access token、secret value、密码、完整 prompt 或未脱敏 query 出现在日志、审计或响应中。
- migration revision 不一致或核心约束缺失。
- active config 无法加载或关键进程 config version 滞后。
- ServiceBootstrap 失败但业务 API 仍对外 ready。
- 审计写入失败且涉及高风险写操作。
- 配置发布失败但 active pointer 被错误切换。
- 索引重建失败影响旧 active index 服务。
- 权限收紧失败后 access block 被释放。

## 15. 发布后观察

发布后至少观察一个业务低峰窗口或 30 分钟，记录：

- API 错误率和 P95 延迟。
- 查询降级率和降级原因。
- Worker 队列积压、失败率和重试。
- Model Gateway 调用失败率和耗时。
- audit write failure。
- ready 状态波动。
- 配置版本滞后。
- 权限阻断和 forbidden doc 命中结果。

如发现安全相关异常，立即执行 No-Go 处理；如仅为性能退化，按故障等级决定限流、扩容、降级或回退应用版本。
